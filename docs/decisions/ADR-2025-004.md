# ADR-2025-004: Gap Remediation — Five Critical Gaps Before v1.0

**Status:** Ratified — Implementation Complete  
**Date:** 2026-06-15  
**Council:** Architecture Council, ChurchCore LMS  
**Supersedes:** None  
**Related:** ADR-2025-001 (RLS Invariants), ADR-2025-002 (Academic Structure), ADR-2025-003 (AI Pipeline)

## Context

A structured gap analysis (2026-06-12) identified five issues that must be resolved before the platform can be considered production-ready. The analysis moved the readiness estimate from 60–65% to ~75%, with these five items representing the remaining critical path.

---

## Decision

Implement all five gaps in a single coordinated PR. Each gap has a defined scope boundary; work outside that boundary is deferred to subsequent ADRs.

## Gap Inventory

### GAP-001 — Course ↔ Blueprint UI link is missing

**Problem:** `courses.blueprint_id` was added by migration 040 but no UI existed to set it. Instructors could not link a course to a blueprint, making the academic bridge triggers inert.

**Fix:** Added `BlueprintSelector` component (`src/components/courses/BlueprintSelector.tsx`). Updated `CourseForm` to accept `blueprints[]` and `initialBlueprintId` props and save `blueprint_id` to the database. Updated the course edit page server component to fetch blueprints and pass them down. Added RLS policies for `course_blueprints` in migration 043 so that authenticated instructors can read blueprints linked to their own courses.

**Files changed:**
- `src/components/courses/BlueprintSelector.tsx` — new
- `src/components/courses/CourseForm.tsx` — added blueprint field
- `src/app/courses/[id]/edit/page.tsx` — fetches blueprints, passes to form
- `supabase/migrations/20240601000043_backfill_cohort_bridge_enrollments.sql` — blueprint RLS policies

### GAP-002 — Academic bridge backfill is missing

**Problem:** Migration 040 added bridge triggers for FUTURE direct enrollments, but existing active enrollments were never backfilled into `enrollments`.

**Fix:** Migration 043 runs a batched idempotent `INSERT ... ON CONFLICT DO NOTHING` loop over all active `direct_enrollments` rows that have a corresponding course (via `blueprint_id`). Also mirrors the `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` guards so the migration is safe to re-apply.

**Files changed:**
- `supabase/migrations/20240601000043_backfill_cohort_bridge_enrollments.sql` — batched backfill loop

**Tests:** `tests/e2e/enrollment-bridge.test.ts`

---

### GAP-003 — Assignment `attempt_number` is always wrong after the first resubmission

**Problem:** `submitAssignment` in `src/app/actions/learning.ts` used `existing ? 2 : 1` for `attempt_number`. Any second-or-later resubmission always wrote `2` regardless of actual count. Rows in `block_submissions` had incorrect `attempt_number` values.

**Fix (application layer):** Replaced the heuristic with a `COUNT(*)` query: fetch the count of non-deleted submissions for `(block_id, user_id)` and use `count + 1` as the next attempt number.

**Fix (data layer):** Migration 044 recomputes `attempt_number` for all existing non-deleted rows using `ROW_NUMBER() OVER (PARTITION BY user_id, block_id ORDER BY submitted_at ASC)`.

**Files changed:**
- `src/app/actions/learning.ts` — `submitAssignment`: COUNT(*) attempt number
- `supabase/migrations/20240601000044_correct_attempt_numbers.sql` — retroactive data fix

---

### GAP-004 — `generate-embedding` Edge Function silently drops jobs when OPENAI_API_KEY is missing

**Problem:** The Edge Function checked for `OPENAI_API_KEY` before parsing the request body. If a `job_id` was already in `embedding_jobs` (from a recovery path), returning 500 early left the job permanently stuck in `pending` or `processing`.

**Fix:** Moved payload parsing to happen before the env-var check. If `OPENAI_API_KEY` is missing and a `job_id` is present in the payload, the function now marks that job `status='failed'` with `error_message='OPENAI_API_KEY not configured on server'` before returning 500. The Supabase URL/key check remains a hard fail with no job update (those vars are infrastructure-level, not user-configurable).

**Files changed:**
- `supabase/functions/generate-embedding/index.ts` — narrowly restructured env-var check path

### GAP-005 — Guardian RPC security audit

**Problem:** The ADR spec required verification that all guardian-scoped RPC functions have proper `auth.uid()` ownership predicates before any new guardian-adjacent features are added.

**Audit result (performed 2026-06-12):** **No security incident.**

| Function | Predicate | Verdict |
|---|---|---|
| `get_guardian_students()` | `WHERE guardian_uid = v_uid` where `v_uid` derives from `auth.uid()` | SECURE |
| `get_guardian_student_overview(UUID)` | verifies `guardian_links WHERE guardian_uid = v_guardian_uid` before returning data | SECURE |
| `get_my_academic_performance()` | `WHERE user_id = current_user_uid()` | SECURE |

Note: the ADR spec referenced `get_student_performance_summary`, which does not exist. The actual function is `get_my_academic_performance()`. No action required.

**Fix:** Audit-only. No schema or code changes. Test spec created to lock in the security invariants.

**Files changed:**
- `tests/e2e/guardian-rpcs.test.ts` — executable audit spec (dormant until test runner is configured)

### Infrastructure: system health check subsystem

**Problem:** No mechanism existed to detect stuck embedding jobs, missing env vars, or RLS invariant drift in production.

**Fix:** Added `system_health_checks` table (migration 045), `system-health-check` Edge Function, and `SystemHealthPanel` admin UI component.

**Files changed:**
- `supabase/migrations/20240601000045_system_health_checks.sql`
- `supabase/functions/system-health-check/index.ts`
- `src/components/admin/SystemHealthPanel.tsx`

---

## Security Invariants Maintained

- No RLS bypass. All new policies use `current_user_uid()` and `current_user_role()` — never `public.profiles` directly.
- `system_health_checks` is insert/update-only via service role (Edge Function). No client-side write path.
- `OPENAI_API_KEY` remains server-side only. The new job-failure path in the Edge Function does not expose the key or its absence to clients.
- `SYSTEM_ACTOR_ID` is a well-known constant, not a secret. It identifies automated writes in audit trails only.

---

## Consequences

- The academic bridge is now fully operational end-to-end: blueprint → section → direct enrollment → course enrollment.
- Assignment resubmission history is now accurate in the database and going forward.
- Operators have a health dashboard to detect embedding pipeline failures within 30 minutes.
- Guardian access control is formally tested and the audit result is on record.
