# ADR-2026-001 — Error Boundary Strategy

**Date:** 2026-06-20
**Status:** Accepted
**Deciders:** Council Review 1

## Context

Council Agent 3 identified that `src/app/platform/error.tsx` and `src/app/(reports)/error.tsx` do not exist. The platform admin console and the reports segment have no fallback if a server component throws. The root `src/app/error.tsx` catches these errors eventually, but the root boundary renders a generic message with no context-appropriate recovery action.

Currently present:
- `src/app/error.tsx` — root boundary
- `src/app/admin/error.tsx`
- `src/app/courses/error.tsx`
- `src/app/courses/[id]/error.tsx`
- `src/app/courses/[id]/learn/error.tsx`
- `src/app/dashboard/error.tsx`

Missing:
- `src/app/platform/error.tsx`
- `src/app/(reports)/error.tsx`

## Decision

Every top-level route group that has its own layout must also have a co-located `error.tsx`. The required set is:

| Route group | error.tsx required |
|---|---|
| `src/app/` | ✓ exists |
| `src/app/admin/` | ✓ exists |
| `src/app/courses/` | ✓ exists |
| `src/app/courses/[id]/` | ✓ exists |
| `src/app/courses/[id]/learn/` | ✓ exists |
| `src/app/dashboard/` | ✓ exists |
| `src/app/platform/` | ✗ must add |
| `src/app/(reports)/` | ✗ must add |

## Error Boundary Contract

All `error.tsx` files must:
1. Be a `'use client'` component (Next.js requirement)
2. Accept `{ error: Error, reset: () => void }` props
3. **Never render `error.message` or `error.stack` in the DOM** — log to Sentry instead
4. Render a context-appropriate recovery action (e.g., "Return to tenant list" for `/platform/`)
5. Call `Sentry.captureException(error)` in a `useEffect` on mount

## Consequences

- All future route groups must add `error.tsx` at the time the layout is created
- Spec-writer must include an error.tsx line in every implementation spec that creates a new layout
- The implementation-validator must check for error.tsx presence in every PR that adds a layout
