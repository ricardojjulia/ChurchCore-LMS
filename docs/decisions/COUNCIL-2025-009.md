# COUNCIL-2025-009: Gap Remediation — Coverage, Deployment, Dead Code, Mobile Admin, Staging

**Status:** RATIFIED — Pending Implementation  
**Date:** 2026-06-16  
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)  
**Related:** COUNCIL-2025-008 (e2e critical path), ADR-2025-006 (gamification quality cycle)  
**Tags:** testing, ci-cd, mobile, staging, dead-code, packaging, edge-functions

---

## Context

The post-v0.18.0 honest gap assessment identified six concrete risks before the platform serves real users:

1. **Server action test coverage** — dangerous mutations (cohort membership, message send, XP award via learning actions) have zero automated tests; the unit suite coverage report lists these files as excluded.
2. **Edge Function not deployed** — `supabase/functions/search-users/` was committed but never wired into `release.yml`. `UserSearchCombobox` silently returns 404 in production on every search.
3. **Dead code** — `src/components/layout/Navbar.tsx` and `src/components/layout/NavLinks.tsx` exist but are unreferenced since the sidebar migration (v0.18.0). They cause contributor confusion and type-check noise.
4. **Mobile admin gap** — every admin route (`/admin/users`, `/admin/cohorts`, `/admin/sections`, `/admin/terms`, `/admin/blueprints`) is unreachable on mobile screens; the sidebar is `hidden md:flex`. For a church context where admins often use phones, this is a meaningful operational gap.
5. **No staging environment** — migrations go from local Supabase directly to production with one manual approval gate but no intermediate environment. A bad migration discovered post-deploy has nowhere safe to validate a fix.
6. **Packaging drift** — `package.json` version drifted to `0.1.0` during early development and was only corrected for v0.18.0; no mechanism prevents future drift.

---

## Decision

Implement all six remediations. Items 3 (dead code) and 6 (packaging) are trivial and must be done first as Phase G0. Items 1, 2, 4, 5 follow as G1–G4.

### Phases

| Phase | Scope | Est. |
|---|---|---|
| G0 | Delete dead nav files; add version:check CI step | ~15 min |
| G1 | Server action tests: cohorts, messages, learning | ~2 h |
| G2 | Edge Function deploy step in release.yml; e2e smoke test | ~30 min |
| G3 | Mobile admin slide-in drawer (server + client components) | ~1 h |
| G4 | Staging environment: docs + deploy-staging job in release.yml | ~30 min |

---

## Amendments Adopted

1. Dead code deletion must be preceded by `grep -r "Navbar\|NavLinks" src/` to confirm no import still references them; if any import remains, it must be removed before the files are deleted
2. Edge Function deploy must be added to `release.yml` after the production approval gate (`approve` job), not as a separate manual command
3. Mobile admin must use a slide-in drawer pattern (not a new bottom nav tab) — the existing 5-tab `MobileBottomNav` is full; adding a tab would disrupt the learner navigation that all users depend on
4. The admin drawer button must be invisible to non-admins — server-side role check before rendering, same `isAdmin` check currently in `MobileBottomNavServer`
5. Staging environment must use a separate Supabase project (not a schema prefix) for true isolation
6. Version consistency check must run as a CI step that reads both `package.json` and `CHANGELOG.md` and exits 1 if the versions differ, preventing a version-less release from reaching production

  const match = changelog.match(/^## \[(\d+\.\d+\.\d+)\]/m)

    process.exit(1)
  }
  console.log(`OK: version ${pkg.version} consistent`)
