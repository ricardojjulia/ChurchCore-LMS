# ADR-2026-004: Engagement Events as Immutable Ledger

**Date:** 2026-06-21
**Status:** ACCEPTED
**Deciders:** Architecture Council
**Tags:** engagement, xp, analytics, data-model, ledger

---

## Context

ChurchCore LMS has a functional XP system (`award_xp()`, `profiles.xp_points`, `profiles.current_level`)
but no audit trail for how XP was earned, no engagement event log, and no streak tracking.
The proposed Engagement Tracker + Ledger feature (Priority 1 in the Sprint 2–4 roadmap) must record
every meaningful learner action — block completion, discussion post, quiz submission, login streak,
attendance — in a way that is auditable, recalculate-able, and resilient to future rule changes.

Two data model approaches were considered:

**Option A — Mutable counters**: Increment `profiles.xp_points` and `profiles.streak_count` directly
on each event, with no detailed event log.

**Option B — Immutable event log**: Append one row per engagement event to a dedicated
`engagement_events` table. Derive all metrics (total XP, streak, leaderboard rank) from this log.

---

## Decision

The engagement system uses **Option B — immutable event log** as the authoritative data store.

`profiles.xp_points` and `profiles.current_level` are retained as a denormalized fast read,
updated atomically by `award_xp()` on every event insert. They are caches, not sources of truth.

The `engagement_events` table is append-only and never mutated after insert. The `id` column is
the only key needed for deduplication (via ON CONFLICT DO NOTHING on `(user_id, source_type, source_id, event_type)`
when the same action would produce duplicate rows).

A companion `engagement_streaks` table tracks per-user streak state (current count, longest, last date).
It is updated by a trigger on `engagement_events` for `event_type IN ('daily_login', 'block_completion')`.

---

## Consequences

**Enables:**
- Full audit trail: every XP award is traceable to a source action
- Metric recalculation: if XP rules change, replay the log
- Rich analytics: engagement heatmaps, cohort completion rates, block-level engagement
- Compliance reporting: churches can show proof of learning activity per member
- Future ML: event log is the training signal for recommendation engines

**Constrains:**
- `engagement_events` must be INSERT-only from application code; no UPDATE or DELETE policies
- `profiles.xp_points` must only be written via `award_xp()` — never via direct UPDATE from app code
- Aggregate queries (leaderboard, org summary) must use indexed rollups or a materialized view,
  not full table scans on `engagement_events`
- New event types (future: watch_video, download_resource) must be added as new `event_type` enum values
  via migration — never as free-text strings

**Known tradeoffs:**
- Storage grows linearly with learner activity; acceptable for church-scale deployments (< 10K learners per org)
- Streak computation requires careful timezone handling (streak resets at midnight local time)
