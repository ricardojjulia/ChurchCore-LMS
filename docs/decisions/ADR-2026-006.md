# ADR-2026-006: SCORM Runtime Deferred to Sprint 4+

**Date:** 2026-06-21
**Status:** ACCEPTED
**Deciders:** Architecture Council
**Tags:** scorm, interoperability, deferred, scope

---

## Context

The gap analysis identified SCORM runtime support (SCORM 1.2 and SCORM 2004 playback) as a
high-demand feature ranked #11 in the Priority Implementation Order. Stakeholder feedback
suggests some church ministries have purchased SCORM content packages from third parties.

A SCORM runtime requires:
1. A server-side SCORM manifest parser (zip extraction + imsmanifest.xml parsing)
2. A client-side SCORM API (JavaScript `API` object on `window` that SCORM content calls)
3. A secure content delivery mechanism (serve SCORM assets from private storage)
4. A persistence layer for SCORM session state (`cmi.*` data model)
5. Completion detection and grade passback to `block_submissions`
6. Compliance testing against the SCORM spec (which varies between 1.2 and 2004)

Implementation effort is rated Very High. Spec compliance requires significant testing against
real SCORM packages. Non-compliant SCORM playback is worse than no SCORM support — it creates
trust damage when purchased content behaves incorrectly.

---

## Decision

SCORM runtime implementation is **deferred to Sprint 4 or later**.

The Sprint 2–3 roadmap (10 features) provides sufficient differentiation without SCORM.
The council will reassess when:
1. A specific enterprise customer or ministry partner has signed a contract requiring SCORM support, OR
2. Sprint 2–3 features ship and SCORM is the top-requested gap, OR
3. A dedicated SCORM engineering resource is available for 2+ sprints

In the interim, teachers who have SCORM content are directed to host it externally (SCORM Cloud
or an LMS that supports it) and embed the completion-tracking URL as an External Link block.

When SCORM is eventually implemented, it will require its own COUNCIL document (COUNCIL-2026-016
or later) with a full council session, security review (imsmanifest.xml parsing is an attack surface),
and a dedicated test suite against known-good SCORM packages.

---

## Consequences

**Enables:**
- Focused Sprint 2–3 scope
- All 10 Sprint 2–3 features ship without SCORM complexity

**Constrains:**
- ChurchCore LMS cannot play back externally-purchased SCORM content in this release
- Teachers with SCORM libraries need an external workaround for now
- Any future SCORM implementation must go through a full council session and cannot reuse
  application-layer patterns (it requires a custom browser API shim)

**Deferred rationale on record:** SCORM compliance risk (non-compliant runtime worse than none)
outweighs the incremental customer value at current scale.
