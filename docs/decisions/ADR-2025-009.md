# ADR-2025-009: Phase 2 Complete Tenant Isolation

**Date:** 2026-06-18
**Status:** Accepted
**Council:** COUNCIL-2025-015 (7/7 unanimous)
**Supersedes:** Partial scope of ADR-2025-008 (Phase 1 identified but deferred these tables)
**Tags:** multi-tenancy, RLS, security, org_id

---

## Context

ADR-2025-008 (Phase 1) established the multi-tenant architecture and achieved org_id
isolation on 16 tables. A post-Phase-1 audit found 30 additional tables whose RLS
policies check only `current_user_role()` without a `current_user_org_id() = org_id`
guard. This allows any authenticated staff user to read data from any tenant.

The following table groups are affected:
- **Course content:** `modules`, `course_blocks`, `content_pages`, `submissions`
- **HQ governance:** `hq_tasks`, `hq_decisions`, `hq_risks`, `hq_sessions`
- **Academic structure:** `academic_terms`, `program_tracks`, `course_blueprints`,
  `course_sections`, `access_windows`, `meeting_schedules`, `global_cohorts`,
  `cohort_members`, `cohort_section_enrollments`, `direct_enrollments`,
  `enrollment_jobs`, `enrollment_audit_log`
- **Collaboration:** `section_groups`, `section_group_members`, `group_threads`,
  `group_posts`
- **AI / analytics / audit:** `embeddings`, `embedding_jobs`, `ai_query_log`,
  `analytics_events`, `admin_audit_log`, `user_audit_log`
- **User rewards:** `profile_badges`

---

## Decision

### 1. Per-tenant vs. platform-global classification

**Per-tenant (30 tables — get org_id + org-scoped RLS):**
All 30 tables listed in Context above.

**Platform-global (no org_id — intentionally shared across all tenants):**
- `badges` — system-defined achievement definitions; orgs award them but don't define them
- `block_types` — LMS content block type registry
- `role_permissions` — platform role permission schema
- `system_health_checks` — platform infrastructure monitoring

**Verify-only (org_id already exists, confirm RLS is enforced):**
- `report_definitions`, `report_artifacts` — added org_id in migration 20250710090100;
  verify policies cover the column

### 2. Backfill strategy

Every table gets org_id backfilled from its closest org-linked ancestor:

| Source | Tables |
|--------|--------|
| `courses.org_id` via `course_id` | `modules`, `course_blocks`, `content_pages`, `submissions` |
| `profiles.org_id` via `created_by → auth.users → profiles.auth_id` | `hq_tasks`, `hq_decisions`, `hq_risks`, `global_cohorts`, `academic_terms`, `program_tracks`, `course_blueprints`, `course_sections` |
| `profiles.org_id` via `user_id → auth.users → profiles.auth_id` | `hq_sessions`, `cohort_members`, `direct_enrollments`, `ai_query_log`, `analytics_events`, `profile_badges` |
| `course_sections.org_id` via `section_id` | `access_windows`, `meeting_schedules`, `section_groups`, `cohort_section_enrollments`, `enrollment_jobs` |
| `section_groups.org_id` via `group_id` | `section_group_members`, `group_threads` |
| `group_threads.org_id` via `thread_id` | `group_posts` |
| `course_blocks.org_id` via polymorphic `source_id` where `source_type = 'course_block'` | `embeddings`, `embedding_jobs` |
| `profiles.org_id` via `uid = user_id` (bare uuid, not FK) | `admin_audit_log`, `user_audit_log`, `enrollment_audit_log` |

Rows that cannot be resolved (no matching profile/course) default to the first
`organizations` row — safe for single-tenant dev instances; investigate in production.

### 3. RLS policy pattern

All new org-scoped policies follow the Phase 1 pattern:

```sql
-- Read: any user in the same org, or platform admin
USING (public.is_platform_admin() OR public.current_user_org_id() = org_id)

-- Write: staff in the same org, or platform admin
USING (
  (public.is_platform_admin() OR public.current_user_org_id() = org_id)
  AND public.current_user_role() IN ('admin', 'manager', 'teacher')
)
WITH CHECK (same as USING)
```

Append `AND profile_roles.tenant_active = true` is implicit via the
`current_user_org_id()` function which reads from `profile_roles` — suspended tenant
users will have no matching row and the function returns NULL, blocking all access.

### 4. Migration sequencing

Three migrations, each independently deployable:

| Migration | Tables | Rationale |
|-----------|--------|-----------|
| `20260618200400` | `modules`, `course_blocks`, `content_pages`, `submissions`, `hq_tasks`, `hq_decisions`, `hq_risks`, `hq_sessions` | Highest IP risk; course content and org governance data |
| `20260618200500` | `academic_terms`, `program_tracks`, `course_blueprints`, `course_sections`, `access_windows`, `meeting_schedules`, `global_cohorts`, `cohort_members`, `cohort_section_enrollments`, `direct_enrollments`, `enrollment_jobs`, `enrollment_audit_log`, `section_groups`, `section_group_members`, `group_threads`, `group_posts` | Academic structure and collaboration layer |
| `20260618200600` | `embeddings`, `embedding_jobs`, `ai_query_log`, `analytics_events`, `admin_audit_log`, `user_audit_log`, `profile_badges` | AI/analytics/audit layer; sensitive but lower direct attack surface |

### 5. Performance

Add `CREATE INDEX IF NOT EXISTS idx_<table>_org ON public.<table>(org_id)` for every
new org_id column. The RLS hot path reads this column on every authenticated query.

### 6. What is NOT changed

- `badges` table: no org_id. Remains platform-global.
- `platform_admins`, `platform_audit_log`, `organizations`, `profile_roles`: already
  correct from Phase 1.
- Application code: no changes needed. The RLS enforcement is transparent to the app.
  Existing queries will automatically filter to the user's org.

---

## Consequences

**After these migrations:** Full org_id isolation across all 46 tables that hold
tenant data. A rogue authenticated user from Tenant A cannot read any row belonging
to Tenant B through any API path.

**Remaining Phase 3 scope (not in this ADR):**
- GDPR data export per tenant
- Platform admin impersonation / support mode with dedicated audit trail
- Inter-tenant content marketplace (course template sharing between orgs)
- Org-custom badge definitions
