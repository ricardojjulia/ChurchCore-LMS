# ADR-2026-003 — Security Hardening: Middleware Scope, RLS Helper Consistency, and Type-Safety Policy

**Date:** 2026-06-21
**Status:** Accepted
**Deciders:** Automated council security audit (June 21, 2026)

---

## Context

A full codebase security audit uncovered six categories of issues across middleware, storage, RLS policy patterns, Postgres trigger functions, API error handling, and TypeScript type safety. All items were fixed in the same session. This ADR records the architectural decisions so future code follows the same patterns.

---

## Decision 1 — Middleware Must Carve Out `/join` and `/api/`

**Problem:** `middleware.ts` redirected every unauthenticated request to `/login`, including `/join/[slug]` (self-registration) and all `/api/*` routes. Stripe webhook POSTs from Stripe's servers arrived unauthenticated and were silently dropped as 302 redirects.

**Decision:** The auth-redirect guard must always exclude:
- `/join` — self-registration is intentionally unauthenticated
- `/api/` — all API routes authenticate themselves (service role, Stripe signature, etc.) and must never be intercepted by middleware

```typescript
if (
  !user &&
  !pathname.startsWith('/login') &&
  !pathname.startsWith('/auth') &&
  !pathname.startsWith('/join') &&
  !pathname.startsWith('/api/')
) {
  return NextResponse.redirect(new URL('/login', request.url))
}
```

**Consequence:** Any future public-facing route group (e.g., `/verify/`, `/embed/`) must be added to this exclusion list at the time the route is created.

---

## Decision 2 — Private Storage Buckets Must Never Use `getPublicUrl()`

**Problem:** `AssignmentPlayer.tsx` called `getPublicUrl()` on the `assignment-files` bucket (which is `public = FALSE`) and used the result as a fallback if `createSignedUrl()` failed. The fallback URL would be refused by Supabase Storage, silently serving a broken link.

**Decision:** Any component that reads from a private bucket must:
1. Call only `createSignedUrl()` — never `getPublicUrl()`
2. Treat a `createSignedUrl()` failure as a hard error (surface to the user), not a graceful degradation

---

## Decision 3 — All RLS Policies Must Use Helper Functions

**Problem:** Two RLS policies on `organizations` used an inline subquery (`id IN (SELECT org_id FROM profile_roles WHERE auth_id = auth.uid())`) instead of the project-standard `current_user_org_id()` helper. Functionally equivalent today, but creates drift risk: future changes to `profile_roles` schema would need to be replicated in every policy that uses the inline form.

**Decision:** Every RLS policy must use the canonical helpers:
- `current_user_uid()` — not `auth.uid()` directly (except when comparing against `auth.users.id` FKs)
- `current_user_org_id()` — not inline `(SELECT org_id FROM profile_roles ...)`
- `current_user_role()` — not `(SELECT role FROM profile_roles ...)`
- `is_platform_admin()` — not any inline check against `platform_admins`

The one valid exception: `direct_enrollments.user_id` is a FK to `auth.users(id)`, not `profiles.uid`. Policies on tables with that FK pattern may correctly compare against `auth.uid()`.

---

## Decision 4 — SECURITY DEFINER Functions Must Read `profile_roles` for Caller UID/Role

**Problem:** Four guardian SECURITY DEFINER functions (`get_guardian_students`, `get_guardian_student_overview`, `link_guardian_to_student`, `unlink_guardian_from_student`) resolved the calling user's UID and role via `profiles`, not `profile_roles`. This is inconsistent with the project convention established in migration `20240601000014_fix_rls_recursion.sql`.

**Decision:** All SECURITY DEFINER functions must resolve the caller's identity via `profile_roles`:
```sql
SELECT uid, role INTO v_uid, v_role
FROM public.profile_roles WHERE auth_id = auth.uid();
```

**Exception:** `link_guardian_to_student` also looks up a guardian by email — that lookup stays in `profiles` because `profile_roles` does not store email addresses. This exception must be commented in the function body.

---

## Decision 5 — API Routes Must Not Return Raw DB Error Messages

**Problem:** Four API routes returned `error.message` directly from Supabase errors, exposing internal constraint names, bucket paths, and SQL function names to clients.

**Decision:** All API route error responses must use generic client-safe strings. Internal details go to server-side logs only. The pattern:
```typescript
// BAD
return Response.json({ error: error.message }, { status: 500 })

// GOOD
return Response.json({ error: 'Upload failed' }, { status: 500 })
```

---

## Decision 6 — `as any` Must Be Documented; Prefer Typed Alternatives

**Problem:** 28 instances of `as any` existed in production code without explanatory comments, violating the CLAUDE.md rule.

**Decision:**
1. Where a typed alternative exists (select already includes the field), remove `as any` entirely
2. Where the cause is a Supabase join inference limitation, define an explicit interface at the call site and cast to it (`as CourseRow`, `as ThreadRow`, etc.)
3. Where neither is immediately practical, add an `// eslint-disable-next-line @typescript-eslint/no-explicit-any -- [reason]` comment on the line before
4. `as any` in test files (vi.mocked partial Supabase client stubs) is exempt from this rule

**Consequence:** The `any` count in production source must not grow without a documented reason. New Supabase queries that join tables should define a typed interface for the result shape rather than using `as any` inline.

---

## Consequences

- Middleware exclusion list is now documented here; spec-writer must reference this ADR when adding new public routes
- Storage: any new private bucket must be documented as private in the migration comment and all read paths must use signed URLs
- RLS: inline `profile_roles` subqueries in policies are a migration review failure condition
- SECURITY DEFINER functions: all new functions must use `profile_roles` for caller resolution; `profiles` email lookup is the only accepted exception
- API routes: raw `error.message` in client responses is a code review failure condition
- TypeScript: `as any` without a comment or typed alternative is a code review failure condition
