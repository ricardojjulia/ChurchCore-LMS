# ADR-2026-008 — Assessment Enforcement Contract

**Date:** 2026-06-23  
**Status:** Accepted  
**Deciders:** Council Review 2 (agents 3 + 4)

---

## Context

Several block types expose configuration fields in the course builder that carry behavioral intent — limits on retakes, minimum passing scores, required viewing — but these fields are persisted to `course_blocks.content` and never read by the corresponding server action or player component.

The pattern is: builder form saves the value → block is stored → player component renders without reading the field → server action processes the submission without reading the field. The value exists in the database but is invisible to the runtime.

This creates a gap between what a teacher configures and what a student experiences. A teacher who sets "2 attempts, 80% minimum" sees those values in the form UI on every edit. A student who takes the quiz 10 times and scores 30% on every attempt is never informed of the limit or threshold.

---

## Decision

**Any field stored inside `course_blocks.content` that expresses a constraint is a binding runtime constraint, not a display hint.**

The following rules apply:

1. **Attempt limits** (`attempts_allowed` or equivalent): the server action that processes a submission must count prior attempts for the same `(block_id, enrollment_id)` pair and return an error if the limit is exceeded before processing the submission. The player component must display remaining attempts.

2. **Minimum grade thresholds** (`requirements.minimum_grade_pct` or equivalent): after scoring, the server action must compare the earned score against the threshold and mark the submission accordingly (e.g., `status = 'returned'` with feedback "Score below minimum"). The player component must display the threshold to the student before they submit.

3. **Viewing requirements** (`requirements.must_view` or equivalent): when the player cannot verify completion programmatically (e.g., an embedded YouTube video), the player must present a manual confirmation action ("Mark as Watched"). The server action records this confirmation as the completion signal, not mere block view.

4. **All new block types** that introduce constraint fields must document enforcement in the corresponding server action at the time of implementation. A builder form that saves a constraint without an enforcement path is a build-time error.

---

## Rationale

Church schools using ChurchCore LMS for ministry credentialing, Bible school programs, or academic courses depend on assessment integrity. A student who can retry a quiz indefinitely or who can advance past a failing grade undermines the value of any certificate issued by the platform. The assessment system is the trust anchor of the product.

Storing values without enforcing them also creates a false sense of security for teachers: they configure a policy, see it in the UI, and reasonably believe it is active.

---

## Consequences

- `submitQuiz` in `src/app/actions/learning.ts` must be updated to read `attempts_allowed` from block content and count prior submissions.
- `BlockPlayer` must pass `attemptsAllowed` and `minimumGradePct` to `QuizPlayer`.
- `QuizPlayer` must display remaining attempts and the passing threshold before submission.
- Future block types with constraint fields must enforce them server-side at implementation time.
- This ADR does not apply to display-only fields (e.g., `due_date` labeling, `description` text). Only fields that express a behavioral limit or requirement are covered.
