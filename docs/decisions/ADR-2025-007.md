# ADR-2025-007: Course Catalog Classification — Track Grouping and Filtering

**Supersedes:** None  
**Related:** ADR-2025-001 (blueprint wiring), ADR-2025-006 (prior quality cycle)  
**Tags:** UX, courses, classification, program-tracks, server-components

---

## Context

The data model connecting courses to program tracks was complete and correct (`program_tracks → course_blueprints → courses` via `blueprint_id`). The UX did not reflect it:

- `/courses/page.tsx` selected only `id, title, description, status, min_required_level, owner_id` — no blueprint or track join
- The course grid was ordered by `created_at DESC` with no classification structure
- `CourseCard` showed no blueprint code or track indicator
- No filter UI existed — students could not browse by program track
- The page used `redirect('/login')` — the only page in the codebase inconsistent with the `/auth/login` pattern

**Impact:** Every course sat in a flat chronological list. Students had no way to browse by curriculum area. The program track data model investment was invisible in the UX.

---

## Decision

Fix the UX in `courses/page.tsx` and `CourseCard.tsx`. No schema changes required.

### Approach

**Server-component filtering via `searchParams`.** Track filter chips are `<Link>` elements pointing to `?track=<id>`. No `'use client'` boundary is needed. Bookmarkable, shareable, survives page refresh.

**In-memory grouping after server fetch.** The dataset is small (church course catalog). Filter by track ID in JavaScript after the Supabase query returns. Avoids PostgREST nested filter syntax complexity.

**Track chips only when ≥ 2 tracks have visible courses.** A single-track installation sees no redundant filter UI.

**"Other" group last.** Courses with no blueprint or no track appear in an "Other" section, sorted after all named tracks.

**Blueprint code badge on CourseCard.** `blueprintCode?: string` prop added. Renders as a monospace badge below the title. Optional — no badge when undefined. No change to existing props or callers.

## Files Modified

### `src/components/lms/CourseCard.tsx`

- Added `blueprintCode?: string` to `Props` interface
- Renders `<span className="... font-mono ...">` badge below title when prop is present
- All existing props, logic, and callers are unchanged

### `src/app/courses/page.tsx`

Full replacement. Changes:

1. **`searchParams` prop** — `async` (Next.js 15+/16 pattern); `activeTrack` extracted
2. **`redirect('/login')` → `redirect('/auth/login')`** — consistency repair
3. **Extended courses query** — adds `blueprint:course_blueprints(course_code, title, program_track_id, track:program_tracks(id, name, code))` join
4. **Parallel fetch** — `program_tracks` fetched with `Promise.all` alongside the courses query (no serial waterfall)
5. **Track filter chips** — rendered above the grid when `tracksWithCourses.length >= 2`
6. **Grouped display** — `groupByTrack()` helper groups `filtered` array by track; section headers rendered only when `groups.length > 1`
7. **Filtered display** — `← All tracks` breadcrumb renders when `activeTrack` is set
8. **Three-case empty state** — active track / staff no courses / student no published
9. **`blueprintCode` passed to `CourseCard`** from `course.blueprint?.course_code`
10. **Progress bar** — CSS custom property pattern: `style={{ '--progress-w': '…%' }}` on outer div, `[width:var(--progress-w)]` on inner div. No direct `width` inline style.

All existing behavior preserved: enrollment progress bars, staff edit/build/analytics/grades actions, status badges, teacher `owner_id` filter, student published-only filter.

### Known Diagnostic — Progress Bar Inline Style

The IDE (VS Code extension) emits a warning on `style={{ '--progress-w': … }}` at the progress bar container. This warning fires on any `style={}` prop regardless of whether it contains a CSS custom property or a display property.

**This is a false positive for this case.** A dynamic runtime percentage value (0–100%) cannot be expressed as a static Tailwind class. The CSS custom property pattern is the Tailwind-idiomatic solution: set `--progress-w` on the container via `style`, reference it on the fill element via `[width:var(--progress-w)]` in `className`. The warning severity is **Warning** (not Error) and does not affect build or runtime.

If the project adds a Stylelint or ESLint config, add an inline suppression comment at this specific line.
