## Decisions

### GAP-009 — `.input` CSS class undefined

**Problem:** 38 form inputs across the admin suite (blueprints, terms, sections, cohorts, groups) used `className="input w-full"` but the `.input` class was never defined in `globals.css`. The `--input` CSS variable exists (shadcn color token) but is not the same as a `.input` utility class. All 38 inputs rendered as raw unstyled browser defaults.

**Fix:** Added `@layer components` block to `src/app/globals.css` defining:
- `.input` — full design-system form input: border, background, padding, placeholder color, WCAG AA focus ring (`focus-visible:ring-2`), disabled state, file input variant.
- `.btn`, `.btn-primary`, `.btn-outline` — button utility classes matching shadcn Button variants, used by `error.tsx` and `not-found.tsx` (which cannot import React client components).

**Collision audit (required by council spec):** Searched all source files and node_modules for conflicting `className="input"` usage. No third-party library uses this class name. Global scope is confirmed safe.

**Dark mode:** `bg-background`, `border-input`, `ring-ring`, `text-muted-foreground` all have dark-mode variants in the `.dark {}` CSS variable block (lines 53–93, `globals.css`). Dark mode is covered by the existing token system.

### GAP-011 — No root-level error or 404 pages

**Problem:** Unhandled exceptions and 404s fell through to bare Next.js defaults — no branding, no navigation back, no error reference for support tickets.

**Files created:**

- `src/app/error.tsx` — `'use client'` (Next.js requirement). Never renders `error.message`, `error.stack`, or any Error object property in the DOM. Only `error.digest` (opaque server-generated reference code) is safe to show. Includes a `reset()` try-again button and a dashboard link. Development-only console logging. Warm, non-clinical copy.

- `src/app/not-found.tsx` — server component. Generic message that does not echo the requested path. Dashboard link. Next.js returns HTTP 404 automatically.

- `src/app/loading.tsx` — was missing. Created with centered spinner using Tailwind `animate-spin`.

**Phase 2 (deferred):** Segment-level error boundaries at `/admin/error.tsx`, `/courses/error.tsx`, etc. for route-specific recovery flows. Documented as a comment in `error.tsx`.

### GAP-012 — CohortMemberPanel required raw UUID paste to find members

**Problem:** The only way to find a cohort member was to paste their raw `auth.users` UUID. No search or filter existed for the active member list, making cohort management admin-hostile for cohorts with >20 members.

**Fix:** Added search to `CohortMemberPanel.tsx` and a new `searchCohortMembers` server action to `cohorts.ts`.

**Search implementation:**
- Search input at top of panel, matching UsersControls visual pattern.
- Debounce: **300 ms** — matched exactly from `UsersControls.tsx` (`debounceRef.current = setTimeout(..., 300)`).
- Escape key clears search: `onKeyDown={(e) => e.key === 'Escape' && setQuery('')}`.
- Member count display: `"{n} of {total} members"` when searching, `"{total} members"` when cleared.
- ≤200 members: client-side filter by `auth_user.email` or `user_id`.
- >200 members: server-side call to `searchCohortMembers(cohortId, query)`.
- Empty state: `"No members found matching your search."` using same card style as the existing empty state.

**Security:** `searchCohortMembers` server action requires admin role (via `requireAdmin()`). The first predicate in every DB query is `.eq('cohort_id', cohortId)` — cohort scope is enforced at the SQL layer, not filtered away client-side. RLS on `cohort_members` provides a second enforcement layer.

**Note:** The "Add Member" form (UUID paste) is unchanged. The council noted this is Phase 2 scope (user search on add). The search added here is for filtering existing members, not for discovering users.

### GAP-010 — No test runner configured

**Problem:** 5 dormant e2e spec files existed but Vitest was not installed. `package.json` had no test scripts. No CI coverage was possible.

**Installation:**
```
npm install -D vitest @vitejs/plugin-react @vitest/coverage-v8 jsdom @testing-library/react @testing-library/jest-dom
```

**Files created:**
- `vitest.config.ts` — jsdom environment, globals: true, path alias `@` → `./src`, coverage via v8, `passWithNoTests: true` (CI safe from day one)
- `src/tests/setup.ts` — imports `@testing-library/jest-dom`

**Scripts added to `package.json`:**
- `test:unit` — `vitest` (watch mode)
- `test:unit:run` — `vitest run` (single pass, CI)
- `test:coverage` — `vitest run --coverage`
- `test:e2e` — `vitest run tests/e2e src/tests/e2e` (requires external infra)

**Coverage threshold:** Not yet set. Run `npm run test:coverage` after first unit tests are added, observe the reported percentage, then set `thresholds.lines` at `(measured_value - 5)` and document here.

## Triage Report — 5 Dormant Spec Files

| Spec File | Tests | Exists | Resolves | Clean | Action |
|-----------|------:|--------|----------|-------|--------|
| `tests/e2e/guardian-rpcs.test.ts` | 6 | YES | YES | **NO** | **FIXED** — 4 hardcoded credentials replaced with env vars (`GUARDIAN_A_TEST_EMAIL`, `GUARDIAN_A_TEST_PASSWORD`, `STUDENT_A_TEST_EMAIL`, `STUDENT_A_TEST_PASSWORD`). `00000000-0000-0000-0000-000000000099` is a test sentinel UUID (non-existent by design) — retained as named constant `SENTINEL_UNKNOWN_UUID`. `// @vitest-environment node` pragma added. |
| `tests/e2e/enrollment-bridge.test.ts` | 4 | YES | YES | YES | **FIXED** — `// @vitest-environment node` pragma added. No credentials found. All values via env vars. |
| `src/tests/e2e/health-route.test.ts` | 5 | YES | YES | YES | **FIXED** — `// @vitest-environment node` pragma added. No credentials found. |
| `src/tests/e2e/course-search.test.ts` | 5 | YES | YES | YES | **FIXED** — `// @vitest-environment node` pragma added. No credentials found. |
| `src/tests/e2e/admin-health-page.test.ts` | 5 | YES | YES | YES | **FIXED** — `// @vitest-environment node` pragma added. No credentials found. |

**Why `// @vitest-environment node`:** All 5 specs make real network calls (HTTP `fetch` to app URLs + Supabase client calls). They require `node` environment, not jsdom. The pragma ensures they run correctly when invoked via `npm run test:e2e`.

**Why excluded from `test:unit:run`:** These are integration/e2e tests that require `SUPABASE_TEST_URL`, `APP_BASE_URL`, valid auth tokens, a seeded test database, and a running Next.js app. They cannot pass in CI without that infrastructure. They are grouped under `test:e2e` and run separately once integration infra is provisioned.

**CI integration status:** `test:unit:run` is safe to add to CI today. It exits 0 with no tests (`passWithNoTests: true`). As unit tests are added, they will be picked up automatically.
