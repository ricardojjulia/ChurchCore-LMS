# ADR-2025-002: Academic Structure — Blueprint / Section / Cohort Model

**Date:** 2025-01-31
**Status:** ACCEPTED
**Council Vote:** 6/6 Unanimous
**Session:** COUNCIL-2025-002
**Tags:** domain-model, enrollment, academic-structure, security

---

## Context

ChurchCore LMS began with a single `courses` table handling all course delivery. As ministry use cases grow — multi-denomination deployments, cohort-based programs, prerequisite chains, synchronous and asynchronous delivery — the flat model cannot express the required structure without denormalization.

A pre-review identified six gaps in an initial specification. All six were resolved before ratification.

---

## Decision

Adopt a six-entity academic model with two implementation phases.

### Entities

1. **Program Tracks** — categorical entity (not free text); groups cohorts by program
2. **Academic Terms** — nested hierarchy (max depth 4); JSONB config with defined inheritance semantics
3. **Course Blueprints** — abstract curriculum template; carries no scheduling or enrollment data
4. **Course Sections** — scheduled instance of a blueprint within a term; `resolved_config` snapshot stored at creation
5. **Access Windows** — separated from meeting schedules; this is the content-gating security boundary
6. **Global Cohorts** — cross-course student groupings; enrollment via state machine with full audit log

### Mandatory Constraints

- `TIMESTAMPTZ` for every timestamp column. No exceptions. `NOW()` on the server determines access; client time is never used.
- Access window checks use `NOW() >= start_date AND NOW() <= end_date + grace_days`. This is the security boundary — bugs here affect content access, not just display.
- The `source` column on `direct_enrollments` is trigger-set only; application code cannot write it directly.
- `config` and `resolved_config` columns are never exposed to student-role queries.
- Enrollment status transitions are enforced by a DB trigger, not application convention. Invalid transitions raise a database exception.
- All enrollment status changes are written to `enrollment_audit_log` automatically.
- Bulk enrollment is idempotent, resumable (batch cursor), and requires admin role. Every call is audit-logged.

### Phase Gate

**Phase 1B does not begin until Phase 1A has full pgTAP coverage signed off by the QA Lead.** Access window enforcement must be verified before any enrollment logic builds on top of it.

---

## Rejected Alternatives

| Alternative | Reason Rejected |
|---|---|
| Single `schedule` table with nullable time fields | Null fields carry semantic meaning; query logic leaks everywhere; different index strategies needed |
| Free-text `program_track` on cohorts | Categoricals on free text always diverge; "Full Time MBA" vs "FT MBA" become separate tracks in reporting |
| Term config as convention only | Unenforceable; one developer stores wrong timezone and access checks silently break |
| Client-side time for access checks | DST, timezone offset bugs, and client clock tampering all become access control vulnerabilities |
| Application-enforced enrollment transitions | Runtime-enforced invariants are bypassed by direct SQL; DB trigger is the only reliable enforcement boundary |

---

## Consequences

**Positive**
- Section delivery format (sync/async/hybrid/self-paced) is cleanly separated from content gating
- Meeting schedule bugs cannot accidentally affect access window checks
- Bulk enrollment failures are recoverable without data loss
- Withdrawal retains academic records for 7 years (configurable) by default
- Program tracks enable AI tutor to personalize by track without string matching

**Negative**
- ~13 new tables; Phase 1A alone is 3 sprints for a small team
- `resolved_config` snapshot requires a recalculation trigger on ancestor term config changes
- Materialized view (`effective_enrollments`) requires a refresh strategy (pg_cron + trigger)
- Each `direct_enrollment` row now carries audit log overhead on every status change

---

## Implementation Phases

### Phase 1A — Academic Skeleton (3 sprints)
Tables: `program_tracks`, `academic_terms`, `course_blueprints`, `course_sections`, `access_windows`, `meeting_schedules`

Functions: `resolve_term_config()`, `check_section_access()`

Deliverables: All RLS policies + full pgTAP coverage before Phase 1B unlocks

### Phase 1B — Enrollment Engine (3 sprints)
Tables: `global_cohorts`, `cohort_members`, `enrollment_jobs`, `cohort_section_enrollments`, `direct_enrollments`, `enrollment_audit_log`

View: `effective_enrollments` (materialized)

Deliverables: State machine trigger, batch enrollment Edge Function, admin enrollment wizard

### Phase 2 — Section Groups (2 sprints)
Tables: `section_groups`, `section_group_members`

### Phase 3 — AI + Analytics Integration (2 sprints)
AI tutor aware of cohort + program track; enrollment analytics; access window expiry notifications; pgvector integration

### Phase 4 — Advanced Institutional Features (3 sprints)
Waitlist, CSV import, self-enrollment with cohort auto-assignment, access window extension workflow
