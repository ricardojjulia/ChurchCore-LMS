# ADR-2025-001: Rich Text Editor Selection — Tiptap over TinyMCE

**Date:** 2025-01-31
**Status:** ACCEPTED
**Deciders:** Architecture Council
**Tags:** editor, content, security, AI, data-model
**Review:** CR-2025-001

---

## Context

ChurchCore LMS requires rich text editing for course pages, discussion
prompts, and assignment descriptions. We evaluated TinyMCE (incumbent
proposal) against Tiptap, Quill, and plain Markdown with preview.

The evaluation criteria were:

1. Compatibility with Next.js App Router (no SSR conflicts)
2. Output format suitability for AI semantic search
3. Security profile (XSS surface, CSP compatibility)
4. Licensing cost and vendor dependency
5. Realtime collaboration readiness
6. Bundle size on learner-facing routes

---

## Decision

**We will use Tiptap (ProseMirror-based) as the rich text editor
across all content authoring surfaces.**

Content is stored as JSONB (Tiptap document format, `format_version = "tiptap-v2"`)
in Supabase. HTML is generated server-side via `@tiptap/html`'s `generateHTML`
at render time — never stored. A `format_version` column tracks document schema
versions for future migration scripts.

---

## Consequences

### Positive

- JSON storage enables AI content indexing without HTML parsing
- No vendor API key or CDN dependency (MIT license, fully self-hosted)
- Strict CSP possible — no `unsafe-inline` / `unsafe-eval` required
- Headless architecture gives full design system control
- ProseMirror foundation enables future realtime collaboration via Yjs
- Tree-shakeable extensions keep learn-route bundles minimal
- `body_text` generated column enables full-text search via `tsvector`

### Negative

- Higher initial implementation complexity vs. drop-in TinyMCE
- Team must learn Tiptap extension API for custom block types
- Paste-from-Word behavior requires explicit testing and tuning
- No built-in file manager (Supabase Storage integration is manual — Phase 2)

### Neutral

- Tiptap v2 (stable) chosen; v3 roadmap monitored
- `format_version` column future-proofs schema migration
- Mobile editing deferred to Phase 2 with a viewport warning in Phase 1

---

## Alternatives Considered

| Option | Rejected Reason |
|---|---|
| TinyMCE | HTML output; CSP violation (`unsafe-eval`); license cost; poor AI integration |
| Quill | Unmaintained (last major release 2019); delta format less portable |
| Slate.js | Too low-level; high custom build cost for no clear advantage |
| Markdown + Preview | Poor UX for non-technical ministry content creators |
| BlockNote | Promising but immature; revisit at Phase 4 |

---

## Compliance Notes

- All `content_pages` RLS policies covered by pgTAP tests before UI ships
- `embedding` column must never be exposed to learner-role queries (Phase 3)
- XSS test cases must pass in CI before any page-render code ships
- CSP headers validated via `securityheaders.com` score ≥ A before production deploy

---

## Implementation Phases

| Phase | Scope |
|---|---|
| 1 (current) | Tiptap install, `content_pages` table + RLS, RichTextEditor component, PageEditor route, pgTAP tests |
| 2 | Image uploads (Supabase Storage), discussion editor, autosave indicator, unsaved-changes guard, mobile warning |
| 3 | pgvector embeddings, AI tutor semantic search, `content_pages_public` view |
| 4 | Realtime collab (Yjs), content versioning, PDF export, block templates |

## Review Date
