# ADR-2025-008: Multi-Tenant Architecture — Shared Schema with RLS Isolation

**Status:** Ratified
**Date:** 2026-06-18
**Council:** Architecture Council, ChurchCore LMS (7/7 unanimous — COUNCIL-2025-014)
**Supersedes:** None
**Related:** ADR-2025-001 (RLS source of truth), ADR-2025-004 (identity model)
**Tags:** multi-tenancy, platform-admin, RLS, organizations, security

---

## Context

ChurchCore LMS is a SaaS platform. Each client organization (a church, seminary, or
ministry) is a **tenant** — a fully isolated unit of data and users. The system must
enforce that Tenant A cannot read or modify Tenant B's data under any circumstances.

**Current state at the time of this ADR:**
- `organizations` and `org_members` tables exist.
- `profiles.org_id` and `courses.org_id` are set and RLS-enforced.
- 13+ other tables — including `announcements`, `messages`, `notifications`,
  `calendar_events`, `block_submissions`, `course_certificates`, `guardian_links`,
  `message_threads`, `messages` — have no `org_id` column and no tenant-scoped RLS.
  Cross-tenant reads on these tables are currently possible.
- No super-admin (platform-level) identity exists. The current `admin` role is
  org-level only.
- No tenant lifecycle management (create, suspend, delete) exists.

---

## Decision

### 1. Isolation Model: Shared Schema, org_id RLS

All tenants share one Postgres database and one schema. Isolation is enforced exclusively
by Row-Level Security. Every table that contains tenant data carries an `org_id uuid`
column with a foreign key to `organizations.id` and RLS policies of the form:

```sql
USING (is_platform_admin() OR current_user_org_id() = org_id)
```

**Why not separate databases or schemas per tenant?** Supabase supports one database per
project. Separate schemas would require connection-string switching, which is incompatible
with Supabase Auth. The RLS-per-table model is battle-tested, operationally simple, and
scales to hundreds of tenants without new infrastructure.

### 2. Platform Admin Identity: Separate Table

A **platform admin** (super-admin) is not a tenant user. It is a separate identity plane.

```sql
CREATE TABLE public.platform_admins (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  auth_id      uuid UNIQUE NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name text NOT NULL,
  created_at   timestamptz NOT NULL DEFAULT now()
);
```

```sql
CREATE OR REPLACE FUNCTION public.is_platform_admin()
RETURNS boolean LANGUAGE sql STABLE SECURITY DEFINER SET search_path = public AS $$
  SELECT EXISTS (SELECT 1 FROM platform_admins WHERE auth_id = auth.uid())
$$;
```

- `is_platform_admin()` reads **only** `platform_admins` — never `profiles.role`.
- Setting `profiles.role = 'admin'` grants org-level admin. It does not grant platform admin.
- The first platform admin is provisioned via a migration with a hardcoded `auth_id`.
  Subsequent platform admins are added by an existing platform admin via the console.
- `platform_admins` has RLS: `USING (is_platform_admin())` on all operations.

### 3. Tenant Lifecycle

`organizations` gains these columns:

| Column | Type | Default | Notes |
|---|---|---|---|
| `status` | `text` | `'trial'` | `CHECK (IN ('trial','active','suspended','deleted'))` |
| `plan` | `text` | `'free'` | `CHECK (IN ('free','standard','premium'))` |
| `trial_ends_at` | `timestamptz` | `now() + 14 days` | Null if plan ≠ trial |
| `deleted_at` | `timestamptz` | `null` | Set on soft-delete |
| `settings` | `jsonb` | `'{}'` | Already exists — extended for branding, features, onboarding |

**Lifecycle transitions:**
```
trial ──────────────────────────────→ active (org admin upgrades/payment)
trial ──[trial_ends_at passed]──────→ suspended (nightly function)
active ──[platform admin action]────→ suspended
suspended ──[platform admin action]─→ active
active/suspended ──[delete action]──→ deleted (soft, deleted_at set)
deleted ──[30 days later]───────────→ [hard purge, nightly job]
```

**Suspended tenant access:** RLS blocks data access. Implemented via `tenant_active boolean`
on `profile_roles`, synced by the `sync_profile_roles` trigger when `organizations.status`
changes. No join in the hot RLS path — a simple column read.

**Soft-delete:** All tables gain an implicit filter `AND (org.deleted_at IS NULL OR is_platform_admin())`.
Implemented by adding `deleted_at IS NULL` to the org lookup in `current_user_org_id()`.

**Hard purge:** A nightly Edge Function deletes in FK dependency order:
block_submissions → enrollments → course_blocks → courses → profiles → org_members → organizations.

### 4. Tables Requiring org_id Addition

The following tables need `org_id uuid NOT NULL REFERENCES organizations(id)` added and
their RLS policies rewritten to the standard pattern:

- `announcements`
- `announcement_reads`
- `calendar_events`
- `notifications`
- `message_threads`
- `message_thread_participants`
- `messages`
- `block_submissions`
- `course_certificates`
- `guardian_links`
- `user_audit_log`
- `admin_audit_log`
- `course_enrollments` (verify, may already have via course join)

Backfill: `org_id` is inferred from `created_by → profiles.org_id` for each table.
Rows where this cannot be inferred default to the first (and historically only) org.

### 5. Platform Admin Console

Route: `/platform` — guarded by `is_platform_admin()` at the server component level.
The layout never renders for tenant users regardless of role.

**Phase 1 features:**
- Tenant list (name, status, plan, user count, course count, health score, trial_ends_at)
- Create tenant (name, slug, plan, trial duration, provision first admin via invite)
- Edit tenant (rename, plan, feature flags, branding)
- Suspend / restore / soft-delete
- Per-tenant usage card
- Platform audit log viewer

### 6. White-labeling and Feature Flags

Stored in `organizations.settings` JSONB:

```json
{
  "branding": {
    "logo_url": "https://...",
    "primary_color": "#4f46e5",
    "org_name_display": "Grace Community Seminary",
    "email_from_name": "Grace Seminary"
  },
  "features": {
    "ai_tutor": true,
    "guardian_portal": false,
    "leaderboard": true,
    "hq": false,
    "reporting": true
  },
  "onboarding": {
    "logo_uploaded": false,
    "first_teacher_invited": false,
    "first_course_created": false,
    "first_announcement_published": false
  }
}
```

The app root layout reads `organizations.settings` server-side for the authenticated user's
org and provides branding via CSS variables. The sidebar reads `features` to conditionally
show/hide modules.

### 7. Scheduled Functions

| Function | Schedule | Action |
|---|---|---|
| `expire-trials` | Nightly | Sets `status = 'suspended'` where `trial_ends_at < now() AND status = 'trial'` |
| `purge-deleted-tenants` | Nightly | Hard-deletes orgs where `deleted_at < now() - interval '30 days'`, FK order |

### 8. Platform Audit Log

```sql
CREATE TABLE public.platform_audit_log (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  actor_id    uuid NOT NULL REFERENCES auth.users(id),
  action      text NOT NULL,  -- 'create_tenant', 'suspend_tenant', 'provision_user', etc.
  target_org  uuid REFERENCES organizations(id),
  payload     jsonb,
  created_at  timestamptz NOT NULL DEFAULT now()
);
```

Every platform admin action (create, suspend, delete, plan change, provision user) writes here.
RLS: `USING (is_platform_admin())`.

## Consequences

**Positive:**
- Complete tenant isolation enforced at the database layer — not the application layer.
- Platform admin is architecturally separate from tenant admin — no escalation path.
- White-labeling and feature flags require zero schema changes to add or modify.
- Trial enforcement is automatic — no manual intervention needed.
- The existing `organizations` table is extended, not replaced.

**Negative / Trade-offs:**
- The RLS audit migration is extensive: 13 tables get `org_id` backfilled and their
  policies rewritten. This is a correctness fix for an existing gap.
- Adding `tenant_active` to `profile_roles` adds one more field to the sync trigger.
- Platform admin bypasses on every table's RLS add a function call to every row check.
  `is_platform_admin()` is STABLE and will be inlined by the planner for typical queries.

**Deferred:**
- GDPR/data export (Phase 2)
- Impersonation/support mode (Phase 2)
- Inter-tenant content marketplace (Phase 3)
