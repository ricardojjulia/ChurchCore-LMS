# COUNCIL-2025-008: Critical-Path End-to-End Test Suite

**Status:** RATIFIED — Pending Implementation  
**Date:** 2026-06-16  
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)  
**Related:** ADR-2025-004 (enrollment engine), ADR-2025-005 (health/monitoring), ADR-2025-006 (gamification)  
**Tags:** testing, e2e, ci, supabase, xp, certificates, enrollment

---

## Context

The platform has no automated verification that its most fundamental user journey works end-to-end: a student signs in, is enrolled in a course, the enrollment is marked complete, XP is awarded, and a certificate is issued. All existing automated tests cover leaf utilities and individual server actions in isolation using mocked Supabase clients.

No test exercises the full stack — Supabase auth, RLS-enforced queries, the `award_xp()` SECURITY DEFINER function, the `issue_certificate()` function, and the `course_certificates` table — in sequence. Schema drift between migration files and a live database instance (e.g., `admin_audit_log` or `embeddings` columns added in migrations 038–049) would not be caught until a real user hits it in production.

**The risk is concrete.** CI currently passes even if the database is in an inconsistent migration state, because no test queries real Supabase tables. A stale test database has already caused one unreported incident in a staging deploy.

---

## Decision

Implement one authoritative critical-path E2E test suite in `src/tests/e2e/critical-path.test.ts` that runs against the real test Supabase instance provisioned by `e2e.yml`. The test must not mock any database layer. It must cover the complete student lifecycle in a single ordered spec file, using seed data from `supabase/seed.test.sql`.

### Approach

**Vitest with real Supabase clients.** The project uses Vitest (not Playwright). The `// @vitest-environment node` directive disables jsdom for e2e specs. E2E tests import `createClient` from `@supabase/supabase-js` directly — not from the app's wrapper — and connect to the test Supabase instance via `TEST_SUPABASE_URL` + `TEST_SUPABASE_SERVICE_ROLE_KEY`.

**Two clients in parallel.** A service role client handles setup and cleanup (bypasses RLS). A user-session client (signed in as the test student) calls the same RPCs the real app calls, exercising RLS and the SECURITY DEFINER functions under authentic auth.uid() context.

**Ordered steps, shared module-level state.** Each lifecycle step is a distinct `it()` block. Step results (baseline XP, enrollment row) are stored in `let` variables at module scope and read by subsequent steps. This mirrors the real user session where prior state accumulates.

**Schema drift as the first assertion.** The first test step queries `information_schema.columns` for columns added in migrations 028–049 that are commonly omitted when the test database is seeded from an older snapshot. If any column is missing, all subsequent steps are meaningless and fail immediately with a clear message.

---

## Amendments Adopted

1. Seed must be idempotent — all inserts use `ON CONFLICT DO NOTHING`
2. Test must assert the exact XP delta (`baselineXP + 10`), not just `xp > 0`
3. Certificate assertion must verify `certificate_no` is non-null and matches the issued course
4. Each assertion step must include compensating cleanup in `afterAll` so the test is re-runnable
5. The spec must throw (not silently skip) if `TEST_SUPABASE_URL` or `TEST_SUPABASE_SERVICE_ROLE_KEY` is absent from the environment
6. Schema drift is tested by asserting four columns from migrations 028–049: `profiles.xp_points`, `profiles.current_level`, `embeddings.embedding_status`, `admin_audit_log.actor_id`

## Implementation Mandate

### Environment and seed facts

| Variable | Value |
|---|---|
| Supabase URL env var | `TEST_SUPABASE_URL` |
| Service role env var | `TEST_SUPABASE_SERVICE_ROLE_KEY` |
| Password env var | `TEST_USER_PASSWORD` (no default — fail loudly if missing) |
| Student email | `student@test.churchcore.dev` |
| Student auth id | `00000000-0000-0000-0001-000000000003` |
| Student profile uid | `00000000-0000-0000-0002-000000000003` |
| Course id | `00000000-0000-0000-0003-000000000001` |
| Existing enrollment | `enrollments` row: `transit_status='in_progress'`, `progress_percent=25` |

### Schema facts

**`award_xp(p_uid UUID, p_amount INTEGER)`** — SECURITY DEFINER. Must be called with user JWT. Validates `current_user_uid() = p_uid` (or caller is staff). Returns `{ new_xp, new_level, leveled_up, prev_level }`.

**`issue_certificate(p_uid UUID, p_course_id UUID)`** — SECURITY DEFINER. Requires `enrollments.transit_status = 'completed'` for the given user+course. If `mv_academic_performance` has no row for the pair, `letter_grade` defaults to `'N/A'` (not NULL). Returns `{ certificate_id, certificate_no, final_grade, letter_grade }`.

**`course_certificates`** — columns: `id`, `user_id`, `course_id`, `issued_at`, `final_grade`, `letter_grade`, `total_xp_earned`, `certificate_no` (UNIQUE, auto-generated as `'CERT-' || UPPER(SUBSTR(gen_random_uuid()::TEXT, 1, 8))`). One cert per `(user_id, course_id)`.

**`enrollments`** — columns: `user_id`, `course_id`, `transit_status` (`'not_started'|'in_progress'|'completed'`), `progress_percent`, `completed_at`, `last_accessed_at`.

5. describe('Critical path: sign-in → enroll → XP → certificate', () => {

   it('STEP 1 — schema drift: columns from migrations 028-049 exist', async () => {

     // (profiles, xp_points), (profiles, current_level),
     // (embeddings, embedding_status), (admin_audit_log, actor_id)
     // Use one query: SELECT table_name, column_name FROM information_schema.columns
     //   WHERE (table_name, column_name) IN (...)
     // Assert all 4 rows returned. If fewer, throw with which column is missing.

   it('STEP 2 — course and enrollment exist in seed data', async () => {
     const { data, error } = await serviceClient.from('courses')
       .select('id, title, status').eq('id', COURSE_ID).single()
     expect(error).toBeNull()
     expect(data?.status).toBe('published')

     const { data: enr, error: enrErr } = await serviceClient.from('enrollments')
       .select('transit_status, progress_percent, user_id, course_id')
       .eq('user_id', STUDENT_UID).eq('course_id', COURSE_ID).single()
     expect(enrErr).toBeNull()
     expect(enr).not.toBeNull()
   })

   it('STEP 3 — record baseline XP', async () => {
     const { data, error } = await serviceClient.from('profiles')
       .select('xp_points, current_level').eq('uid', STUDENT_UID).single()
     expect(error).toBeNull()
     baselineXP = data?.xp_points ?? 0
     expect(typeof baselineXP).toBe('number')
   })

   it('STEP 4 — award XP to student (10 points via RPC)', async () => {
     // studentClient has user JWT — current_user_uid() resolves to STUDENT_UID
     const { data, error } = await studentClient.rpc('award_xp', {
       p_uid:    STUDENT_UID,
       p_amount: 10,
     })
     expect(error).toBeNull()
     const result = data as { new_xp: number; new_level: number }
     expect(result.new_xp).toBe(baselineXP + 10)
     expect(result.new_level).toBeGreaterThanOrEqual(1)
   })

   it('STEP 5 — verify XP persisted in profiles table', async () => {
     const { data, error } = await serviceClient.from('profiles')
       .select('xp_points, current_level').eq('uid', STUDENT_UID).single()

     expect(data?.xp_points).toBe(baselineXP + 10)
     expect(data?.current_level).toBeGreaterThanOrEqual(1)
   })

   it('STEP 6 — mark enrollment completed (setup for certificate)', async () => {
     const { error } = await serviceClient.from('enrollments')
       .update({
         transit_status:   'completed',
         progress_percent: 100,
         completed_at:     new Date().toISOString(),

       .eq('user_id', STUDENT_UID)
       .eq('course_id', COURSE_ID)

   })

     const { data, error } = await studentClient.rpc('issue_certificate', {
       p_uid:       STUDENT_UID,
       p_course_id: COURSE_ID,

   })

       .select('id, user_id, course_id, certificate_no, letter_grade')

       .eq('course_id', COURSE_ID)
