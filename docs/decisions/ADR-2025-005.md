# ADR-2025-005: Gap Remediation — Three Routing and Wiring Gaps (COUNCIL-2025-005)

**Status:** Ratified — Implementation Complete  
**Date:** 2026-06-15  
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)  
**Supersedes:** None  
**Related:** ADR-2025-004 (Gap Remediation — Five Critical Gaps)

---

## Context

Following the completion of ADR-2025-004, a re-evaluation placed overall platform readiness at ~83%. Three gaps remained that all shared the same root pattern: **a UI component existed but was not reachable through a navigable route, or a route existed but its supporting API surface was absent.**

The Architecture Council (COUNCIL-2025-005) identified these as high-severity routing/wiring gaps and ratified a remediation plan on 2026-06-15 with a 6/6 unanimous vote. All three were resolved in a single coordinated implementation.

---

## Decision

Implement all three gaps in strict dependency order: GAP-007 → GAP-006 → GAP-008. Shared utilities, navigation updates, and the auth guard audit are treated as sub-tasks of this same milestone, not separate ADRs.

---

## Gap Inventory

### GAP-007 — Course Search Returns No Results

**Problem:** `src/app/api/search/route.ts` queried `courses` with `.eq('is_published', true)` and `.select('id, title, description, is_published')`. The `courses` table does not have an `is_published` column — it uses `status TEXT` with the value `'published'`. Every search call silently returned an empty result set because Postgres filters on a non-existent column without error.

**Fix:** Changed the column filter to `.eq('status', 'published')` and the SELECT to include `status, course_blueprints ( course_code, title )`. Added `.order('title', { ascending: true }).limit(5)` for consistent ordering. Added a non-empty empty-state message: `{ courses: [], message: 'No courses found for your search. Try different keywords.' }` so the UI can display a useful prompt rather than a blank list.

**Files changed:**
- `src/app/api/search/route.ts` — column fix + blueprint join + ordering + empty-state message

---

### GAP-006 — `/api/health` Route Missing

**Problem:** `SystemHealthPanel` (the admin health UI component) made fetch calls to `/api/health` (GET to read results, POST to trigger the Edge Function), but no such route existed. Every interaction with the health panel silently failed with a 404.

**Fix:** Created `src/app/api/health/route.ts` with:
- `requireAdmin()` guard that reads role from `profiles` (not `user_metadata`) — consistent with all other admin pages.
- `GET` handler: reads `system_health_checks` via the shared `getHealthChecks()` utility and returns `{ checks, timestamp }`.
- `POST` handler: invokes the `system-health-check` Edge Function via service-role client, then reads and returns the freshly persisted rows.

**Files changed:**
- `src/app/api/health/route.ts` — new
- `src/lib/queries/getHealthChecks.ts` — new shared utility (single query shape used by both route and server page)
- `src/types/health.ts` — new type definitions (`HealthStatus`, `SystemHealthCheck`, `HealthCheckResponse`)

---

### GAP-008 — `/admin/health` Page Missing

**Problem:** `SystemHealthPanel` was a component with no page. There was no navigable route to `/admin/health`, no link in the admin nav, and no server-side data fetch wiring it to `getHealthChecks()`. The component was effectively invisible to users.

**Fix:**
- Created `src/app/admin/health/page.tsx` — server component with `force-dynamic`, auth guard (redirects to `/auth/login` if unauthenticated, `/dashboard` if non-admin), fetches `initialChecks` via `getHealthChecks()`, renders `<SystemHealthPanel initialChecks={initialChecks} />`.
- Updated `src/components/layout/NavLinks.tsx` — added `/admin/health` entry with `healthBadge: true` and `adminOnly: true`. Added `healthErrorCount` prop; badge renders current error count from the DB.
- Updated `src/components/layout/Navbar.tsx` — added a count-only query on `system_health_checks` where `status = 'error'` for admin sessions; passes `healthErrorCount` to `NavLinks`.
- Updated `src/components/dashboard/AdminDashboard.tsx` — added `HealthWidget` inline component that renders a color-coded summary card linking to `/admin/health`, and accepts `healthChecks?: SystemHealthCheck[]` prop.
- Updated `src/app/dashboard/page.tsx` — for admin role, fetches health checks and passes them to `AdminDashboard`.

**Files changed:**
- `src/app/admin/health/page.tsx` — new
- `src/components/layout/NavLinks.tsx` — health nav entry + badge
- `src/components/layout/Navbar.tsx` — health error count fetch
- `src/components/admin/SystemHealthPanel.tsx` — full rewrite with state machine (`idle | loading | success | error | not_found`), `action_url` support, expandable metadata, relative timestamps
- `src/components/dashboard/AdminDashboard.tsx` — `HealthWidget` + `healthChecks` prop
- `src/app/dashboard/page.tsx` — health check fetch for admin

## Schema Alignment (Migration 046)

The original `system_health_checks` table (migration 045) used column names `details` and `checked_at`. The council spec defined `metadata` and `last_checked` to match the semantic of storing structured context vs. a simple label. Migration 046 (`20240601000046_align_health_checks_schema.sql`) renamed both columns and added `action_url TEXT` for deep-link navigation from panel rows to the relevant admin page.

All code — Edge Function, API route, shared utility, panel component, and type definitions — uses the post-migration column names exclusively.

## Auth Guard Audit

As part of this work, all pages under `src/app/admin/` were audited for consistent auth guards. Every page must:
1. Call `supabase.auth.getUser()` as the first server-side operation.
2. Redirect to `/auth/login` (not `/login`) if no session.
3. Read `role` from the `profiles` table (not `user_metadata`).
4. Redirect non-admins to `/dashboard`.

The audit found no deviations that were introduced by ADR-2025-004 work. The `/admin/health` page was authored to match this pattern from the start.

## PR Template Update

Added a **Frontend & Routing Checklist** section to `.github/pull_request_template.md` to institutionalize the checks that caught these three gaps:

- Page file exists that renders the component at a navigable route
- Route is linked from appropriate navigation
- API route exists for every fetch/mutation the component performs
- Auth guards exist on both the page and the API route
- Loading, success, and error states are implemented

---

## Consequences

**Positive:**
- Course search now returns results (critical user-facing regression fixed).
- System health monitoring is end-to-end: Edge Function → DB → API route → page → navigation badge → dashboard widget.
- The PR template checklist prevents the same class of routing gap from going undetected in future PRs.
- Auth guard pattern is codified: `profiles` table, `/auth/login` redirect, no `user_metadata` role reads.

**Negative / Tradeoffs:**
- The `Navbar` now makes an additional DB query on every page load for admin sessions (health error count). This is a single count-only query and is acceptable; if it becomes a bottleneck, it can be moved to a background polling interval.
- `SystemHealthPanel` is now a relatively complex client component (state machine, expandable rows, relative timestamps). The complexity is contained within one file.
