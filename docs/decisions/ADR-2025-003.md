# ADR-2025-003: Unified AI Embedding Architecture

**Date:** 2025-01-31
**Status:** ACCEPTED
**Council Vote:** 6/6 Unanimous
**Session:** COUNCIL-2025-003
**Prerequisite ADRs:** ADR-2025-001 (Tiptap), ADR-2025-002 (Academic Structure)
**Tags:** AI, embeddings, pgvector, security, search, context

---

## Context

CR-2025-001 (Tiptap content pages) and ADR-2025-002 (Academic Structure) both schedule AI layer
work in Phase 3. If they ship independently, two divergent vector strategies result — one for
content retrieval, one for academic context — that share the same query surface (the AI tutor)
with no unified schema, no shared RLS boundary, and no consistent search API contract.

This ADR was ratified before either Phase 3 begins.

---

## Decision

Adopt a single unified embedding architecture with six ratified positions:

| Decision Point | Decision |
|---|---|
| Embedding storage | Single `embeddings` table — not a vector column per content table |
| Embedding model | OpenAI `text-embedding-3-small` (1536 dimensions) |
| Pipeline trigger | Edge Function on publish event (fast path) + pg_cron recovery (slow path) |
| Context assembly | Server-side only via `build_tutor_context()` — never client-assembled |
| RLS boundary | `embeddings` table gated by section enrollment + access window; SECURITY INVOKER on search function |
| Search API contract | Typed `TutorQueryContextInternal` / `TutorPromptContext` split; versioned via `contextVersion` field |

## Architecture

### AI Tutor Request Flow

```
Student query
      │
      ▼
/api/ai/tutor  (Server Route — Next.js App Router)
      │
      ├── 1. Authenticate: auth.uid()
      ├── 2. build_tutor_context(userId, sectionId)   ← DB function, SECURITY INVOKER
      ├── 3. Access window check (raises 403 if closed or enrollment non-active)
      ├── 4. Generate query embedding  → OpenAI (server-side, use-and-discard)
      ├── 5. search_content_chunks()                  ← DB function, SECURITY INVOKER + RLS
      ├── 6. Assemble prompt (system prompt + context chunks)
      ├── 7. OpenAI streaming completion
      └── 8. Write to ai_query_log (hashed query, no PII)
```

### Embedding Pipeline

```
content_pages INSERT/UPDATE (status = 'published')
      │
      ├── (fast path) DB webhook → Edge Function: generate-embedding
      │       ├── Validate source exists
      │       ├── Chunk Tiptap JSON (≤1200 chars, 150-char overlap, semantic boundaries)
      │       ├── OpenAI text-embedding-3-small per chunk
      │       ├── INSERT INTO embeddings (service role)
      │       ├── UPDATE embedding_jobs (status = 'complete', model_version pinned)
      │       └── UPDATE content_pages (embedding_status = 'complete', chunk_count)
      │
      └── (recovery path) pg_cron every 10 min
              └── Requeue pending >5 min or failed with attempt_count < 3
```

## TypeScript Contract

```typescript
// types/ai.ts

// Internal: assembled server-side, never serialized to any response
export interface TutorQueryContextInternal {
  userId:              string;   // Never exposed to client
  sectionId:           string;
  sectionCode:         string;
  blueprintTitle:      string;
  termName:            string;
  deliveryFormat:      'synchronous' | 'asynchronous' | 'hybrid' | 'self_paced';
  cohortName:          string | null;
  cohortCode:          string | null;
  programTrackName:    string | null;
  programTrackCode:    string | null;
  enrollmentStatus:    'active' | 'suspended' | 'completed';
  accessWindowOpen:    boolean;
  accessWindowEnd:     string;         // ISO 8601 UTC
  contentChunks:       ContentChunk[];
  queryText:           string;
  retrievedAt:         string;         // ISO 8601 UTC
  contextVersion:      string;         // 'v1' — bump on breaking prompt changes
}

// External: safe to log or pass to non-sensitive consumers
export interface TutorPromptContext {
  sectionId:        string;
  blueprintTitle:   string;
  termName:         string;
  deliveryFormat:   string;
  cohortName:       string | null;
  programTrackName: string | null;
  contentChunks:    ContentChunk[];
  contextVersion:   string;
}

export interface ContentChunk {
  chunkId:      string;
  sourceType:   'content_page' | 'discussion' | 'assignment_description';
  sourceId:     string;
  sourceTitle:  string;
  chunkIndex:   number;
  chunkText:    string;
  similarity:   number;   // cosine similarity 0–1
  sectionId:    string;
  publishedAt:  string;
}
```

**`queryEmbedding` is explicitly absent.** Query embeddings are generated in-flight and discarded
immediately after `search_content_chunks()` returns. They are never stored on the context object.

## Context Version Rules

A `contextVersion` bump is required when:
- Any field is **removed** from `TutorQueryContextInternal`
- Any field **changes type**
- The system prompt logic changes in a way that produces **materially different responses** for the same input

A version bump is **not** required for adding new optional fields.

Version bumps are recorded as amendments to this ADR. The `contextVersion` value is logged in
`ai_query_log` to support production debugging and prompt A/B analysis.

## Security

### RLS Boundaries

| Table | Read Access | Write Access |
|---|---|---|
| `embeddings` | Enrolled students, active enrollment + open access window only | Service role only |
| `embedding_jobs` | Admin, manager | Service role only |
| `ai_query_log` | Admin, manager | Service role only |

### Critical Security Constraints (Non-Negotiable)

- `search_content_chunks()` uses `SECURITY INVOKER` — the calling user's RLS context applies
- `search_content_chunks()` calls `check_section_access()` explicitly as a second gate
- `build_tutor_context()` uses `SECURITY INVOKER` — only returns the calling user's own context
- OpenAI API key lives in Edge Function environment variables **only** — never in Next.js server env, never in client bundle, never in any committed file
- Query embeddings are **never persisted** — generated in-flight, used for search, discarded
- Context assembly is **server-side only** — no partial assembly completed client-side
- `userId` lives only in `TutorQueryContextInternal` — never in any HTTP response body

### Enrollment Withdrawal — Immediate Access Revocation

A trigger on `direct_enrollments.status` fires `REFRESH MATERIALIZED VIEW CONCURRENTLY
effective_enrollments` when status changes to `withdrawn` or `suspended`. This closes the access
window within seconds of withdrawal, not on the next pg_cron cycle (up to 15 minutes).

This is a **security requirement**, not a performance optimization. It must ship with Phase 3A.

### Prompt Injection

Course content is placed in clearly delimited user-turn context blocks — never interpolated into
the system prompt. The system prompt explicitly instructs the model to treat content as data, not
instructions. The `chunk_text` field is never raw-interpolated.

### Data Residency

Course content is transmitted to OpenAI for embedding generation under OpenAI's standard API
Data Processing Agreement. Content is not used for model training (paid API accounts, default
settings). Churches with data sovereignty requirements should be flagged for evaluation of the
self-hosted embedding path at Phase 4.

## Consequences

### Positive
- Single source of truth for all AI-queryable content across content types
- RLS boundary is clean — `section_id` scopes every query
- Context assembly is typed, versioned, and server-side testable
- No vector data on `content_pages` — reduces table bloat
- Pipeline is fully observable via `embedding_jobs` table
- Semantic chunking preserves coherence across chunk boundaries
- `contextVersion` enables safe prompt evolution and production debugging

### Negative
- Phase 0 migration must complete and pass QA before either Phase 3 ships
- Edge Function cold start adds ~200ms latency to first embedding after deploy
- OpenAI API dependency — outage affects pipeline (mitigated by retry queue in `embedding_jobs`)
- `ivfflat` index requires rebuild as row count grows (see Ops Runbook entry below)
- Polymorphic `source_id` has no DB-level FK — integrity maintained by service-role-only write + nightly orphan cleanup job

### Constraints (Pinned)
- Minimum similarity threshold: `0.72` (configurable constant — not hardcoded in SQL)
- Maximum chunks returned per query: `8`
- Maximum chunk size: `1200` characters with `150`-character sliding window overlap
- Embedding model: `text-embedding-3-small` — version pinned in `embedding_jobs.model_version`
- Model version change requires full re-index and ADR amendment

## Ops Runbook Additions (Phase 3A Deliverable)

### ivfflat Index Monitoring
- `lists` value should approximate `sqrt(row_count)` in the `embeddings` table
- Rebuild schedule: at 10,000 rows → `lists = 100`; at 100,000 rows → `lists = 316`
- A monitoring query must be added to the ops runbook alerting when `row_count > lists^2`
- Owner: The Data Engineer

### Nightly Orphan Cleanup
- pg_cron job `embeddings-orphan-cleanup` runs at 03:00 UTC
- Marks inactive any embedding where the source `content_page` no longer exists
- Owner: The Data Engineer

### Canary Embedding Deployment Check
- A known test phrase is embedded at baseline and stored
- On each deploy, the same phrase is re-embedded and cosine similarity compared to baseline
- If similarity < `0.99`, the check fails — indicates model version change
- Failure blocks deploy and triggers re-index procedure
- Owner: The Engineer

## Rejected Alternatives

| Alternative | Reason Rejected |
|---|---|
| Vector column per content table | Divergent strategies per content type; no unified search; separate RLS per column |
| Lazy on-read embedding generation | High latency on first student query; no pipeline observability |
| pg_cron batch embedding only | Delay between publish and AI availability; event fidelity lost |
| OpenAI `text-embedding-3-large` (3072 dims) | ~3x cost, ~2x storage; marginal quality gain for formal prose (ministry/theological content) |
| Self-hosted Ollama / Nomic Embed | Operational overhead (GPU infra) exceeds benefit at Phase 3 scale |
| Supabase native AI inference | Same infra dependency as self-hosted Ollama; rejected on same grounds |
| Client-assembled tutor context | Security violation — student context assembled client-side is trivially tampered with |
| SECURITY DEFINER on search function | Bypasses caller's RLS context — exactly wrong for a user-scoped search boundary |

## Phase Gate

**Phase 3A is blocked until all of the following are complete:**

- [ ] Migration `20240601000038_unify_embeddings.sql` applied to staging and verified
- [ ] `embeddings` table RLS policies pgTAP tested — including INSERT denial for authenticated roles
- [ ] Three RLS isolation tests passing: own-section only; two-section student sees queried section only; withdrawn student locked out within 60 seconds
- [ ] ivfflat monitoring added to ops runbook
- [ ] Nightly orphan check pg_cron job verified
- [ ] Immediate withdrawal refresh trigger tested
- [ ] `TutorQueryContextInternal` / `TutorPromptContext` types merged to `types/ai.ts`

**Phase 3B is blocked until Phase 3A is in production.**

**Feature ships to production only when:**

- [ ] `build_tutor_context()` pgTAP tested
- [ ] Prompt injection test cases passing
- [ ] Canary embedding deployment check wired into CI
- [ ] Response inspection test: `userId` and vector values absent from all HTTP responses
- [ ] `ai_query_log` table migration applied
- [ ] `embedding_status` badge ("AI Ready" / "Indexing" / "Failed") visible in instructor content list
- [ ] Product FAQ published: course content not used for OpenAI model training

## Review Date

Phase 4 planning session. Evaluate:

- Self-hosted embedding model (Ollama, Nomic) when OpenAI embedding cost exceeds $200/month
- HNSW index migration from IVFFlat when `embeddings` table exceeds 100,000 rows
- Multi-section AI context (student enrolled in multiple concurrent courses)
- Instructor-facing AI analytics ("What concepts are students querying most?")
- Cross-course concept linking via embedding similarity
