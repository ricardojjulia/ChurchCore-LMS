# COUNCIL-2026-009: Drag-and-Drop Course Builder

**Status:** RATIFIED — Pending Implementation
**Date:** 2026-06-21
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)
**Related:** ADR-2026-005 (@dnd-kit/core library selection)
**Tags:** builder, drag-and-drop, course-blocks, sort-order, accessibility

---

## User Story

**As a teacher**, I want to drag course blocks into a different order in the course builder
so that I can restructure my course without repeatedly clicking "move up" / "move down".

**Acceptance Criteria:**
1. In the course builder block list, blocks can be reordered by drag-and-drop
2. The reorder is persisted to the database immediately after drop (not on a separate "save" button)
3. Keyboard drag mode works (Tab to block, Space to grab, Arrow keys to move, Space to drop)
4. Move-up / move-down buttons remain as an accessible fallback
5. `sort_order` is updated in the database for all affected blocks in a single server action call

**Edge Cases:**
- Drag during a pending save (debounce: ignore new drag until save completes)
- Single block in list → drag handles shown but no-op if dropped in same position
- Network error during save → show error toast, revert optimistic UI update

**Out of Scope:**
- Drag blocks between courses
- Drag blocks into nested groups (this sprint)
- Reorder other entities (terms, sections, cohorts)

---

## Gap Being Addressed

The builder (`src/app/courses/[id]/build/`) has blocks listed in `sort_order` order with
move-up/move-down buttons. This is functional but slow for large courses. `course_blocks.sort_order`
is a FLOAT column indexed for ordering. @dnd-kit/core (ratified in ADR-2026-005) is the chosen library.

---

## Wildcard Pre-Audit

**Float sort_order means no renumbering.** When a block moves from position 3 to position 1,
the new sort_order = (sort_order[0] + sort_order[1]) / 2 — midpoint insertion. This avoids
updating all rows after a reorder. The only risk is precision exhaustion after ~50 midpoint
insertions, but at church-scale content (< 200 blocks per course) this is not a practical concern.

**Optimistic UI, then persist.** Update the local state immediately on drop (smooth UX), then
call the server action in the background. If the server action fails, revert the state and show
an error. Do NOT wait for the server before updating the UI.

**Read the builder before writing code.** The builder at `src/app/courses/[id]/build/` may be
a Server Component that renders a Client Component list, or it may be fully client-side. The DnD
list must be a Client Component. If the builder page is a Server Component, the block list already
delegates to a Client Component — find it and add @dnd-kit to it.

**`@dnd-kit/core` AND `@dnd-kit/sortable` are both needed.** The `@dnd-kit/sortable` package
provides the `SortableContext` and `useSortable` hook. Install both.

## Positions

### 🏛️ The Architect

Architecture: The block list in the builder is wrapped in `<DndContext>` + `<SortableContext>`.
Each block row is a `useSortable` item. On `onDragEnd`, compute new `sort_order` values and call
`reorderCourseBlocks` server action.

New server action: `reorderCourseBlocks({ courseId, reorderedIds })` where `reorderedIds` is the
new ordered array of block IDs. The action computes evenly-spaced `sort_order` values (1, 2, 3... or
computed midpoints) and updates all changed rows.

Simplest approach for `sort_order` computation: after drag, assign sort_order = index + 1 (integer,
not midpoint) for all blocks in the list. This is safe since `sort_order` is FLOAT. Avoids precision
edge cases entirely.

### 🧭 Product Manager

UX: Each block row has a drag handle (⠿ icon) on the left. Cursor changes to `grab` on hover.
On drag start, the dragged block gets a visual ghost. On drop, the list reorders smoothly.
Save indicator: small spinner / "Saving..." text appears during server action; "Saved" on success.

### ⚙️ The Engineer

Install: `npm install @dnd-kit/core @dnd-kit/sortable`
(Add to package.json dependencies)

New/modified client component: wherever the block list renders in the builder — wrap with:
```tsx
import { DndContext, closestCenter, KeyboardSensor, PointerSensor, useSensor, useSensors } from '@dnd-kit/core'
import { SortableContext, sortableKeyboardCoordinates, useSortable, verticalListSortingStrategy } from '@dnd-kit/sortable'
import { CSS } from '@dnd-kit/utilities'

// Each block row:
function SortableBlock({ block, ... }) {
  const { attributes, listeners, setNodeRef, transform, transition, isDragging } = useSortable({ id: block.id })
  const style = { transform: CSS.Transform.toString(transform), transition, opacity: isDragging ? 0.5 : 1 }
  return (
    <div ref={setNodeRef} style={style}>
      <button {...attributes} {...listeners} className="cursor-grab p-1 text-muted-foreground">
        ⠿
      </button>
      {/* existing block row content */}
    </div>
  )
}
```

`onDragEnd` handler:
```typescript
function handleDragEnd(event: DragEndEvent) {
  const { active, over } = event
  if (!over || active.id === over.id) return
  setBlocks(prev => {
    const oldIdx = prev.findIndex(b => b.id === active.id)
    const newIdx = prev.findIndex(b => b.id === over.id)
    const reordered = arrayMove(prev, oldIdx, newIdx)
    // Persist asynchronously
    reorderCourseBlocks({ courseId, reorderedIds: reordered.map(b => b.id) })
    return reordered
  })
}
```

Server action in `src/app/actions/learning.ts` (or relevant actions file):
```typescript
export async function reorderCourseBlocks({
  courseId, reorderedIds
}: { courseId: string; reorderedIds: string[] }): Promise<{ error?: string }>
```
- Auth + role check (teacher/admin/manager)
- Verify course belongs to caller's org
- Service client: update each block's sort_order = index + 1 in a single transaction (use Promise.all or a loop)
- Return `{}` on success

### 🛡️ Security Lead

`reorderCourseBlocks` must:
1. Verify caller is teacher/admin/manager (not a student)
2. Verify the course's org_id matches the caller's org_id (prevent cross-tenant block reordering)
3. Verify all block IDs in `reorderedIds` belong to the specified courseId (prevent block injection)

Use service client for the UPDATE (bypasses per-row RLS, but org ownership is verified in step 2-3).

### 🔬 QA Lead

Manual tests:
1. Drag block 3 above block 1 → order updates → refresh page → order persisted
2. Keyboard drag: Tab to handle, Space, Arrow up, Space → block moves up
3. Drag fails (mock network error) → UI reverts to original order
4. Move-up button still works after DnD is added

### 🗄️ Data Engineer

`course_blocks.sort_order` is FLOAT. Assigning sequential integers (1, 2, 3...) after each
reorder is correct and avoids precision drift. No migration needed.
Index `idx_course_blocks_course_order(course_id, sort_order)` already exists from migration 002.

## Council Verdict (6/6 Unanimous)

| Decision | Verdict |
|---|---|
| Library | @dnd-kit/core + @dnd-kit/sortable (ADR-2026-005) |
| sort_order strategy | Sequential integers after each reorder (no midpoint) |
| Persistence | Immediate on drop (optimistic UI + background save) |
| Keyboard support | @dnd-kit native keyboard mode |
| Move buttons | Retained as fallback |
| Cross-tenant validation | courseId must match caller's org before any UPDATE |
