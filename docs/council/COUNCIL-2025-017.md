## Positions

### üèõÔ∏è The Architect

**Email (Item 1):** Two distinct email needs with different solutions:

Path A ‚Äî **Auth emails (invite, magic-link, password reset):** These are sent by
Supabase Auth. Resend can receive them via Supabase's custom SMTP setting in the
project dashboard (Auth ‚Üí SMTP Settings). Point Supabase at
`smtp.resend.com:587`, auth with Resend API key, from address set to the org's
`email_from_name` from `organizations.settings.branding`. No code change. One
dashboard config.

Path B ‚Äî **Transactional emails (digest, weekly summary, badge award, enrollment
confirmation):** These are sent by the application, not Supabase Auth. Use the
already-installed `resend` package via a shared `src/lib/email.ts` wrapper. All
templates are React Email components. The `/api/digest` route is already structured
to generate content ‚Äî it just needs a `sendEmail()` call at the end.

The architecture is clean: Supabase Auth SMTP ‚Üí Resend for system auth emails.
`resend` package ‚Üí Resend API for application-triggered emails. One Resend account,
two paths.

**Stripe (Item 8):** Minimal viable billing architecture: one Stripe product per plan
tier (Starter, Growth, Enterprise). Webhook handler at `/api/stripe/webhook` updates
`organizations.status` and `organizations.plan` on `customer.subscription.created`,
`invoice.payment_succeeded`, `invoice.payment_failed`, and
`customer.subscription.deleted`. The existing Edge Function `expire-trials` already
handles the trial-to-suspended state machine ‚Äî Stripe just needs to trigger
`active` ‚Üí `suspended` on payment failure and `trial` ‚Üí `active` on first payment.
No new DB columns needed; the lifecycle columns from migration `20260618200100` cover
all states.

**Self-registration (Item 7):** The `organizations` table has a `slug` column
(added in migration `20240601000002`, unique constraint). The platform admin already
sets the slug when creating a tenant (visible in `tenants/new/page.tsx`).
`/join/[slug]` is a public page ‚Äî no auth required to land, but user must create an
account or sign in to complete enrollment. The flow: land on `/join/[slug]` ‚Üí
read org by slug ‚Üí show org name/branding ‚Üí "Sign up / Sign in" ‚Üí after auth,
run `enrollSelf(orgId)` server action ‚Üí redirect to `/dashboard`. This requires one
new public RLS policy: `SELECT` on `organizations` for `anon` role where `status =
'active'` (so suspended orgs don't accept new sign-ups).

### üß≠ Product Manager

**Priority re-ranking:** The Wildcard's re-ordering is correct. CSP and Storage bucket
RLS are higher priority than rate limiting because they are active security postures
(bucket is public now), whereas rate limiting is a cost risk that only materializes
at scale.

**Email UX details that matter:**
- Invite email subject: `"[Church Name] has invited you to ChurchCore LMS"`
- Weekly summary email should be opt-in per user (preference in `profiles.settings`),
  not blast-to-all. Add a boolean `notifications.weekly_summary` to the profile
  settings object before wiring the Edge Function.
- Badge award notification email: "You earned the [badge name] badge!" ‚Äî high
  engagement. Ship with the badge award flow, not later.

**Self-registration UX:** The `/join/[slug]` page must show the org's branding
(primary color, logo_url from `organizations.settings.branding`) before the user
creates an account. This reduces drop-off. The org name and a "Join [Name]" CTA
should be visible immediately, no auth required.

**Stripe pricing gates:** The feature flag system (`organizations.settings.features`)
already exists. Wire plan tier to feature flags on subscription activation:
- Starter: `courses, reporting` enabled; `ai_tutor, hq` disabled
- Growth: all features enabled; seat limit 500
- Enterprise: all features enabled; no seat limit; custom branding
No new infrastructure needed ‚Äî just set `settings.features` in the Stripe webhook
handler based on the plan product.

### ‚öôÔ∏è The Engineer

**CSP headers (fastest ‚Äî do first):**
`next.config.js` is empty. Add a `headers()` async function returning one rule for
`source: '/(.*)'`. Required allowlists given the codebase:
- `script-src`: `'self'` + `'unsafe-inline'` (Tailwind JIT, Next.js hydration)
- `frame-src`: `youtube.com`, `www.youtube.com`, `player.vimeo.com`
  (VideoPlayer.tsx renders iframes for these)
- `connect-src`: `'self'`, Supabase project URL, `api.openai.com`
- `img-src`: `'self'`, Supabase Storage URL, `data:`
- `style-src`: `'self'`, `'unsafe-inline'` (Tailwind)
Do NOT use `'unsafe-eval'` ‚Äî no eval usage detected. Tight CSP.

**Storage bucket RLS (4 hours):**
The gap: `content-images` bucket is public. Path structure is
`${auth.uid()}/${timestamp}.ext`. Auth-scoped but not org-scoped.
Fix: add Supabase Storage RLS policies to `content-images` bucket. `INSERT` and
`SELECT` policies check that the path's first segment matches `auth.uid()` (already
true) AND that the user's org is not suspended. For org-scoped reads, the path should
be changed to `${org_id}/${auth.uid()}/${timestamp}.ext` so cross-tenant guessing
requires knowing another org's UUID (which is not exposed in the UI). Update the
upload route at `src/app/api/upload/image/route.ts` to use the new path structure.
Add a migration for the RLS policies on the bucket.

**Rate limiting (4 hours):**
No rate limiting packages are installed. Options:
- Vercel KV (Redis, native Vercel integration) ‚Äî requires Vercel Pro plan
- Upstash Redis (per-request pricing, no plan requirement) ‚Äî preferred for cost control
- In-memory with `node-cache` ‚Äî NOT suitable (no persistence across Edge invocations,
  no shared state across instances)

Recommendation: **Upstash Redis**. Free tier is sufficient for beta.
`@upstash/ratelimit` + `@upstash/redis` packages. Per-user limit: 20 requests/minute
on `/api/ai/tutor`, 5/minute on `/api/ai/weekly-summary`. Identify user by JWT `sub`
(not IP, to allow office WiFi NAT). Return `429` with `Retry-After` header.

**Resend wiring (1 day):**
`resend` is installed. Create `src/lib/email.ts`:
```typescript
import { Resend } from 'resend'
export const resend = new Resend(process.env.RESEND_API_KEY)
export async function sendEmail({ to, subject, react }) {
  return resend.emails.send({ from: process.env.EMAIL_FROM!, to, subject, react })
}
```
Email templates live in `src/emails/` as React Email components. Ship three templates
with launch: `InviteEmail`, `EnrollmentConfirmationEmail`, `WeeklySummaryEmail`.
For Supabase Auth system emails (invite, password reset), configure custom SMTP in
Supabase dashboard ‚Üí Auth ‚Üí SMTP Settings:
- Host: `smtp.resend.com` Port: `587`
- Username: `resend` Password: `[RESEND_API_KEY]`
- Sender name: from `organizations.settings.branding.email_from_name`

**Cross-tenant RLS penetration tests (1 day):**
Follow the pattern in `src/tests/e2e/critical-path.test.ts`. Requires two seed orgs
with two users, each in a different org. Service role creates the data; user-JWT
queries must return empty for cross-tenant rows. Test file:
`src/tests/e2e/rls-isolation.test.ts`. Required env vars are already defined:
`TEST_SUPABASE_URL`, `TEST_SUPABASE_ANON_KEY`, `TEST_SUPABASE_SERVICE_ROLE_KEY`.
Need two additional: `TEST_USER_A_EMAIL` / `TEST_USER_A_PASSWORD` (Org A user) and
`TEST_USER_B_EMAIL` / `TEST_USER_B_PASSWORD` (Org B user). Test 10 high-risk tables:
`courses`, `modules`, `submissions`, `hq_tasks`, `enrollments`, `profile_badges`,
`group_posts`, `ai_query_log`, `admin_audit_log`, `user_audit_log`.

**Self-registration `/join/[slug]` (1 day):**
New public page at `src/app/join/[slug]/page.tsx`. Reads org by slug using service
client (anonymous access). Shows org name + branding. After Supabase Auth sign-up:
trigger adds profile via `handle_new_user` trigger (already exists). New server action
`enrollSelf(orgId)` inserts into `profile_roles` for the org. Redirect to
`/dashboard`. No new migrations needed ‚Äî existing tables cover this. One new RLS
policy: `SELECT` on `organizations` where `slug = $1 AND status = 'active'` granted
to `anon`.

**Stripe (3 days):**
Install `stripe` package. New files:
- `src/lib/stripe.ts` ‚Äî Stripe client
- `src/app/api/stripe/webhook/route.ts` ‚Äî webhook handler (verify signature with

- `src/app/platform/tenants/[id]/billing/page.tsx` ‚Äî billing UI (plan, seats, invoice)
- `src/app/api/stripe/create-checkout/route.ts` ‚Äî creates checkout session
No new DB columns ‚Äî use existing `organizations.plan`, `organizations.status`.
Webhook events handled: `checkout.session.completed` ‚Üí set `status='active'`,
`invoice.payment_failed` ‚Üí set `status='suspended'`,
`customer.subscription.deleted` ‚Üí set `status='suspended'`,
`customer.subscription.updated` ‚Üí update `plan`.

### üõ°Ô∏è Security Officer

**Storage bucket ‚Äî the active cross-tenant leak:**
The bucket `content-images` stores files at `{auth_uid}/{timestamp}.ext`.
The public URL is predictable if an attacker knows any other user's `auth_id`.
`auth_id` values appear in `admin_audit_log.actor_id` and could leak through verbose
error messages. Fix the path structure to `{org_id}/{auth_uid}/{timestamp}.ext` and
add bucket RLS:
```sql
-- Only org members can INSERT into their own org's folder
-- Only org members can SELECT from their org's folder
-- Platform admins bypass via service role
```
This is a migration + route code change.

**CSP ‚Äî `frame-src` must be explicit:**
`VideoPlayer.tsx` embeds YouTube (`youtube.com`) and Vimeo (`player.vimeo.com`).
Without explicit `frame-src` allowlisting, the CSP will block all video playback in
browsers. This is a functional regression risk if CSP is deployed incorrectly.
Include `frame-src youtube.com www.youtube.com player.vimeo.com` explicitly.
Also: the `/api/ai/*` routes connect to `api.openai.com` ‚Äî include this in
`connect-src`. The Supabase Realtime WebSocket goes to `wss://*.supabase.co` ‚Äî
include in `connect-src`.

**Rate limiting ‚Äî JWT sub, not IP:**
Rate limit by JWT `sub` (user UUID), not IP address. IP-based limits fail for:
- Office NAT (many users behind one IP)
- VPN users
- Mobile users switching IPs
Extract `sub` from the session via `supabase.auth.getUser()` ‚Äî this is already done
at the top of every AI route. Use the user UUID as the rate limit key.

**Stripe webhook ‚Äî signature verification is mandatory:**
The Stripe webhook handler must call `stripe.webhooks.constructEvent()` with the
raw body (not parsed JSON) and `STRIPE_WEBHOOK_SECRET`. Do NOT parse the body as JSON
before verifying. Next.js App Router: disable body parsing for the webhook route by
reading `req.text()` before calling `constructEvent`. This prevents webhook spoofing.

**Auth email security:** When Supabase sends invite emails via Resend SMTP, the
invite token is in the URL. Resend's email delivery tracking must be disabled or
the token becomes visible in Resend's dashboard to anyone with access. Set Resend
tracking OFF (`open_tracking: false`, `click_tracking: false`) for auth emails.

### üÉè The Wildcard

**What the council missed or underweighted:**

1. **Environment variable inventory is unmanaged.** The codebase will need:
   `RESEND_API_KEY`, `EMAIL_FROM`, `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`,
   `STRIPE_PUBLISHABLE_KEY`, `UPSTASH_REDIS_REST_URL`, `UPSTASH_REDIS_REST_TOKEN`,
   `SENTRY_DSN`, `SENTRY_AUTH_TOKEN`.
   None of these exist yet. Before writing any code, create `src/env.ts` (a validated
   env schema using plain TypeScript ‚Äî no Zod needed, just `process.env.X ?? throw`
   guards) so missing env vars fail loudly at startup rather than silently at runtime.
   This costs 30 minutes and prevents 3 hours of "why is Resend not sending" debugging.

2. **The weekly-summary AI route is a Resend dependency,** not a Stripe dependency.
   Once Resend is wired, the Edge Function for weekly digest is ~2 hours of work:
   one CRON Edge Function that calls `get_my_academic_performance` per user and fires
   `sendEmail()`. This should be in the Resend prompt, not deferred to Phase 2.

3. **The Storage RLS policies need a migration file, not just a dashboard config.**
   Supabase Storage RLS policies can be written as SQL in a migration using
   `storage.objects` table policies. Otherwise the policy lives only in the dashboard
   and will be lost on a reset or staging-environment setup. Create migration
   `20260618200700_storage_rls.sql`.

4. **`handle_new_user` trigger needs to set `org_id` for self-registered users.**
   When a user self-registers via `/join/[slug]`, they go through Supabase Auth signup.
   The `handle_new_user` trigger fires and creates their `profiles` row. But it won't
   know the `org_id` unless it's passed in `raw_user_meta_data`. The `/join/[slug]`
   signup call must pass `{ data: { org_id: orgId } }` to `supabase.auth.signUp()`.
   The trigger must read `raw_user_meta_data.org_id` and set `profiles.org_id`. If
   the trigger ignores this field today, self-registration silently creates orphan
   profiles with no org ‚Äî and all RLS blocks them. This is the single highest-risk
   integration point in the self-registration feature.

5. **Rate limiting needs a `429` response with `Retry-After` header,** but the AI
   tutor uses streaming (`text/event-stream`). If a rate limit fires mid-stream, the
   client receives a broken stream. Check rate limit BEFORE opening the OpenAI stream.
   The check must be synchronous (before `await openai.chat.completions.create()`).

6. **Stripe webhook idempotency:** Stripe sends webhooks at least once ‚Äî retries are
   possible. The webhook handler must be idempotent. Use `stripe_event_id` as a
   deduplication key in a new `stripe_events` table (or `platform_audit_log` with
   `action = 'stripe_webhook'` and `metadata.stripe_event_id`). If the same event is
   received twice, skip processing.

7. **`join/[slug]` needs a CAPTCHA or at minimum honeypot.** A public sign-up page with
   no bot protection will be spammed. Cloudflare Turnstile is free and works with Next.js.
   Without it, every active tenant's org-slug becomes a spam account creation vector.
