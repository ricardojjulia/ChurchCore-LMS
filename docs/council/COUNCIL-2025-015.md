# COUNCIL-2025-015: Phase 2 Complete Tenant Isolation

**Date:** 2026-06-18
**Status:** Ratified — Unanimous (7/7)
**Resulted in:** ADR-2025-009, PHASE2-ISOLATE-IMPLEMENT.md
**Tags:** multi-tenancy, RLS, security, org_id, phase-2

---

## Question

Phase 1 (COUNCIL-2025-014) achieved org_id isolation on 16 tables. A post-Phase-1 audit
identified 20+ additional tables whose RLS checks role but not org_id. A staff user from
Tenant A can currently read course content, HQ project data, academic structure, and
AI/analytics logs belonging to Tenant B. Classify every remaining un-isolated table as
per-tenant or platform-global, and produce a definitive migration strategy to close all
remaining cross-tenant leakage vectors.

---

## Positions

### 🏛️ The Architect

Phase 1 proved the pattern works. The only question here is classification. My read:

**Per-tenant** (org_id + org-scoped RLS required):
`modules`, `course_blocks`, `content_pages`, `submissions` (legacy),
`hq_tasks`, `hq_decisions`, `hq_risks`, `hq_sessions`,
`academic_terms`, `program_tracks`, `course_blueprints`, `course_sections`,
`access_windows`, `meeting_schedules`,
`global_cohorts`, `cohort_members`, `cohort_section_enrollments`, `direct_enrollments`,
`enrollment_jobs`, `enrollment_audit_log`,
`embeddings`, `embedding_jobs`, `ai_query_log`,
`analytics_events`, `admin_audit_log`, `user_audit_log`,
`section_groups`, `section_group_members`, `group_threads`, `group_posts`,
`profile_badges`

**Platform-global** (no org_id, intentionally shared):
`badges` (system-defined achievement definitions — orgs award but do not define them),
`block_types` (LMS content type registry), `role_permissions` (platform role schema),
`system_health_checks` (platform infrastructure monitoring)

Backfill pattern is established and proven. 3-migration sequencing avoids locking any
single migration for too long.

### 🧭 Product Manager

Align on classification before code touches anything.

`academic_terms` and `program_tracks`: these look like shared reference data ("Fall 2026",
"Bachelor of Theology") but each client runs on their own calendar and their own programs.
Grace Church's "Fall Term 2026" is not City Ministry's. **Per-tenant.**

`course_blueprints` and `course_sections`: blueprint is an org's course template library.
Section is a specific scheduled run. Definitively per-tenant.

`hq_tasks / hq_decisions / hq_risks`: this is organizational project governance IP. If
Tenant B's admin can read Tenant A's strategic decisions and risk register, that is a
serious confidentiality breach. **Highest priority gap.**

`badges`: unanimous agreement — system-level XP/achievement definitions are platform-global.
Custom org badges are Phase 3. `profile_badges` (award records) must be org-scoped since
they record which user in which org context earned what.

### ⚙️ The Engineer

**Backfill path by table group:**

- **Via `course_id`:** `modules`, `course_blocks`, `content_pages`, `submissions`
  → `courses.org_id` (courses now have org_id after Phase 1 migration 300)

- **Via `created_by → auth.users → profiles.auth_id → profiles.org_id`:**
  `hq_tasks`, `hq_decisions`, `hq_risks`, `global_cohorts`, `academic_terms`,
  `program_tracks`, `course_blueprints`, `course_sections`

- **Via `user_id → auth.users → profiles.auth_id → profiles.org_id`:**
  `hq_sessions`, `cohort_members`, `direct_enrollments`, `ai_query_log`,
  `analytics_events`, `profile_badges`

- **Via `section_id → course_sections → created_by → profiles.org_id`:**
  `access_windows`, `meeting_schedules`, `section_groups`, `cohort_section_enrollments`,
  `enrollment_jobs`, `enrollment_audit_log`

- **Via `group_id → section_groups → section_id → course_sections → org`:**
  `section_group_members`, `group_threads`, `group_posts`

- **Via polymorphic `source_id` (source_type IN course_block, content_page):**
  `embeddings`, `embedding_jobs`
  → join to `course_blocks.course_id → courses.org_id`
  or `content_pages.course_id → course_blueprints.created_by → profiles.org_id`

- **Via `user_id → profiles` (bare uuid, no FK):**
  `admin_audit_log`, `user_audit_log`, `enrollment_audit_log`
  → join `profiles` on `uid = user_id`

**3-migration split:**
- `20260618200400`: course content group (modules, course_blocks, content_pages, submissions, hq_*)
- `20260618200500`: academic structure (terms, tracks, blueprints, sections, cohorts, enrollment tables, access windows, meeting schedules, section groups/threads/posts)
- `20260618200600`: AI/analytics/audit (embeddings, embedding_jobs, ai_query_log, analytics_events, admin_audit_log, user_audit_log, enrollment_audit_log, profile_badges)

**RLS pattern** (unchanged from Phase 1):
```sql
USING (public.is_platform_admin() OR public.current_user_org_id() = org_id)
```

Staff-scoped write policies additionally check `current_user_role()`.

**Indexes:** Add `CREATE INDEX IF NOT EXISTS idx_<table>_org ON <table>(org_id)` for every
new org_id column (hot RLS path).

### 🛡️ Security Officer

Priority ranking of gaps (highest risk first):

1. **`hq_tasks / hq_decisions / hq_risks`** — strategic org IP. An authenticated teacher
   from any org can currently `SELECT *` these via the API. Must ship in Migration 301.

2. **`embeddings`** — contains the full plaintext of course blocks and content pages. The
   most sensitive cross-tenant IP after raw student records. Ship in Migration 303.

3. **`ai_query_log`** — contains user queries sent to OpenAI. Contains implicit PII (user
   thought patterns, questions). Must be org-scoped.

4. **`admin_audit_log` / `user_audit_log`** — behavioral surveillance data. If leaked
   cross-tenant, violates GDPR Article 5 (purpose limitation).

5. **`hq_sessions`** — current user-scoped policy (`user_id = auth.uid()`) prevents
   cross-user reads but the `admins read all` policy (`current_user_role() = 'admin'`)
   allows any org's admin to read any other org's session history. Must add org scope.

6. **All enrollment tables** — enrollment status is sensitive educational record data
   (FERPA equivalent). Must be org-scoped.

Additional constraint: `badges` table definitions MUST remain platform-global. Do not add
org_id to `badges`. `profile_badges` links users to badges in an org context — add org_id.

No security regressions: every new policy must include the platform admin bypass:
`public.is_platform_admin() OR public.current_user_org_id() = org_id`.

### 🔬 The Tester

For each of the 30 tables being isolated, require these RLS test pairs before marking
the migration complete:

| Test | Expected |
|------|----------|
| Same-org authenticated user reads own row | ✅ allowed |
| Cross-org authenticated user (same role) reads foreign row | ❌ blocked |
| Suspended-tenant user reads own row | ❌ blocked (via tenant_active) |
| Platform admin reads any row | ✅ allowed |
| Unauthenticated request | ❌ blocked |

Special cases:
- `hq_sessions`: verify admin from Tenant A cannot read Tenant B user sessions
- `embeddings`: verify source_type filter prevents orphan backfill rows from leaking
- `enrollment_audit_log`: verify bare `user_id uuid` backfill path is correct
  (not a FK, must match `profiles.uid` not `profiles.auth_id`)

Accept "NOTICE: policy does not exist, skipping" — these are safe. Treat any `ERROR`
as a blocking failure.

### 🚀 DevOps Officer

All migrations MUST be idempotent:
- `ADD COLUMN IF NOT EXISTS`
- `DROP POLICY IF EXISTS` before every `CREATE POLICY`
- `CREATE INDEX IF NOT EXISTS`

The backfill UPDATEs can be slow on large datasets. Order within each migration:
ADD COLUMN → backfill from linked table → default remaining → SET NOT NULL → rewrite RLS → add index.

Default-remaining rows (after failed backfill joins) MUST fall back to first org only for
single-tenant dev instances. In production, any NULL after primary backfill indicates data
integrity issue and should be investigated — the DEFAULT fallback is a safety net only.

Push with: `supabase db push` after each migration file is ready. Test one at a time.

### 🃏 The Wildcard

**Gaps not addressed in any agent's list:**

1. **`content_pages`** — course blueprint content pages (`course_id` references
   `course_blueprints.id`). Not in the Architect's list. Medium risk — contains lesson
   text. Include in Migration 301.

2. **`embedding_jobs`** — job queue feeding embedding generation. Has `source_id` and
   `section_id`. If not org-scoped, a cross-tenant user could observe Tenant B's embedding
   pipeline status. Include in Migration 303.

3. **`enrollment_audit_log.user_id`** is a bare `uuid` (not a FK). The Tester is right to
   flag this. Backfill must join `profiles` on `uid = user_id`, not `auth_id = user_id`.

4. **`section_groups` cascade gap** — `section_group_members`, `group_threads`, and
   `group_posts` are all in the Architect's list but the backfill chain is 3–4 hops deep.
   Engineer's backfill path is correct. Verify cascade DELETE still works after org_id NOT
   NULL is set (it will — cascades fire before constraints are checked).

5. **`analytics_events`** — has 5 org_id column references in migrations but zero
   `CREATE POLICY` statements referencing it. It appears org_id exists on the table but
   RLS was never written. Verify and add missing policies in Migration 303.

6. **`report_definitions` / `report_artifacts`** — has 6 org_id refs in migrations
   (from the reporting core tables migration 20250710090100). Likely already correctly
   scoped, but worth verifying RLS enforcement before signing off. Flag for verification,
   not a new migration.

7. **`hq_sessions` `admins read all` policy** — Architect listed it as per-tenant but
   the current policy body `USING (current_user_role() = 'admin')` grants cross-tenant
   admin reads. Security Officer correctly identified this as a live gap. Confirm it is
   included in Migration 301 policy rewrite.

8. **`submissions` (legacy table)** — distinct from `block_submissions` (which was fixed
   in Phase 1). The legacy `submissions` table has `course_id` and `student_id` columns.
   Phase 1 only fixed `block_submissions`. Legacy `submissions` must be fixed in
   Migration 301.

---

## Unanimous Decision (7/7)

**Classification ratified:**
- 30 tables → per-tenant (org_id + org-scoped RLS)
- 4 tables → platform-global (badges, block_types, role_permissions, system_health_checks)
- `report_definitions` / `report_artifacts` → verify existing RLS, no new migration needed

**3-migration approach adopted:**
- `20260618200400` — Course content + HQ tables
- `20260618200500` — Academic structure
- `20260618200600` — AI/analytics/audit/profiles

**Wildcard additions 1–8 all adopted unanimously.**
