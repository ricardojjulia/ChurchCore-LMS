# COUNCIL-2026-001: Guardian Email Notification Bridge

**Date:** 2026-06-20
**Status:** Ratified — Unanimous (7/7)
**Tags:** guardian, email, notifications, edge-function, resend, db-trigger

---

## Question

The guardian portal at `/guardian` is a fully functional read-only dashboard (levels, progress,
grades, certificates). But a guardian who must actively log in to check their child's progress
stops checking within weeks. The competitive moat COUNCIL-2025-016 identified is automatic
push-to-inbox alerts, not a portal that waits to be visited.

What is the minimal, correct architecture for notifying guardians by email when:
1. Their ward completes a course (cert issued or `course_enrollments.status` → `completed`)
2. Their ward earns a badge (`profile_badges` INSERT)
3. (Optional/deferred) Their ward falls below a progress threshold

## Wildcard Pre-Audit (fires before deliberation)

**Inventory of what already exists — read the code, not assumptions:**

- `guardian_links` table is live in migration `20240601000033_phase8_delivery.sql`.
  Schema: `guardian_uid UUID`, `student_uid UUID`, both FKs to `profiles(uid)`.
  Unique constraint on `(guardian_uid, student_uid)`.

- `get_guardian_students()` returns all students linked to the calling guardian (RPC, SECURITY DEFINER).
  `get_guardian_student_overview()` returns progress detail for one student.
  Both functions verify the guardian↔student link before returning data.

- `course_enrollments.status` is the source of truth for completion. The `generate-certificate`
  Edge Function fires from a `course_enrollments` status update (check its trigger).

- `profile_badges` table exists (added in `20260618200600_phase2_ai_audit_rewards_rls.sql`).
  Badge awards happen as INSERTs into this table.

- `src/lib/email.ts` has a working `sendEmail()` wrapper using Resend. Three relevant React Email
  templates already exist: `BadgeAwardEmail.tsx`, `EnrollmentConfirmationEmail.tsx`, and
  `WeeklySummaryEmail.tsx`. The `BadgeAwardEmail` template is exactly what we need for item 2.

- `weekly-digest` Edge Function exists at `supabase/functions/weekly-digest/`. It already
  accesses learner progress data — the guardian notification logic can follow the same pattern.

- `profiles.settings` is a JSONB column. Notification preferences can be added as a nested
  field (`settings.notifications.guardian_emails: boolean`) without a migration (JSONB is schema-free).
  Default: `true` (opt-out, not opt-in — guardians expect to be informed).

**What is NOT built:**
- No DB trigger fires on `course_enrollments` status change.
- No DB trigger fires on `profile_badges` INSERT.
- No mechanism looks up the guardian's email for a given student_uid.
- No Edge Function or API route sends guardian emails.
- No `CourseCompletionGuardianEmail.tsx` template (only `EnrollmentConfirmationEmail.tsx` exists,
  which is different — that's for the student on enrollment, not for the guardian on completion).

**Risk flag:** `profiles.settings` notification preference must default to `true` in the
application layer (not the DB default, which is `null`). If the preference key is absent,
treat it as enabled. The guard is: `settings?.notifications?.guardian_emails !== false`.

## Positions

### 🏛️ The Architect

**Two-layer architecture: DB trigger → Edge Function → Resend.**

Layer 1 — Postgres triggers (the right tool for "something happened"):
- `AFTER UPDATE OF status ON public.course_enrollments` — fires when `status` transitions
  to `'completed'`. Inserts a row into a new `guardian_notification_queue` table:
  `(student_uid, event_type: 'course_completed', payload: {course_title, course_id}, created_at)`.
- `AFTER INSERT ON public.profile_badges` — same queue, `event_type: 'badge_awarded'`, payload
  includes `badge_name`.

Layer 2 — Edge Function `send-guardian-notifications` (scheduled CRON, every 5 minutes):
- SELECTs unprocessed rows from `guardian_notification_queue` (where `sent_at IS NULL`).
- For each row: looks up `guardian_links WHERE student_uid = row.student_uid` to get guardian UIDs.
- Looks up guardian `profiles.email` and `profiles.settings.notifications`.
- Skips guardians with `settings.notifications.guardian_emails = false`.
- Calls `sendEmail()` with the appropriate template.
- Marks the queue row `sent_at = NOW()`.

**Why a queue table instead of sending inline from the trigger?**
Postgres triggers cannot make HTTP calls. Triggers also run inside the committing transaction —
if email sending fails, the DB row rollbacks. The queue decouples reliably and allows retries.

**Alternative considered: pg_net (Supabase's async HTTP extension).**
pg_net can fire HTTP calls from triggers without a queue. However:
- pg_net is not available on all Supabase plans.
- It has no built-in retry on failure.
- Debugging pg_net failures is harder than inspecting a queue table.
Verdict: queue table + scheduled Edge Function is more robust.

### 🧭 Product Manager

**Opt-out model, not opt-in.** Guardians are paying attention to their children's education.
Defaulting to opt-in (no emails until enabled) means most guardians never see a notification.
Default: notifications ON. Provide an unsubscribe link in every email that calls a route
`/api/guardian/unsubscribe?token=[signed-jwt]` to flip the preference without requiring login.

**Email copy:**
- Course completion: `"[Student Name] finished [Course Title]!"` — celebratory, 2 sentences,
  link to guardian portal. Do NOT show grades in the email (privacy — email is insecure transport).
- Badge: `"[Student Name] earned the [Badge Name] badge!"` — single-sentence, badge icon, portal link.

**Frequency guard:** If a student completes 3 courses in one day (bulk import + admin enrollment),
the guardian should not receive 3 separate emails. Add a `debounce` check: consolidate all
events within a 30-minute window into one digest email per guardian per window. Queue rows
with `debounce_until = created_at + '30 minutes'::interval`; Edge Function only processes
rows where `debounce_until < NOW() AND sent_at IS NULL`.

**What NOT to build now:**
- Progress-fall-behind alert (requires defining a threshold and a schedule — separate council).
- SMS/push notifications (separate channel, separate infrastructure).
- Guardian self-service notification preferences UI (add a `/guardian/settings` page in Sprint 3).

### ⚙️ The Engineer

**Migration: `guardian_notification_queue` table.**

```sql
CREATE TABLE IF NOT EXISTS public.guardian_notification_queue (
  id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  student_uid    UUID NOT NULL REFERENCES public.profiles(uid) ON DELETE CASCADE,
  event_type     TEXT NOT NULL CHECK (event_type IN ('course_completed', 'badge_awarded')),
  payload        JSONB NOT NULL DEFAULT '{}',
  debounce_until TIMESTAMPTZ NOT NULL DEFAULT (NOW() + INTERVAL '30 minutes'),
  sent_at        TIMESTAMPTZ,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
ALTER TABLE public.guardian_notification_queue ENABLE ROW LEVEL SECURITY;
-- Only service role can read/write this table (Edge Function uses service client)
CREATE POLICY "guardian_notification_queue: service only"
  ON public.guardian_notification_queue FOR ALL TO authenticated
  USING (false) WITH CHECK (false);
CREATE INDEX idx_guardian_notification_queue_unsent
  ON public.guardian_notification_queue (debounce_until, sent_at)
  WHERE sent_at IS NULL;
```

**Triggers:**

```sql
CREATE OR REPLACE FUNCTION public.queue_guardian_notification_on_completion()
RETURNS TRIGGER LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  IF NEW.status = 'completed' AND (OLD.status IS DISTINCT FROM 'completed') THEN
    INSERT INTO public.guardian_notification_queue (student_uid, event_type, payload)
    SELECT
      NEW.user_id,
      'course_completed',
      jsonb_build_object('course_id', NEW.course_id, 'org_id', NEW.org_id)
    WHERE EXISTS (
      SELECT 1 FROM public.guardian_links WHERE student_uid = NEW.user_id
    );
  END IF;
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_guardian_notification_course_completion
  AFTER UPDATE OF status ON public.course_enrollments
  FOR EACH ROW EXECUTE FUNCTION public.queue_guardian_notification_on_completion();

CREATE OR REPLACE FUNCTION public.queue_guardian_notification_on_badge()
RETURNS TRIGGER LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  INSERT INTO public.guardian_notification_queue (student_uid, event_type, payload)
  SELECT
    NEW.user_uid,
    'badge_awarded',
    jsonb_build_object('badge_name', NEW.badge_name, 'org_id', NEW.org_id)
  WHERE EXISTS (
    SELECT 1 FROM public.guardian_links WHERE student_uid = NEW.user_uid
  );
  RETURN NEW;
END;
$$;

CREATE TRIGGER trg_guardian_notification_badge
  AFTER INSERT ON public.profile_badges
  FOR EACH ROW EXECUTE FUNCTION public.queue_guardian_notification_on_badge();
```

**Edge Function `send-guardian-notifications`:**
- Runs on a 5-minute CRON schedule.
- Uses service role key (from Deno env) to bypass RLS on `guardian_notification_queue`.
- For each unsent row where `debounce_until < NOW()`:
  1. Fetch `guardian_links WHERE student_uid = row.student_uid`
  2. Fetch guardian profiles (email, display_name, settings)
  3. Fetch student display_name from profiles
  4. Fetch course_title or badge_name from payload (look up in `courses` or `profile_badges`)
  5. For each guardian: skip if `settings?.notifications?.guardian_emails === false`
  6. Send email using `sendEmail()` pattern (call Resend API directly — same env var)
  7. UPDATE queue row: `sent_at = NOW()`

**New React Email template: `GuardianCourseCompletionEmail.tsx`**
- Props: `guardianName`, `studentName`, `courseTitle`, `portalUrl`, `unsubscribeUrl`
- Subject: `"${studentName} completed ${courseTitle} 🎓"`

**Unsubscribe route: `src/app/api/guardian/unsubscribe/route.ts`**
- GET handler. Accepts `?token=[signed JWT]`. Decodes token (use `SUPABASE_JWT_SECRET`).
  Token payload: `{ guardian_uid, exp }`. Sets
  `profiles.settings.notifications.guardian_emails = false` for that guardian_uid via
  service client. Returns a plain HTML confirmation page (no React, no layout — user is
  not logged in). Tokens expire in 30 days.

**Unsubscribe token generation:** In `sendEmail()` call, generate a short-lived JWT signed
with `SUPABASE_JWT_SECRET` containing `{ guardian_uid, sub: "guardian-unsub", exp: 30days }`.
Use Node.js `crypto.createHmac` (no new packages needed).

### 🛡️ Security Officer

**No PII in trigger payloads.** The `guardian_notification_queue.payload` column stores
`course_id` and `org_id`, not course_title or email. Titles and emails are looked up at
send time by the Edge Function using the service client. This means the queue table contains
no readable PII — only UUIDs.

**SECURITY DEFINER on triggers is required.** Both trigger functions use `SECURITY DEFINER`
so they can INSERT into `guardian_notification_queue` regardless of who caused the
`course_enrollments` update. The `EXISTS (SELECT 1 FROM guardian_links)` guard prevents
queueing notifications for students with no guardian — avoids unnecessary rows.

**Unsubscribe token security:**
- JWT must be signed with `SUPABASE_JWT_SECRET` (already in env).
- Token payload must include `exp` (30 days), `sub: "guardian-unsub"`, and `guardian_uid`.
- Route must verify: token is valid, sub is exactly `"guardian-unsub"`, token is not expired.
- Do NOT accept `guardian_uid` as a plain query param — IDOR vulnerability.

**Edge Function service key:**
- `SUPABASE_SERVICE_ROLE_KEY` is already available in Supabase Edge Function env as `SUPABASE_SERVICE_ROLE_KEY`.
- Never log the queue row content (could include org_id which is sufficient to scope an attack).

**RLS on `guardian_notification_queue`:** The `USING (false)` policy means no authenticated
user can read or write this table via the anon/user client. Only the service role (Edge Function)
has access. This is the correct posture.

### 🔬 The Tester

**Unit tests (Vitest):**
1. `queue_guardian_notification_on_completion` trigger — seed a student+guardian link, update
   `course_enrollments.status` to `completed`, assert a queue row was inserted with correct payload.
2. Same trigger — update status to `in_progress` (not `completed`), assert no queue row.
3. Same trigger — student has no guardian link, assert no queue row.
4. `queue_guardian_notification_on_badge` — INSERT profile_badge, assert queue row.
5. Unsubscribe route — valid token → 200, sets preference; expired token → 400; wrong sub → 400.

**E2E tests (`src/tests/e2e/guardian-notifications.test.ts`):**
1. Service role sets `course_enrollments.status = 'completed'` for student-a.
2. Assert `guardian_notification_queue` has a row for student-a (requires guardian link to exist in seed).
3. Invoke `send-guardian-notifications` Edge Function manually (POST with service key).
4. Assert `sent_at` is now populated on the queue row.
5. Assert no duplicate row when `status` is updated again (idempotency via `debounce_until`).

**Edge case: guardian with `notifications.guardian_emails = false`.**
Set the preference, trigger a completion, invoke Edge Function, assert `sent_at` IS NULL
(email was skipped, not sent).

### 🃏 The Wildcard

**What the council missed or underweighted:**

1. **The `course_enrollments.user_id` is a `profiles.uid`, not an `auth.users.id`.**
   The trigger function must join `profiles WHERE uid = NEW.user_id` to get the student's
   `auth_id` if needed — but since we only need `guardian_links WHERE student_uid = NEW.user_id`
   and `guardian_links.student_uid` IS a `profiles.uid`, the join is correct. Confirm the
   `profile_badges` table uses `user_uid` (profiles.uid) not `auth_id` before writing the trigger.

2. **Debounce window creates a test timing issue.** The e2e test that checks `sent_at IS NOT NULL`
   will fail if the debounce window hasn't passed. Either:
   - In the e2e test, directly UPDATE the queue row to set `debounce_until = NOW() - INTERVAL '1 second'`
     before calling the Edge Function, OR
   - Expose a `force=true` query param on the Edge Function (service-key-only) that bypasses
     the debounce check. Safer to do the direct UPDATE in the test.

3. **`courses` table does not store title in `course_enrollments`.** The Edge Function must
   JOIN `courses.title WHERE id = queue.payload->>'course_id'` to get the course title for
   the email. This is a cross-table lookup in the Edge Function — straightforward but must
   be included in the implementation prompt.

4. **Unsubscribe link in transactional email must survive email client link-tracking rewrites.**
   Some corporate email clients rewrite all links through their own tracking proxy. A signed
   JWT in the URL will still work (the proxy just forwards the full URL). No additional work needed.

5. **`profile_badges` column for user is likely `user_uid`, not `user_id`.** The trigger
   references `NEW.user_uid` — confirm this against the actual migration before writing the
   trigger function. If the column is `user_id`, the trigger will fail silently.

---

## Council Verdict (7/7 Unanimous)

Build the guardian email bridge in one focused prompt. Architecture: DB triggers → queue table
→ scheduled Edge Function → Resend. Opt-out model. 30-minute debounce per guardian per event window.

| Decision | Verdict |
|---|---|
| Trigger architecture | Queue table (not pg_net) |
| Notification default | Opt-out (enabled by default) |
| Debounce window | 30 minutes |
| Templates needed | New `GuardianCourseCompletionEmail.tsx` + existing `BadgeAwardEmail.tsx` |
| Unsubscribe | Signed JWT, `/api/guardian/unsubscribe` GET route |
| CRON | 5-minute interval on Edge Function |
| Preference storage | `profiles.settings.notifications.guardian_emails` (JSONB, no migration) |
| PII in queue | No — UUIDs only, titles looked up at send time |

## Implementation Prompt

**Files to create:**
- `supabase/migrations/20260620100000_guardian_notification_queue.sql`
- `supabase/functions/send-guardian-notifications/index.ts`
- `src/emails/GuardianCourseCompletionEmail.tsx`
- `src/app/api/guardian/unsubscribe/route.ts`

**Files to modify:** None (triggers are in the migration; no app-layer changes needed).

**Work:**

1. **Migration** `20260620100000_guardian_notification_queue.sql`:
   - CREATE TABLE `guardian_notification_queue` with columns: `id UUID PK`, `student_uid UUID FK→profiles(uid) CASCADE`, `event_type TEXT CHECK IN ('course_completed','badge_awarded')`, `payload JSONB NOT NULL DEFAULT '{}'`, `debounce_until TIMESTAMPTZ DEFAULT NOW()+30min`, `sent_at TIMESTAMPTZ NULL`, `created_at TIMESTAMPTZ DEFAULT NOW()`.
   - ENABLE ROW LEVEL SECURITY. Policy: `FOR ALL TO authenticated USING (false) WITH CHECK (false)`.
   - CREATE INDEX on `(debounce_until, sent_at) WHERE sent_at IS NULL`.
   - Verify `profile_badges` column name for user reference (check migration `20260618200600`) — use the correct column name in the trigger.
   - CREATE FUNCTION `queue_guardian_notification_on_completion()` SECURITY DEFINER: `IF NEW.status = 'completed' AND OLD.status IS DISTINCT FROM 'completed' THEN INSERT WHERE EXISTS guardian_links`.
   - CREATE TRIGGER `trg_guardian_notification_course_completion` AFTER UPDATE OF status ON `course_enrollments`.
   - CREATE FUNCTION `queue_guardian_notification_on_badge()` SECURITY DEFINER: INSERT on badge INSERT where EXISTS guardian_links.
   - CREATE TRIGGER `trg_guardian_notification_badge` AFTER INSERT ON `profile_badges`.

2. **Edge Function** `supabase/functions/send-guardian-notifications/index.ts`:
   - Serve via Deno. Accept POST (for manual invocation) and GET (for CRON).
   - Verify caller: if request has `Authorization`, validate it is the service role key. If called by Supabase CRON, no auth header needed.
   - SELECT from `guardian_notification_queue WHERE debounce_until < NOW() AND sent_at IS NULL LIMIT 50`.
   - For each row: JOIN guardian_links, JOIN profiles for guardian email/settings, JOIN profiles for student display_name, JOIN courses for course title (if course_completed).
   - Skip guardian if `settings?.notifications?.guardian_emails === false`.
   - For `course_completed`: call Resend with `GuardianCourseCompletionEmail`. For `badge_awarded`: call Resend with `BadgeAwardEmail` addressed to guardian.
   - Generate unsubscribe JWT: HMAC-SHA256 signed with `Deno.env.get('SUPABASE_JWT_SECRET')`, payload `{ sub: 'guardian-unsub', guardian_uid, exp: now+30days }`.
   - After send: UPDATE queue `SET sent_at = NOW()`.
   - On Resend error: log but continue (do not throw — other guardians must still receive).

3. **Email template** `src/emails/GuardianCourseCompletionEmail.tsx`:
   - Props: `guardianName: string`, `studentName: string`, `courseTitle: string`, `portalUrl: string`, `unsubscribeUrl: string`.
   - Subject line baked into props usage: caller passes `subject="[studentName] completed [courseTitle]"`.
   - Body: celebratory 2-sentence copy. No grades. Link to guardian portal. Small unsubscribe footer link.

4. **Unsubscribe route** `src/app/api/guardian/unsubscribe/route.ts`:
   - GET handler. Read `token` from `req.nextUrl.searchParams`.
   - Verify HMAC-SHA256 signature using `SUPABASE_JWT_SECRET`. Check `sub === 'guardian-unsub'`. Check `exp > Date.now()/1000`.
   - On valid token: service client UPDATE `profiles SET settings = jsonb_set(settings, '{notifications,guardian_emails}', 'false') WHERE uid = guardian_uid`.
   - Return plain HTML `200` response: `"You have been unsubscribed from guardian notifications."` No layout, no React.
   - On invalid/expired token: return `400` plain HTML `"This unsubscribe link has expired."`.

**Security:**
- `guardian_notification_queue` RLS blocks all authenticated access — service role only.
- Unsubscribe token verified via HMAC before any DB write.
- No email addresses or names stored in queue payload (UUIDs only).
- Edge Function uses `SUPABASE_SERVICE_ROLE_KEY` from Deno env — never exposed to client.

**Verification:**
- `npm run typecheck`
- `npm run lint`
- `supabase db push` — migration applies cleanly
- E2E: update a `course_enrollments.status` to `completed` for a student with a guardian link, confirm queue row appears
- E2E: invoke Edge Function manually with service key, confirm `sent_at` populated and Resend shows delivery
- Unit: unsubscribe route — valid token → 200 → preference set; invalid token → 400
