# COUNCIL-2026-003: Org Self-Serve Billing Page

**Date:** 2026-06-21
**Status:** Ratified — Unanimous (7/7)
**Tags:** billing, stripe, org-admin, customer-portal, subscription

---

## Question

Org admins currently have no way to view their subscription plan, see invoice history, or
manage their billing without contacting the platform admin. The Stripe infrastructure is
already fully built (webhook, checkout, plans, idempotency). What is the minimal correct
page at `/admin/billing/` that gives org admins self-serve visibility and control?

---

## Wildcard Pre-Audit

**What already exists — confirmed by reading the code:**

- `src/lib/stripe.ts` — Stripe client with `apiVersion: '2026-05-27.dahlia'`. ✓
- `src/app/api/stripe/webhook/route.ts` — fully built. Handles `checkout.session.completed`,
  `invoice.payment_failed`, `customer.subscription.deleted`, `customer.subscription.updated`.
  Idempotency via `platform_audit_log`. Signature verification via `constructEvent`. ✓
- `src/app/api/stripe/create-checkout/route.ts` — creates Stripe Checkout session, restricted
  to org admins and platform admins. ✓
- `src/lib/stripe-plans.ts` — `PRICE_TO_PLAN` map and `PLAN_FEATURES`. ✓
- `src/app/platform/tenants/[id]/billing/page.tsx` — platform-admin billing view with
  `BillingActions.tsx`. Exists for platform admins. ✓
- `organizations.plan` and `organizations.status` — source of truth for plan and lifecycle. ✓
- `organizations.stripe_customer_id` — check migration to confirm this column exists (likely
  in `20260618200100_tenant_lifecycle.sql`). If it exists, Customer Portal sessions can be
  created immediately. If not, it's the first thing the webhook sets.

**What does NOT exist:**
- `/admin/billing/` page for org admins.
- `/api/stripe/portal` route to create a Stripe Customer Portal session.
- No Stripe Customer Portal product is configured in the Stripe dashboard (separate step —
  must be done in Stripe dashboard → Billing → Customer Portal settings before the route works).

**Key constraint:** Stripe Customer Portal requires that the org's `stripe_customer_id` be set.
This is set by the webhook when `checkout.session.completed` fires. Orgs on the free plan
with no Stripe interaction have no `stripe_customer_id` — the portal link cannot be generated
for them. The billing page must handle this gracefully (show "No active subscription" state).

## Positions

### 🏛️ The Architect

**Two components: a read page + a portal API route.**

The billing page at `/admin/billing/` reads the org's current plan from `organizations`
(already in RLS scope for org admins). It shows plan name, status, and feature flags.
If `stripe_customer_id` is set, it renders a "Manage Billing" button that POSTs to
`/api/stripe/portal` to get a Customer Portal session URL, then redirects.

`/api/stripe/portal` (new route):
1. Auth: org admin only (via `profiles.role`).
2. Get `stripe_customer_id` from `organizations WHERE id = callerOrgId`.
3. If null: return 400 `{ error: 'No active subscription' }`.
4. Call `stripe.billingPortal.sessions.create({ customer: stripe_customer_id, return_url })`.
5. Return `{ url }`. Client redirects to it.

**Why Customer Portal and not custom UI?**
- Stripe Customer Portal handles: plan upgrades, downgrades, cancellation, invoice download,
  payment method update, tax ID. Building this ourselves takes weeks.
- The portal is Stripe-hosted and PCI-compliant. We never touch raw card data.
- Cons: less brand control. Acceptable for a church LMS where billing is infrequent.

**Can org admins cancel?** Yes — through the Customer Portal. Cancellation triggers
`customer.subscription.deleted` → webhook sets `organizations.status = 'suspended'`.
This is already handled. No new code path needed.

### 🧭 Product Manager

**Page layout — three sections:**

1. **Current plan card**: Plan name badge (Free / Starter / Growth / Enterprise), status
   badge (Active / Suspended / Trial), renewal date (from Stripe via portal — we don't store
   it locally). Features list from `PLAN_FEATURES[org.plan]`.

2. **"Manage Billing" CTA**: Only visible when `stripe_customer_id` is set.
   - Label: "Manage Subscription & Invoices"
   - Opens Stripe Customer Portal in same tab (redirect, not new tab — portal has a "return"
     button that brings them back to `/admin/billing/`).
   - If no Stripe customer: show "Contact support to upgrade your plan." with a mailto link.

3. **Plan comparison table** (lightweight): Show what's included in each plan tier so the
   admin understands what upgrading would give them. Read from `PLAN_FEATURES` in
   `stripe-plans.ts` — no additional data fetching.

**Free plan UX:** Most orgs start on free. Show their current features clearly. Show what
they'd get on Starter/Growth. Add "Upgrade" button that triggers the existing checkout flow
(`/api/stripe/create-checkout`). Pass `priceId` for the Starter plan.

**Cancellation warning:** When the admin lands on the portal and cancels, Stripe sends a
`customer.subscription.deleted` webhook → org gets suspended. The admin returns to
`/admin/billing/` via the portal's return URL. Show a banner if `org.status === 'suspended'`:
"Your subscription has been cancelled. Members cannot access the platform. Contact support
to reactivate."

### ⚙️ The Engineer

**New route: `src/app/api/stripe/portal/route.ts`**

```typescript
export const runtime = 'nodejs'

export async function POST(req: NextRequest) {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })

  const { data: profile } = await supabase
    .from('profiles')
    .select('role, org_id')
    .eq('auth_id', user.id)
    .single()

  if (profile?.role !== 'admin') {
    return NextResponse.json({ error: 'Admin access required' }, { status: 403 })
  }

  const svc = createServiceClient()
  const { data: org } = await svc
    .from('organizations')
    .select('stripe_customer_id, status')
    .eq('id', profile.org_id)
    .single()

  if (!org?.stripe_customer_id) {
    return NextResponse.json({ error: 'No active subscription' }, { status: 400 })
  }

  const session = await stripe.billingPortal.sessions.create({
    customer:   org.stripe_customer_id,
    return_url: `${process.env.NEXT_PUBLIC_APP_URL}/admin/billing`,
  })

  return NextResponse.json({ url: session.url })
}
```

- Fetch `organizations.plan, status, stripe_customer_id, settings` via user's Supabase client.

- Show plan features from `PLAN_FEATURES[org.plan]` (import from `stripe-plans.ts`).

**Navbar link:** Add "Billing" to the admin sidebar in `SidebarClient.tsx` and
`MobileAdminDrawer.tsx` (the two nav systems for admin users).

### 🛡️ Security Officer

**`stripe_customer_id` must be fetched via service client**, not the user's Supabase client.
The `stripe_customer_id` is not a secret (it's a Stripe ID like `cus_xxx`), but it must
not be readable by RLS-restricted users who might query other orgs. The service client
bypasses RLS — only use it server-side in the route handler.

**Portal URL is single-use and short-lived.** Stripe Customer Portal session URLs expire
quickly. Do not cache them. Generate fresh on each button click.

**Cancellation → suspension is automatic** via the existing webhook. No special handling
needed in the billing page. The page reflects the current `status` from `organizations`.

**Org admin cannot access platform-admin billing routes.** The existing `/platform/tenants/[id]/billing/`
page already checks `is_platform_admin()`. The new `/admin/billing/` page checks
`profile.role === 'admin'` — these are separate access gates.

**No card data ever touches our server.** The Customer Portal is Stripe-hosted. The only
thing we exchange is a session URL.

### 🚀 DevOps Officer

**Stripe Customer Portal must be configured in Stripe dashboard before this feature works.**
Go to: Stripe Dashboard → Billing → Customer Portal → Settings. Enable:
- Invoice history (required for admins to view past invoices)
- Payment method update
- Plan change (if you want self-serve upgrades through the portal)
- Cancellation (choose "immediately" or "at end of period")

This is a one-time dashboard setup, not a code change.

**New env var:** None. `STRIPE_SECRET_KEY` is already required by the existing checkout route.

## Council Verdict (7/7 Unanimous)

Build the billing page and portal route. No new DB infrastructure. Stripe Customer Portal
setup in dashboard is a prerequisite (one-time manual step).

| Decision | Verdict |
|---|---|
| UI approach | Stripe Customer Portal (not custom UI) |
| Can admins cancel | Yes — through portal, webhook handles suspension |
| Free plan | Show features + "Upgrade" CTA to existing checkout |
| Portal API | New `/api/stripe/portal` POST route |
| New DB columns | None (confirm `stripe_customer_id` exists first) |
| Sidebar link | Add "Billing" to admin nav |
