# COUNCIL-2026-002: Bulk User CSV Import

**Date:** 2026-06-20
**Status:** Ratified — Unanimous (7/7)
**Tags:** admin, users, csv, bulk-import, invite, validation

---

## Question

The current user onboarding path for org admins is: navigate to `/admin/users`, fill in the
invite form one user at a time, wait for the invite email to send. A church importing a 200-person
roster does this 200 times.

What is the minimal, correct architecture for a bulk CSV import that:
1. Allows admins to upload a CSV and invite multiple users in one operation
2. Reports per-row success and failure clearly
3. Does not silently skip rows
4. Respects tenant isolation (admin can only import into their own org)
5. Does not create users in auth.users without an invite (no password-less orphan accounts)

---

## Wildcard Pre-Audit (fires before deliberation)

**What already exists:**

- `src/app/admin/users/InviteUserForm.tsx` — single-user invite form.
- `src/app/actions/admin.ts` — `inviteUser` server action calls
  `supabase.auth.admin.inviteUserByEmail()` using the service client.
  This sends a Supabase-generated invite email with a one-time link.
  The invited user must click the link to set their password — no password is
  ever set in the code path.
- `profiles` table has: `uid`, `auth_id`, `display_name`, `email`, `role`, `status`, `org_id`.
  A `handle_new_user` trigger creates the profile row when GoTrue creates the auth user.
  The trigger reads `raw_user_meta_data` to set `display_name` and `org_id`.
- `user_role` ENUM values: `admin`, `manager`, `teacher`, `student`, `guardian`.
- The `inviteUserByEmail` API accepts `data: { org_id, role, display_name }` in
  `options.data` (raw_user_meta_data). This is already used in the existing server action.

**What does NOT exist:**
- No CSV upload route.
- No server-side CSV parsing utility.
- No batch invite server action.
- No `/admin/users/import/` page.
- No `papaparse` or CSV parsing library in `package.json`.

**Key constraint discovered:**
`inviteUserByEmail` is rate-limited by Supabase to 3 invites per hour on the hosted free plan.
On Pro plan the limit is higher but still exists. A 200-row CSV cannot be processed synchronously
in one request — this will hit the rate limit and/or Vercel's 60-second serverless function timeout.

This means the council must address whether to:
(A) Use the GoTrue invite flow (rate-limited, sends invite emails) — correct flow but slow for bulk
(B) Use `createUser` with a temporary password + `email_confirm: true` — bypasses invite emails,
    users receive no welcome/set-password prompt until the admin sends them their credentials
(C) Queue invites and process them in batches with rate-limit spacing via a background job

---

## Positions

### 🏛️ The Architect

**Use option (A) — GoTrue invite flow — with a queue-and-drain architecture.**

The invite email is the product's mechanism for user onboarding. Bypassing it (option B) means
the admin must separately communicate passwords to 200 people. That is worse UX than waiting
for the invite batch to process.

Architecture:
1. Admin uploads CSV. Server action validates all rows synchronously and returns a preview
   (number of valid rows, list of errors). No users are created at this step.
2. Admin confirms. A second server action enqueues all validated rows into a new
   `import_jobs` table with status `pending`.
3. A Supabase Edge Function `process-import-job` is called via `pg_net` (or via a webhook
   from the platform) to drain the queue. It calls `inviteUserByEmail` for each row, spacing
   calls with a 1.2-second delay to stay under the rate limit.
4. The admin sees a live status page polling `/api/admin/import-jobs/[id]` for progress.

**Simpler alternative for MVP (recommended):** Cap import at 50 rows. Process synchronously
on the server (50 × 1.2s = 60 seconds, right at the Vercel timeout limit). Return a result
table. No queue table, no Edge Function, no polling. Suitable for church ministries (typical
roster is 20–80 active learners). If the org needs > 50 at once, they split the CSV.

**Council recommendation: 50-row synchronous cap for Sprint 2. Queue architecture in Sprint 3.**

The 50-row cap is explicit in the UI: the upload form rejects CSVs over 50 rows with a clear
error. This avoids the queue infrastructure while shipping the feature today.

### 🧭 Product Manager

**CSV column schema (order must not matter — headers-based parsing):**

| Header (case-insensitive) | Required | Notes |
|---|---|---|
| `email` | Yes | Must be valid email format |
| `display_name` | Yes | Non-empty string |
| `role` | Yes | One of: `admin`, `manager`, `teacher`, `student`, `guardian` |

No password column. No org_id column (set server-side from the calling admin's org).

**Validation rules (row level):**
- Email is invalid format → row error: "Invalid email"
- Email already exists in `profiles` for this org → row error: "Already a member"
- Email already exists in `profiles` for a DIFFERENT org → row error: "Email registered to another organization" (do not reveal which org — privacy)
- Role is not one of the valid enum values → row error: "Invalid role. Must be one of: admin, manager, teacher, student, guardian"
- `display_name` is empty → row error: "Display name is required"

**Preview step before inviting (gate):** After upload, show a table:
```
Row | Email                     | Name           | Role    | Status
1   | alice@church.org          | Alice Smith    | teacher | ✓ Ready
2   | bob@church.org            | Bob Jones      | student | ✓ Ready
3   | badformat                 | —              | student | ✗ Invalid email
4   | existing@church.org       | —              | teacher | ✗ Already a member
```
Admin can download a corrected-errors CSV or proceed with valid rows only.

**Success reporting:** After invites are sent, show the same table with actual send status:
`Invite sent`, `Already a member (skipped)`, `Send failed (try again)`.

**File size cap:** 50 rows max. Reject files > 50 data rows before parsing. Show error:
"CSV exceeds 50 rows. Split your file and import in batches."

### ⚙️ The Engineer

**No new npm packages needed.** CSV parsing for a 50-row file with known headers does not
require `papaparse`. Use the browser's native `File.text()` + manual split on newlines + split
on commas. Edge case: quoted fields with commas (e.g., `"Smith, Jr."`). For a simple 3-column
CSV with known headers, a minimal parser is safe. If `display_name` may contain commas,
either document that display names must not contain commas OR add a micro-parser for quoted fields.

**Recommendation: document the constraint.** Display names with commas are rare. The UI should
say: "Display names must not contain commas." This avoids a parsing dependency.

**File: `src/app/admin/users/import/page.tsx`** — Server Component shell with:
- File input (`<input type="file" accept=".csv">`) — not a drag-drop zone (overkill for v1)
- Client component `ImportForm` handles file reading, preview table, and submit
- No multi-step wizard — one page, two UI states (upload view → preview/results view)

**File: `src/app/admin/users/import/ImportForm.tsx`** — `'use client'`
- `useState` for: `file`, `rows: ParsedRow[]`, `results: ImportResult[]`, `status: 'idle'|'preview'|'importing'|'done'`
- `handleFileChange`: reads file → parses → validates client-side → shows preview table
- `handleImport`: calls server action with validated rows → shows results

**File: `src/app/actions/admin.ts`** (add to existing file):
```typescript
export async function bulkInviteUsers(
  rows: { email: string; display_name: string; role: string }[]
): Promise<{ email: string; status: 'sent' | 'skipped' | 'failed'; reason?: string }[]>
```
- Auth check: calling user must be `admin`.
- Org check: get caller's `org_id` from profile.
- Row-level validation: re-validate server-side (client can be tampered).
- Check `profiles WHERE email = row.email AND org_id = callerOrgId` → skip if exists.
- Check `profiles WHERE email = row.email AND org_id != callerOrgId` → error (no org name disclosed).
- Call service client `auth.admin.inviteUserByEmail(email, { data: { org_id, role, display_name } })`.
- Collect per-row result. Return array (never throw — caller handles individual failures).
- Rate limit: 50-row cap enforced server-side (`if (rows.length > 50) throw new Error`).

**`parse-csv.ts` (new file `src/lib/parse-csv.ts`):**
```typescript
export interface CsvRow { email: string; display_name: string; role: string; rowNum: number }
export interface ParseResult { rows: CsvRow[]; errors: { rowNum: number; message: string }[] }
export function parseCsv(text: string): ParseResult
```
Pure function, no imports. Headers case-insensitive. Returns both valid rows and errors.
Called client-side in `ImportForm.tsx` for preview. Exact same function re-validated server-side.

**Loading state:** During import, button is disabled with a spinner. Each row shows a
pending indicator that resolves to ✓/✗ as results come back. Since the server action returns
all results at once (not streamed), the table updates atomically after the action resolves.

### 🛡️ Security Officer

**Tenant isolation is the primary risk.** The server action must get `org_id` from the
caller's session (via `supabase.auth.getUser()` + `profiles` lookup), NOT from the request body.
If org_id came from the client, an admin from Org A could import users into Org B.

**Role escalation.** The valid role enum must be validated server-side against the TypeScript
enum. A caller should not be able to import a user with role `platform_admin` or any value
not in the allow-list. The check: `if (!['admin','manager','teacher','student','guardian'].includes(row.role)) skip/error`.

**Admin-only.** The server action must verify `me.role === 'admin'`. Managers and teachers
must not be able to invite users.

**Email disclosure.** If an email belongs to another org, do NOT disclose the org name or
any information about it. Return: "Email is already registered in another organization." Full stop.

**No PII in audit log payload.** Log the batch import to `admin_audit_log` with:
`{ action: 'bulk_invite', count_attempted: N, count_sent: N, count_skipped: N, count_failed: N }`.
Do NOT log individual email addresses in the audit record.

**CSV file is never stored.** The file is read client-side via `File.text()`, the plain-text
rows are sent as JSON to the server action (not the file itself). No multipart upload, no
Supabase Storage write, no temp file on disk.

**Service role key usage:** `inviteUserByEmail` requires the service client (service role key).
This is already the pattern in the existing `inviteUser` server action. Never expose the service
role key to client components.

### 🔬 The Tester

1. Valid 3-column CSV → correct rows returned
2. Headers in different order → same result (order-insensitive)
3. Headers case-insensitive (`EMAIL`, `Email`, `email` all work)
4. Empty display_name → error on that row
5. Invalid email format → error on that row
6. Invalid role → error on that row
7. Row 51 in a 51-row CSV → error reported (cap enforcement in parser)
8. CSV with trailing newline → does not produce a phantom empty row

**Unit tests for `bulkInviteUsers` server action (mock service client):**
1. Non-admin caller → throws 'Unauthorized'
2. > 50 rows → throws 'Too many rows'
3. Email already in same org → returns `{ status: 'skipped' }` without calling invite API
4. Email in different org → returns `{ status: 'failed', reason: 'Email registered to another organization' }`
5. Invite API call fails → returns `{ status: 'failed' }` without crashing others
6. Valid row → calls `inviteUserByEmail` with correct `{ data: { org_id, role, display_name } }`

**E2E test:**

- Preview shows 2 ✓ and 1 ✗

### 🚀 DevOps Officer

**No new infrastructure.** No queue table, no Edge Function, no CRON. 50-row synchronous
import fits within Vercel's serverless timeout. Zero new env vars required.

**Vercel function timeout:** Default is 10 seconds on Hobby, 60 seconds on Pro. At 1.2s/invite
× 50 invites = 60 seconds exactly. Set `export const maxDuration = 60` in the route/action
to opt into the longer timeout. On Hobby plan, cap at 8 rows (8 × 1.2s = ~10s). Add a
plan-based cap check if needed in a future sprint.

**Staging test:** Run a 5-row import against the staging Supabase project. Verify Resend
delivers invite emails. Verify GoTrue creates `auth.users` rows. Verify `handle_new_user`
trigger sets `org_id` correctly from `raw_user_meta_data`.

### 🃏 The Wildcard

**What the council missed or underweighted:**

1. **The `handle_new_user` trigger must read `raw_user_meta_data.role` and set `profiles.role`.**
   If the trigger does not set `role` (only `display_name` and `org_id`), all imported users
   will have a null or default role and will be locked out by RLS until an admin manually sets
   it. Audit the trigger before shipping. If it does not set `role`, add it as part of this prompt.

2. **Resend delivers invite emails — but the `inviteUserByEmail` email is Supabase's auth email,
   not Resend.** The SMTP-via-Resend config from COUNCIL-2025-017 means Supabase Auth invites
   are now delivered by Resend. This is good — but the "From" address must be configured in
   Supabase Auth → SMTP settings. If not configured, invites go from Supabase's default sender
   which looks untrustworthy to end users. Remind the admin setup docs.

3. **Rate limit of 3/hour on free plan.** A 50-row import = 50 invite API calls.
   Even at 1.2s spacing, this will hit the GoTrue rate limit after 3 calls on the free plan.
   Solutions:
   - Upgrade Supabase to Pro plan (removes the 3/hour cap — recommended for any live tenant)
   - OR use `createUser` with `email_confirm: true` + a custom welcome email via Resend (bypasses GoTrue invite rate limit)
   The council has chosen the GoTrue invite path, so the implementation prompt must warn:
   "This feature requires Supabase Pro plan or the rate limit will block imports > 3 users."
   Add this warning to the UI.

4. **Duplicate detection must handle case-insensitive email.** `alice@church.org` and
   `Alice@church.org` are the same address. The duplicate check query must use
   `ilike` or `lower(email)` comparison, not exact match.

5. **CSV download of errors.** After preview shows validation errors, the admin needs to fix
   the source data. Provide a "Download errors as CSV" button that generates a CSV of
   the error rows with an added `error` column. This is pure client-side (no server call)
   and takes ~20 lines of code. High value, low effort.

---

## Council Verdict (7/7 Unanimous)

50-row synchronous import, GoTrue invite flow, no new infrastructure. Cap enforced server-side.
Rate limit warning in UI for free-plan Supabase projects.

| Decision | Verdict |
|---|---|
| Max rows | 50 (enforced client and server) |
| Auth flow | GoTrue `inviteUserByEmail` (invite email, set-password link) |
| Queue architecture | Deferred to Sprint 3 |
| CSV parsing | Hand-rolled (no papaparse), display names must not contain commas |
| Preview step | Yes — validate before inviting |
| Per-row error reporting | Yes — table with ✓/✗ per row |
| Error CSV download | Yes — client-side only |
| PII in audit log | No — counts only, not emails |
| Rate limit handling | Warning in UI; requires Supabase Pro for > 3 users |

**Work:**

1. **`src/lib/parse-csv.ts`** — pure function, no imports:
   - `parseCsv(text: string): ParseResult`
   - Parse headers from first line (case-insensitive). Expect `email`, `display_name`, `role`.
   - Reject if any required header is missing.
   - For rows 1–50: split on `,`, trim whitespace, validate email format (simple regex), validate role in enum, validate display_name non-empty.
   - Reject row 51+ with error: "File exceeds 50-row limit."
   - Return `{ rows: CsvRow[], errors: ParseError[] }`.

2. **`src/app/admin/users/import/ImportForm.tsx`** — `'use client'`:
   - State: `step: 'upload' | 'preview' | 'done'`, `rows`, `errors`, `results`.
   - `handleFile`: read `file.text()`, call `parseCsv()`, set step to 'preview'.
   - Preview table: columns Email, Name, Role, Status (✓/✗ with error message).
   - "Download error rows as CSV" button (generate Blob client-side): appears if `errors.length > 0`.
   - "Send Invites" button: calls `bulkInviteUsers(validRows)`, sets step to 'done', shows results table.
   - Results table: same columns + actual send status from server.
   - Rate limit warning banner: `"Large imports require Supabase Pro plan. Free plan allows 3 invites per hour."` (always visible on this page).

3. **`src/app/admin/users/import/page.tsx`** — Server Component:
   - Auth check: redirect to `/dashboard` if not admin.
   - Render `<ImportForm />`.
   - Breadcrumb: `Users / Import`.

4. **`src/app/actions/admin.ts`** — add `bulkInviteUsers`:
   - Verify caller is `admin` via `supabase.auth.getUser()` + profiles lookup.
   - Verify `rows.length <= 50` (server-side enforcement).
   - Get caller's `org_id` from their profile (never from request).
   - For each row: validate role in allow-list `['admin','manager','teacher','student','guardian']`.
   - Duplicate check: `SELECT 1 FROM profiles WHERE lower(email) = lower(row.email)`. If same org → `skipped`. If different org → `failed, reason: 'Email registered to another organization'`.
   - For valid rows: call service client `auth.admin.inviteUserByEmail(row.email, { data: { org_id: callerOrgId, role: row.role, display_name: row.display_name } })`.
   - Collect result per row. Do NOT throw on individual row failures.
   - After all rows: INSERT into `admin_audit_log`: `{ action: 'bulk_invite', metadata: { count_attempted, count_sent, count_skipped, count_failed } }`.
   - Return array of per-row results.

5. **`src/app/admin/users/page.tsx`** — add Import button:
   - In the page header area, next to "Invite User", add: `<Link href="/admin/users/import" className="btn-secondary text-sm">Import CSV</Link>`.

6. **Audit `handle_new_user` trigger**: read the trigger function in migrations. If it does not set `profiles.role` from `raw_user_meta_data`, create migration `20260620100100_handle_new_user_role.sql` that adds `role = (NEW.raw_user_meta_data->>'role')::user_role` to the INSERT.

- Role allow-list validated server-side before any API call.
- No email addresses in audit log.
- Service role key never leaves the server action.
- Duplicate check uses `lower(email)` for case-insensitive matching.
