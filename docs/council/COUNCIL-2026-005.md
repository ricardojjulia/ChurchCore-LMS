# COUNCIL-2026-005: Graded Discussion Block

**Date:** 2026-06-21
**Status:** Ratified тАФ Unanimous (7/7)
**Tags:** discussion, grading, block-submissions, teacher-ui, learning-engine

---

## Question

`DiscussionPlayer.tsx` is fully built тАФ students post replies that save to `block_submissions`.
The `block_submissions` table already has `score`, `max_score`, `graded_by`, `graded_at`,
and `status` (draft/submitted/graded/returned/excused) columns. What needs to be built is the
teacher grading UI and the grade display for students. What is the grading schema (per-post
vs. participation score) and what is the minimal correct implementation?

---

## Wildcard Pre-Audit

**What exists тАФ confirmed by reading the code:**

`src/components/learning/DiscussionPlayer.tsx`:
- Students post via `block_submissions.insert({ block_id, status: 'submitted', content: { text, type: 'discussion' }, submitted_at })`.
- Replies are fetched via `get_block_discussion_replies(p_block_id)` RPC.
- `Reply` interface has `{ submission_id, user_id, display_name, content, submitted_at, is_own }`.
- Students can edit their own post (UPDATE content). Cannot delete.

`BlockPlayer.tsx` at line 169:

- Comment: "This content type is not yet interactive" тАФ this is WRONG (it IS interactive),
  suggesting the comment was written before `DiscussionPlayer` was added. Remove this comment

`block_submissions` schema (from migration `20240601000002_canvas_block_model.sql`):
- `score numeric` тАФ grade received.
- `max_score numeric` тАФ maximum possible grade.
- `graded_by uuid REFERENCES auth.users(id)` тАФ who graded.
- `graded_at timestamptz` тАФ when graded.
- `status` CHECK IN ('draft','submitted','graded','returned','excused').
- RLS: students see own submissions; teachers/admins see all in their org.

`get_block_discussion_replies` RPC:
- Returns all replies for a block. Need to confirm if it returns `submission_id` per reply тАФ
  it does (visible in `Reply` interface). This is the key for grading: teacher grades by
  `submission_id`.

**What does NOT exist:**
- No teacher view of all student posts for a discussion block.
- No grade input UI for teachers.
- No grade display for students in `DiscussionPlayer`.
- No server action for grading a submission.

**Grading schema decision:**
Discussion grading is **participation-based**: one score per student per discussion block,
reflecting the quality of their overall contribution. Each student has one `block_submissions`
row per block (their post). The teacher sets `score` and `max_score` on that row.

This maps cleanly onto the existing schema тАФ no new columns needed.

## Positions

### ЁЯПЫя╕П The Architect

**Grading schema: participation score (one grade per student per block submission).**

Each student has at most one `block_submissions` row per `block_id` (enforced by the current
insert pattern). The teacher assigns a single score to that submission, representing their
participation quality (engagement, thoughtfulness of post, etc.).

`max_score` is set when the discussion block is created in the course builder (via `DiscussionForm.tsx`).
Default: `max_score = 10`. The teacher can override `max_score` when grading.

**Two new UI surfaces:**

Surface 1 тАФ **Teacher grading panel in `DiscussionPlayer`**:
When the viewing user is a teacher, admin, or manager (check via a `role` prop or context),
each reply shows a "Grade" button. Clicking opens an inline grade form:
- Score input (numeric, 0 to max_score)
- Max score input (editable тАФ teacher may adjust)
- Submit button тЖТ calls `gradeDiscussionSubmission` server action
- Cancel button

After grading: the reply row updates to show `Graded: 7/10`.

Surface 2 тАФ **Student grade display in their own reply**:
When `is_own = true` and the submission has been graded (`status = 'graded'`), show the
grade below their post: `"Grade: 7 / 10"` in a small badge.

**No separate grading page.** Inline grading in the discussion thread is faster and gives
context (teacher sees the post while grading it). A dedicated grading page is appropriate
for assignments; discussions are graded in-context.

### ЁЯзн Product Manager

**Teacher workflow:**
1. Teacher opens the course page containing a discussion block.
2. `DiscussionPlayer` shows all student posts (already the case тАФ `get_block_discussion_replies`
   returns all posts).
3. Each post has a "Grade" button (visible only to teachers/admins).
4. Teacher clicks "Grade" тЖТ inline score/max_score form appears below the post.
5. Teacher enters score тЖТ clicks "Save Grade" тЖТ submission updated to `status: 'graded'`.
6. Post now shows a green "Graded" badge + score.

**Student workflow:**
1. Student opens the course page.
2. `DiscussionPlayer` shows their own post with a `"Grade: 7 / 10"` badge below it (if graded).
3. Other students' posts are visible but without grade info (privacy).

**Edge cases:**
- Ungraded student: no grade badge (just their post).
- No discussion yet: student hasn't posted, teacher cannot grade (no submission row exists).
- Teacher edits a grade: clicking "Grade" on an already-graded submission shows the existing
  score pre-filled. Save overwrites.
- `max_score = 0` in block config: default to 10 (guard in the server action).

**Grade visibility:** A student sees only their OWN grade. Other students' grades are private.
The existing `is_own` flag in `Reply` handles this тАФ render grade display only when `is_own`.

### тЪЩя╕П The Engineer

**New server action `gradeDiscussionSubmission` in `src/app/actions/learning.ts`**
(or `src/app/actions/admin.ts` тАФ check which file handles submission-related actions):

```typescript
export async function gradeDiscussionSubmission({
  submissionId,
  score,
  maxScore,
}: {
  submissionId: string
  score: number
  maxScore: number
}): Promise<{ error?: string }>
```

**Implementation:**
1. Auth: `createClient()` + `getUser()`.
2. Profile: get `role` and `org_id`.
3. Role check: `['admin', 'manager', 'teacher'].includes(role)` тАФ only these can grade.
4. Validate: `score >= 0`, `maxScore > 0`, `score <= maxScore`.
5. Verify the submission belongs to the caller's org (prevent cross-tenant grading):
   `block_submissions JOIN course_blocks JOIN courses WHERE courses.org_id = callerOrgId`.
   Use service client for this JOIN (RLS on block_submissions is `FOR SELECT` to own
   submissions for students; teachers see all in their org тАФ confirm with RLS policy).
6. Service client UPDATE:
   ```sql
   UPDATE block_submissions
   SET score = $score, max_score = $maxScore,
       graded_by = auth.uid(), graded_at = NOW(), status = 'graded'
   WHERE id = $submissionId
   ```
7. Return `{}` on success, `{ error: 'message' }` on failure.

**`DiscussionPlayer.tsx` changes:**

Add `role` prop (teacher/admin/manager shows grade controls; student shows own grade):
```typescript
interface Props {
  blockId:       string
  prompt?:       string
  ownReplyText?: string | null
  viewerRole?:   string   // passed from BlockPlayer
  maxScore?:     number   // default 10
}
```

In `BlockPlayer.tsx` at the `discussion` case: pass `viewerRole` and `maxScore` from the
block's content config.

`Reply` interface: add optional `score?: number | null` and `graded_at?: string | null`
fields. The `get_block_discussion_replies` RPC must return these for the teacher view.
Check the RPC definition in migrations тАФ if it does not return `score`/`graded_at`, update it
OR use a second query for grade data.

**Simplest approach if RPC doesn't return grade data:** After loading replies, fetch
`block_submissions WHERE block_id = blockId SELECT id, score, max_score, status, graded_at`
using the Supabase client (teachers see all due to RLS; students see own). Merge into replies
by `submission_id`.

**Inline grade form in `DiscussionPlayer`:**
```tsx
// Inside the reply render, after the post content:

  gradingId === reply.submission_id

```

Where `canGrade = ['admin', 'manager', 'teacher'].includes(viewerRole ?? '')`.
Where `gradingId` is a `useState<string | null>(null)`.

**`GradeForm` subcomponent** (inline in `DiscussionPlayer.tsx` or extracted):
- Score input: `<input type="number" min="0" max={maxScore} step="0.5" />` (allow half-points)
- Max score input: `<input type="number" min="1" step="1" />`
- "Save Grade" button тЖТ calls `gradeDiscussionSubmission` тЖТ on success, `onSave(score, maxScore)`
- Loading state during save

### ЁЯЫбя╕П Security Officer

**Cross-tenant grading prevention:** The server action must verify the submission belongs to
the caller's org before updating. A teacher from Org A must not be able to grade a submission
from Org B by guessing a `submission_id` UUID.

Verification query: `SELECT bs.id FROM block_submissions bs JOIN course_blocks cb ON cb.id = bs.block_id JOIN courses c ON c.id = cb.course_id WHERE bs.id = $submissionId AND c.org_id = $callerOrgId`

If no row found: return `{ error: 'Not found' }` (do not reveal whether the submission exists
in another org тАФ same response for "not found" and "wrong org").

**Role check is mandatory.** Students must not be able to call `gradeDiscussionSubmission`.
The check `['admin', 'manager', 'teacher'].includes(profile.role)` runs server-side.

**Students only see their own grade** (via `is_own` flag in the frontend). The RLS policy
on `block_submissions` already enforces that students can only SELECT their own rows тАФ
the server-side fetch of grade data respects this automatically.

**`graded_by` is set from the server** using the authenticated user's ID тАФ never from
client-supplied input.

4. **The comment "This content type is not yet interactive" in `BlockPlayer.tsx` at line 183
   is wrong** тАФ `DiscussionPlayer` is interactive. Remove this comment in the implementation

5. **Half-point grades (`step="0.5"`) may not match the `score numeric` column type.** Postgres

## Council Verdict (7/7 Unanimous)

Grading schema: participation score тАФ one `score`/`max_score` per student per block submission.
Grading UI: inline in `DiscussionPlayer`, visible to teachers/admins. Student sees own grade.
No new DB columns or migrations. Pure UI + one server action.

| Decision | Verdict |
|---|---|
| Grading schema | Participation score (one grade per submission row) |
| Grade UI location | Inline in `DiscussionPlayer` |
| Default max_score | 10 (if not in block JSONB) |
| Half-points | Yes (`step="0.5"`) |
| Student grade visibility | Own grade only (`is_own` guard) |
| Cross-tenant protection | Server-side org_id JOIN before UPDATE |
| RPC modification | No тАФ use second query if needed |
| New migration | None |
