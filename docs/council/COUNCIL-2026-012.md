# COUNCIL-2026-012: Badge Auto-Triggers

**Status:** RATIFIED — Pending Implementation
**Date:** 2026-06-21
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)
**Related:** COUNCIL-2026-006 (Engagement Tracker)
**Tags:** badges, gamification, automation, triggers, rewards

---

## User Story

**As a learner**, I want to automatically receive badges when I hit milestones (complete my first
course, earn 100 XP, maintain a 7-day streak) so that I feel recognized for my progress without
a teacher having to award badges manually.

**As an admin**, I want to define badge trigger conditions so that the system awards badges
automatically based on engagement and completion data.

**Acceptance Criteria:**
1. `badges` table gains a `trigger_condition` JSONB column defining what auto-awards the badge
2. Built-in trigger types: `course_completion` (complete N courses), `xp_threshold` (earn X XP total), `streak` (maintain N-day streak), `block_count` (complete N blocks)
3. When a learner hits a trigger condition, `profile_badges` is automatically updated
4. Learner sees a notification when a badge is awarded
5. Existing manual badge award flow is unchanged

**Edge Cases:**
- Badge already awarded → trigger check is idempotent (ON CONFLICT DO NOTHING)
- Multiple triggers could fire simultaneously (completing a course also crosses XP threshold) → both badges awarded
- Admin sets trigger to "5 courses completed" — changing to "3" after some users already earned it does NOT retroactively award those users (trigger fires on new events only)

**Out of Scope:**
- Custom trigger logic (scripted conditions)
- Badge expiry / revocation
- Learner notifications via email (in-app notification only, using existing notifications table)

---

## Gap Being Addressed

`badges` and `profile_badges` tables exist. Manual badge award works (admin INSERT to profile_badges).
But there is no automatic awarding. Teachers manually award badges to learners — this doesn't scale
and misses the motivational moment (award should happen immediately when milestone is hit, not days later).

---

## Wildcard Pre-Audit

**Trigger conditions belong in Postgres, not in a cron job.** The most reliable way to award
badges automatically is a Postgres trigger on `engagement_events` (from COUNCIL-2026-006) or
on `course_certificates`. When a new event is inserted, evaluate all applicable badge conditions
for that user. No cron needed, no polling — fires atomically with the event.

**The badge condition evaluator must be efficient.** When a block completion event is inserted,
you don't want to run `SELECT COUNT(*) FROM course_certificates WHERE user_id = ?` for every
badge in the org. Instead, maintain a `badge_progress` materialized cache (or just use the
`profiles.xp_points` column for XP thresholds). Evaluate only badges whose `trigger_condition`
matches the event type.

**Avoid awarding badges from complex JOINs in a trigger.** If the trigger function is slow, it
blocks the row insert. Keep trigger functions fast: cache needed counts in denormalized columns,
or run badge evaluation asynchronously via `pg_notify` + an Edge Function (v2 consideration).
For v1: evaluate synchronously but with targeted queries (not full-table aggregations).

**`trigger_condition` JSONB schema:**
```json
{ "type": "xp_threshold", "threshold": 100 }
{ "type": "course_completion", "count": 1 }
{ "type": "streak", "days": 7 }
{ "type": "block_count", "count": 10 }
```

## Positions

### 🏛️ The Architect

Migration adds `trigger_condition JSONB` and `is_auto_awarded BOOLEAN DEFAULT FALSE` to `badges`.
Auto-awarded badges have a non-null `trigger_condition`. Manual-only badges have `trigger_condition = NULL`.

A new Postgres function `evaluate_badge_triggers(p_uid UUID, p_org_id UUID, p_event_type TEXT)`
runs after each engagement event. It:
1. Fetches all badges in the org where `trigger_condition` matches the event type
2. For each badge, checks the condition against current user stats
3. If condition met: INSERT INTO profile_badges ON CONFLICT DO NOTHING

This function is called from `record_engagement_event()` (COUNCIL-2026-006) — after the event
is inserted and XP is awarded, call `evaluate_badge_triggers(v_uid, v_org_id, p_event_type)`.

### 🧭 Product Manager

**Admin UI**: In `/admin/badges` (or wherever badges are managed), each badge shows a
"Auto-award trigger" configuration section:
- Trigger type dropdown (Course Completion, XP Threshold, Streak, Block Count, None)
- Threshold/count input

**Learner notification**: When a badge is auto-awarded, insert a row into `notifications`
(if this table exists — check migrations) or equivalent with `type: 'badge_earned'`.

### ⚙️ The Engineer

Add `trigger_condition JSONB` and `is_auto_awarded BOOLEAN DEFAULT FALSE` to `badges` (migration).

`evaluate_badge_triggers` function:
```sql
CREATE OR REPLACE FUNCTION public.evaluate_badge_triggers(
  p_uid UUID, p_org_id UUID, p_event_type TEXT
)
RETURNS VOID LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
DECLARE
  v_badge RECORD;
  v_meets BOOLEAN;
BEGIN
  FOR v_badge IN
    SELECT id, trigger_condition FROM public.badges
    WHERE org_id = p_org_id
      AND is_auto_awarded = TRUE
      AND trigger_condition IS NOT NULL
      AND NOT EXISTS (SELECT 1 FROM public.profile_badges pb WHERE pb.badge_id = v_badge.id AND pb.user_id = p_uid)
  LOOP
    v_meets := CASE v_badge.trigger_condition->>'type'
      WHEN 'xp_threshold' THEN
        (SELECT xp_points FROM public.profiles WHERE uid = p_uid) >=
        (v_badge.trigger_condition->>'threshold')::INTEGER
      WHEN 'course_completion' THEN
        (SELECT COUNT(*) FROM public.course_certificates WHERE user_id = p_uid) >=
        (v_badge.trigger_condition->>'count')::INTEGER
      WHEN 'streak' THEN
        (SELECT current_streak FROM public.engagement_streaks WHERE user_id = p_uid AND org_id = p_org_id) >=
        (v_badge.trigger_condition->>'days')::INTEGER
      WHEN 'block_count' THEN
        (SELECT COUNT(*) FROM public.engagement_events WHERE user_id = p_uid AND event_type = 'block_completion') >=
        (v_badge.trigger_condition->>'count')::INTEGER
      ELSE FALSE
    END;

    IF v_meets THEN
      INSERT INTO public.profile_badges (user_id, badge_id, org_id, awarded_at)
      VALUES (p_uid, v_badge.id, p_org_id, NOW())
      ON CONFLICT DO NOTHING;
    END IF;
  END LOOP;
END;
$$;
```

Wire into `record_engagement_event()`: at the end, call `PERFORM public.evaluate_badge_triggers(v_uid, v_org_id, p_event_type)`.

Add notification insert inside the loop (after successful INSERT to profile_badges) if `notifications` table exists.

### 🛡️ Security Lead

`evaluate_badge_triggers` is SECURITY DEFINER — it bypasses RLS. This is intentional: it runs
as a privileged function that can insert to `profile_badges` without the learner having INSERT
permission (learners cannot award themselves badges manually).

The function validates `org_id` on the badge query: badges are fetched WHERE `org_id = p_org_id`,
which comes from the caller's `profile_roles` record inside `record_engagement_event`. A badge
from another org cannot be awarded.

`ON CONFLICT DO NOTHING` prevents duplicate badge awards — idempotent by construction.

### 🔬 QA Lead

Manual tests:
1. Create badge with trigger `{ type: 'xp_threshold', threshold: 50 }`. Earn 50 XP. Badge appears.
2. Create badge with trigger `{ type: 'course_completion', count: 1 }`. Complete a course. Badge appears.
3. Complete a course twice → badge awarded once (ON CONFLICT DO NOTHING).
4. Badge with no trigger_condition → not auto-awarded, manual only.

### 🗄️ Data Engineer

Migration: `ALTER TABLE badges ADD COLUMN trigger_condition JSONB; ADD COLUMN is_auto_awarded BOOLEAN DEFAULT FALSE`.
`profile_badges` already has `(user_id, badge_id)` unique constraint — ON CONFLICT DO NOTHING works.
Dependencies: COUNCIL-2026-006 (engagement_streaks must exist for streak evaluation).

## Council Verdict (6/6 Unanimous)

| Decision | Verdict |
|---|---|
| Trigger evaluation | Postgres function called from record_engagement_event() |
| Trigger types | xp_threshold, course_completion, streak, block_count |
| Idempotency | ON CONFLICT DO NOTHING on profile_badges |
| Evaluation timing | Synchronous, within event insert transaction |
| Dependency | Requires COUNCIL-2026-006 (engagement_streaks) deployed first |
