# COUNCIL-2025-014: Multi-Tenant Architecture

**Date:** 2026-06-18
**Status:** Ratified — Unanimous (7/7)
**Resulted in:** ADR-2025-008, TENANCY-IMPLEMENT.md
**Tags:** multi-tenancy, platform-admin, RLS, security, organizations

---

## Question

Design the multi-tenant architecture for ChurchCore LMS. Each client organization is a
fully isolated tenant. A platform super-admin can create, modify, and delete tenants via
a dedicated console. Users are provisioned within their own tenant. The Wildcard
identifies any gaps the council did not raise.

---

## Positions

### 🏛️ The Architect

Current state: `organizations` and `org_members` exist. `profiles.org_id` and
`courses.org_id` are wired. At least 13 other tables have no `org_id` and no org-scoped
RLS — `announcements`, `messages`, `notifications`, `calendar_events`,
`course_blocks`, `block_submissions`, `course_certificates`, `guardian_links`,
`user_audit_log`, and others. Cross-tenant reads are currently possible. This is an
active gap, not a theoretical one.

**Architecture:** Shared Postgres schema, org_id RLS isolation. One Supabase project,
all tenants in the same tables. Isolation enforced entirely by RLS. This is the correct
model for this stack.

**Super-admin:** Separate identity plane — `platform_admins` table + `is_platform_admin()`
SECURITY DEFINER function. Platform admin bypass pattern on every table:
`USING (is_platform_admin() OR current_user_org_id() = org_id)`.

**Lifecycle states:** `trial → active → suspended → deleted`. Soft delete with 30-day
retention before hard purge.

**Console route:** `/platform` — completely separate from `/admin` (org-level).

### 🧭 Product Manager

The two admin identities must never blur:
- **Platform Admin:** Manages tenants as business entities. Never enters tenant course data.
- **Org Admin:** Manages their own tenant only.

**Phase 1 Platform Console features:** tenant list with status/usage, create/edit/suspend/
restore/delete tenant, provision first org admin (invite email), per-tenant feature flags,
per-tenant usage card.

**Org onboarding flow:** platform admin creates org → provisions first admin → invite email
→ password set → onboarding checklist (logo, email config, invite teachers, first course).

Supports Wildcard additions: white-labeling, feature flags, trial period, usage metrics,
onboarding checklist, health score.

### ⚙️ The Engineer

**Schema additions to `organizations`:** `status text CHECK (IN 'trial','active','suspended','deleted')`,
`trial_ends_at timestamptz`, `plan text CHECK (IN 'free','standard','premium')`,
`deleted_at timestamptz`. `settings jsonb` already exists — used for white-labeling,
feature flags, email config, onboarding progress.

**New table:** `platform_admins (id uuid PK, auth_id uuid UNIQUE → auth.users, display_name text, created_at)`.

**New function:** `is_platform_admin()` — SECURITY DEFINER, reads `platform_admins` table only.

**Tables missing `org_id`** (need column + RLS rewrite):
`announcements`, `announcement_reads`, `calendar_events`, `notifications`,
`message_threads`, `message_thread_participants`, `messages`,
`block_submissions`, `course_certificates`, `guardian_links`,
`user_audit_log`, `admin_audit_log`, `course_enrollments`.

**RLS pattern:** `USING (is_platform_admin() OR current_user_org_id() = org_id)`.

**Suspended tenant auth:** Add `tenant_active boolean DEFAULT true` to `profile_roles`,
synced by the `sync_profile_roles` trigger from `organizations.status`. RLS reads this
column — no join, no recursion risk.

**Deletion order:** block_submissions → enrollments → course_blocks → courses →
profiles → org_members → organizations.

### 🛡️ Security Officer

1. `is_platform_admin()` reads ONLY `platform_admins`. Never `profiles.role`.
2. `platform_admins` has RLS: only platform admins can read/write it.
3. First platform admin bootstrapped via migration (hardcoded auth UUID) — never via UI.
4. Every platform admin action logged to `platform_audit_log`.
5. Suspended tenants: RLS via `profile_roles.tenant_active = false` — users blocked at
   the data layer, not just the UI.
6. Soft-delete: all tables include `AND (org deleted_at IS NULL OR is_platform_admin())`.
7. Platform admin UI shows metadata only (counts, dates, flags) — never tenant course
   content or student records.
8. **Impersonation deferred to Phase 2.** Too high-risk without dedicated audit + UI
   controls.

### 🔬 The Tester

Required RLS test pairs for every org-scoped table:
- Same-org user: can read ✅
- Cross-org user: blocked ✅
- Platform admin: can read metadata ✅
- Suspended tenant user: blocked ✅
- `is_platform_admin()` returns false for tenant admin ✅
- Role escalation attempt (setting `profiles.role`) does not grant platform admin ✅

Test suite ships with the migration.

### 🚀 DevOps Officer

Provisioning: pure SQL RPC `provision_tenant()` + server action calling Supabase Admin
API for invite email. No Edge Functions for Phase 1 provisioning.

Nightly Edge Function (`purge-deleted-tenants`): hard-deletes organizations where
`deleted_at < now() - interval '30 days'`, in FK dependency order.

Nightly Edge Function (`expire-trials`): sets `status = 'suspended'` where
`trial_ends_at < now() AND status = 'trial'`.

Backfill migration: `org_id` on existing rows inferred from `created_by → profiles.org_id`
where possible; defaults to first org for single-tenant installs.

### 🃏 The Wildcard

**Gaps identified — all accepted by council for Phase 1:**

1. **White-labeling** — logo URL, primary color, custom domain, email from-name — stored in
   `organizations.settings.branding`. Layout reads at server component level to theme the
   entire experience per tenant.
2. **Feature flags** — `organizations.settings.features` controls which modules are enabled
   per tenant (AI tutor, guardian portal, leaderboard, HQ). Sidebar reads tenant features.
3. **Trial expiry enforcement** — nightly function suspends overdue trials. Without it, trials
   never end.
4. **Tenant onboarding checklist** — first-time org admin sees contextual checklist. Progress
   in `organizations.settings.onboarding`. Auto-dismissed at 100%.
5. **`platform_audit_log`** — every platform admin action logged (create, suspend, delete,
   plan change, provision user).
6. **Tenant health score** — 0–100 score derived from: active users / total users, course
   completion rate, last activity. No new tables. Shown in platform dashboard.
7. **Usage metrics** — user count, course count, storage estimate, MAU, plan expiry.
   Aggregated queries from existing tables.

**Deferred items by unanimous council vote:**
- Phase 2: GDPR/data export, Impersonation/support mode
- Phase 3: Inter-tenant content marketplace

---

## Unanimous Decision (7/7)

**Adopted architecture:** Shared Postgres schema, org_id RLS isolation, `platform_admins`
table as a separate identity plane, `/platform` console for tenant lifecycle management,
white-labeling and feature flags in `organizations.settings`, trial enforcement and health
scoring via scheduled functions.

All Phase 1 items ratified unanimously. Phase 2 and Phase 3 items deferred unanimously.
