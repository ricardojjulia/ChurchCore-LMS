**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)

**Tags:** leaderboard, gamification, xp, ranking, community

## User Story

**As a learner**, I want to see how I rank in XP among other learners in my cohort or org
so that I feel motivated by friendly competition and community belonging.

**As an admin**, I want to surface a leaderboard on the dashboard so that learners are
motivated to engage with course content more consistently.

**Acceptance Criteria:**
1. A leaderboard component shows the top 10 learners by `xp_points` within the current org
2. The current learner's rank is always shown, even if they are outside the top 10
3. Leaderboard is scoped to the org (multi-tenant: learners never see other orgs' data)
4. Display names only (no email, no auth_id)
5. Leaderboard is accessible from the learner dashboard
6. Optional: filter by cohort (if cohort context is available)

**Edge Cases:**
- Two learners with identical XP → same rank, both shown (e.g., two learners at rank 3)
- Learner has 0 XP → shown at their rank position with "0 XP"
- Org has fewer than 10 learners → show all learners (not just top 10)
- Learner is rank 1 → "You're #1!" highlight

**Out of Scope:**
- Weekly / monthly leaderboards (all-time only in v1)
- Cross-cohort leaderboards
- Public leaderboards visible outside the org
- Opt-out of leaderboard (v2 privacy feature)

---

## Gap Being Addressed

`profiles.xp_points` exists and is kept in sync by `award_xp()`. A leaderboard is a simple
`ORDER BY xp_points DESC LIMIT 10` query scoped to the org. The infrastructure is ready;
only the UI component and a supporting Postgres function/view are missing.

---

## Wildcard Pre-Audit

**A Postgres view scoped by RLS is the right pattern — not an application-level filter.**
The leaderboard query is `SELECT uid, display_name, xp_points, current_level FROM profiles WHERE org_id = $currentOrgId ORDER BY xp_points DESC LIMIT 10`. This already works if profiles has org_id. But check: does `profiles` have `org_id`? It may use `profile_roles` for org association.

Read the profiles table schema before implementing. If `profiles.org_id` doesn't exist as a direct column (the org association may be via `profile_roles`), the leaderboard query must JOIN to `profile_roles`:
```sql
SELECT p.uid, p.display_name, p.xp_points, p.current_level
FROM public.profiles p
JOIN public.profile_roles pr ON pr.uid = p.uid
WHERE pr.org_id = current_user_org_id()
  AND pr.role = 'learner'
ORDER BY p.xp_points DESC
LIMIT 10
```

Create a Postgres function `get_leaderboard(p_limit INTEGER DEFAULT 10)` that handles this query
and returns the ranked results. The function is called server-side; RLS ensures org isolation.

**Dense ranking.** Use `DENSE_RANK() OVER (ORDER BY xp_points DESC)` to handle ties correctly.
Two learners at 100 XP are both rank 1; the next learner is rank 2 (not rank 3).

**Current user's rank needs a separate query** (or a UNION). The top-10 query may not include
the current user. A second query: `SELECT rank FROM leaderboard_view WHERE uid = current_user_uid()`
or include it in the function return with a flag `is_current_user`.

## Positions

### 🏛️ The Architect

New Postgres function `get_leaderboard(p_limit INTEGER DEFAULT 10)`:
- Returns TABLE(rank BIGINT, uid UUID, display_name TEXT, xp_points INTEGER, current_level INTEGER, is_current_user BOOLEAN)
- Uses DENSE_RANK() + CTEs for top-N and current-user-row
- UNION the two result sets: top 10 + current user row (deduped with WHERE NOT IN top 10)
- Scoped to `current_user_org_id()` via profile_roles JOIN

Security: SECURITY DEFINER; org_id comes from `profile_roles` (caller), not from a parameter.

### 🧭 Product Manager

**Leaderboard UI on learner dashboard**: a card titled "Community Leaderboard" or "This Week's Champions" showing:
- Top 10 rows: Rank | Avatar (initials) | Display Name | XP | Level badge
- Current user's row: always shown (highlighted in blue/primary color), even if rank 11+
- Separator between top-10 and current user's row if user is outside top 10: "..." divider

**Admin dashboard**: can show the full org leaderboard without the current-user highlight.

### ⚙️ The Engineer

Server component: `src/components/engagement/Leaderboard.tsx`
- Calls `supabase.rpc('get_leaderboard', { p_limit: 10 })`
- Renders the ranked table
- Highlights current user's row with `is_current_user` flag
- Shows "..." separator if current user is outside top 10

Pass to the learner dashboard page.

No separate page needed — this is a dashboard widget. Optional: `/leaderboard` standalone page for a larger view (optional in this sprint).

### 🛡️ Security Lead

`get_leaderboard` is SECURITY DEFINER. The org scope comes from `current_user_org_id()` inside
the function — not from a function parameter. A learner cannot pass a different org_id.

Display names only — `uid` is included for identifying the current user's row client-side
(compare with the known current user's uid), but should not be displayed in the UI. The `uid`
returned is the profiles.uid (not auth_id) — safe to expose to the front end (it's a domain ID
with no Supabase auth significance).

RLS on the Supabase query itself also applies. The function additionally enforces org scope.

No email, no auth_id, no personal data beyond display_name and XP.

### 🔬 QA Lead

Manual tests:
1. Learner in a 5-person org → all 5 shown, ranked by XP
2. Learner with 0 XP → shown at their rank, rank = 5 (or however many have higher XP)
3. Two learners tie at 150 XP → both show rank 1; next learner shows rank 2
4. Current user is rank 1 → their row highlighted at the top
5. Current user is rank 12 → top 10 shown + "..." divider + current user's row shown below

### 🗄️ Data Engineer

Read the profiles schema: verify if `profiles.org_id` exists or if org comes from `profile_roles`.
The function JOINs accordingly. `profiles.xp_points` and `profiles.current_level` are the source.
`profiles.display_name` is shown (not the auth email).

Index: check if `(org_id, xp_points DESC)` index exists on profiles. If the leaderboard queries
profile_roles + profiles, a composite index on `profile_roles(org_id, uid)` would help. Assess
the existing indexes before adding new ones.

## Council Verdict (6/6 Unanimous)

| Decision | Verdict |
|---|---|
| Query method | get_leaderboard() SECURITY DEFINER RPC |
| Ranking | DENSE_RANK() for tie handling |
| Org scope | current_user_org_id() inside function (not a parameter) |
| Current user | Always shown in result set (UNION if outside top 10) |
| Displayed data | display_name + xp_points + level + rank only |

=== PHASE B — Migration ===

File: supabase/migrations/20260622150000_leaderboard_function.sql

CREATE OR REPLACE FUNCTION public.get_leaderboard(p_limit INTEGER DEFAULT 10)
RETURNS TABLE(
  rank        BIGINT,
  uid         UUID,
  display_name TEXT,
  xp_points   INTEGER,
  current_level INTEGER,
  is_current_user BOOLEAN
)
LANGUAGE sql SECURITY DEFINER SET search_path = public AS $$
  WITH ranked AS (
    SELECT
      DENSE_RANK() OVER (ORDER BY p.xp_points DESC) AS rank,
      p.uid,
      p.display_name,
      COALESCE(p.xp_points, 0) AS xp_points,
      COALESCE(p.current_level, 1) AS current_level,
      (p.uid = public.current_user_uid()) AS is_current_user
    FROM public.profiles p
    -- Use whichever join is appropriate based on your schema read in Phase A:
    JOIN public.profile_roles pr ON pr.uid = p.uid
    WHERE pr.org_id = public.current_user_org_id()
      AND pr.role IN ('learner', 'teacher', 'manager', 'admin') -- include all roles
  ),
  top_n AS (
    SELECT * FROM ranked ORDER BY rank ASC, xp_points DESC LIMIT p_limit
  ),
  current_user_row AS (
    SELECT * FROM ranked WHERE uid = public.current_user_uid()
    AND uid NOT IN (SELECT uid FROM top_n)
  )
  SELECT * FROM top_n
  UNION ALL
  SELECT * FROM current_user_row
  ORDER BY rank ASC, xp_points DESC
$$;

GRANT EXECUTE ON FUNCTION public.get_leaderboard(INTEGER) TO authenticated;

-- Add index if it doesn't exist:
CREATE INDEX IF NOT EXISTS idx_profiles_xp ON public.profiles(xp_points DESC);
