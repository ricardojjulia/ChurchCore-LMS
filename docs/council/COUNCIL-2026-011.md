# COUNCIL-2026-011: Question Banks

**Status:** RATIFIED — Pending Implementation
**Date:** 2026-06-21
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)
**Related:** COUNCIL-2026-010 (Quiz Extended Types)
**Tags:** quiz, question-bank, randomization, assessment, reuse

---

## User Story

**As a teacher**, I want to save quiz questions to a shared bank so that I can reuse them
across multiple quizzes and courses without re-entering them each time.

**As a teacher**, I want to randomize which questions appear from a bank on each quiz attempt
so that learners cannot share answers between attempts.

**Acceptance Criteria:**
1. Teachers can create named question banks at the org level
2. Teachers can add any question (any type) to a bank
3. In the quiz builder, a teacher can choose "Draw from question bank" instead of writing questions manually — selecting how many questions to draw
4. At quiz load time, the system randomly selects N questions from the bank for that learner's session
5. Randomization is per-learner-per-attempt (different learner = different selection; retake = new selection)
6. Banks are scoped to org (multi-tenant safe)

**Edge Cases:**
- Bank has fewer questions than N requested → use all available questions
- Same question appears in multiple banks → allowed (question has a single source of truth; banks reference it)
- Teacher edits a bank question after learners have taken quizzes from it → existing `block_submissions` are not affected (answers already recorded)

**Out of Scope:**
- Question difficulty tagging / adaptive selection
- Shared banks across orgs (platform-level bank)
- Question version history

---

## Gap Being Addressed

All quiz questions are currently embedded in `course_blocks.content.questions` JSONB. There is
no way to reuse a question across multiple courses. Teachers building similar courses (e.g., multiple
sections of the same Bible study) re-enter questions each time. Question banks solve reuse and
randomization simultaneously.

---

## Wildcard Pre-Audit

**Randomization belongs in a Postgres RPC, not the browser.** If you shuffle questions client-side,
the full question bank is sent to the browser (security risk — teacher has access to all answers).
If you shuffle server-side in a server action, you can send only the selected N questions to the client.
Use a Postgres function `draw_from_bank(p_bank_id UUID, p_count INTEGER, p_seed INTEGER)` that
uses `ORDER BY random()` (or a seeded shuffle using `p_seed` for deterministic retakes). The
seed can be `hashtext(user_id || attempt_number)` for reproducible-per-attempt selections.

**Hybrid quiz: mixed manual + bank questions.** A quiz block should be able to have both hand-written
questions AND bank-drawn questions. Store in content JSONB:
```json
{
  "questions": [...static questions...],
  "bank_draws": [
    { "bank_id": "uuid", "count": 5 }
  ],
  "time_limit_minutes": null
}
```
On quiz load, the server merges static questions + randomly drawn bank questions into the final
question set sent to the client. The client never knows which came from a bank.

**New tables needed.** Two tables:
1. `question_banks` (id, org_id, name, description)
2. `bank_questions` (id, bank_id, question_content JSONB, question_type TEXT)

## Positions

### 🏛️ The Architect

```sql
CREATE TABLE public.question_banks (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id      UUID NOT NULL REFERENCES public.organizations(id) ON DELETE CASCADE,
  name        TEXT NOT NULL,
  description TEXT,
  created_by  UUID REFERENCES public.profiles(uid),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE TABLE public.bank_questions (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  bank_id          UUID NOT NULL REFERENCES public.question_banks(id) ON DELETE CASCADE,
  question_type    TEXT NOT NULL,
  question_content JSONB NOT NULL,  -- same structure as course_blocks.content.questions[N]
  created_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

RPC `draw_from_bank(p_bank_id UUID, p_count INTEGER)` RETURNS SETOF JSONB:
- SELECT question_content FROM bank_questions WHERE bank_id = p_bank_id ORDER BY random() LIMIT p_count
- Called server-side during quiz load

### 🧭 Product Manager

**Bank management UI**: new page `/admin/question-banks` listing org's banks. Create/edit bank. Add/remove questions from a bank.

**Builder integration**: In QuizForm, add a "Draw from Bank" section below hand-written questions. Teacher selects bank (dropdown) and count. Preview shows "5 random questions from 'NT Survey Bank'" — not the actual questions (to maintain randomization).

**Player integration**: QuizPlayer receives already-resolved questions (bank questions merged in by server action on quiz load). No change to QuizPlayer needed for this feature.

### ⚙️ The Engineer

New server action `loadQuizWithBank({ courseId, blockId })`:
- Called from QuizPlayer on mount (instead of just using block.content.questions directly)
- Fetches block content
- If `content.bank_draws` exists: for each draw, call `draw_from_bank(bank_id, count)` RPC
- Merge: static questions + drawn questions (shuffled combined order)
- Return final question array to client

New page: `src/app/admin/question-banks/page.tsx` (list) + `src/app/admin/question-banks/[id]/page.tsx` (edit)

QuizPlayer change: on mount, call `loadQuizWithBank` server action if block has `bank_draws`, else use `content.questions` directly.

### 🛡️ Security Lead

- `question_banks`: org-scoped RLS; teachers/admins can manage; learners cannot read
- `bank_questions`: same RLS as question_banks
- `draw_from_bank()`: SECURITY DEFINER; verifies caller's org_id matches the bank's org_id before drawing
- Learners receive ONLY the drawn questions for their quiz session — they never receive the full bank
- The server action strips `acceptable_answers` from fill_blank questions before sending to client (to prevent answer exposure)

Wait — actually, how does grading work if `acceptable_answers` are stripped? The server action must:
1. Draw questions WITH acceptable_answers
2. Store the drawn question IDs (or the full question set with answers) server-side (in a session table or in block_submissions?)
3. Send questions WITHOUT acceptable_answers to the client
4. On submit, look up the stored answers to grade

This is complex for v1. Simpler alternative: store acceptable_answers encrypted in the question, or just include them in the client and rely on server-side grading. For v1, include all question content in the client — the teacher's acceptable_answers are already visible in the block content for static questions, so bank questions are no different.

Council decision: **For v1, send full question content including acceptable_answers to the client.** Server still grades. The teacher's answers are technically visible in the network response, but this is no worse than current behavior for static questions. v2 can add session-based answer storage.

### 🔬 QA Lead

Manual tests:
1. Create bank with 10 questions; draw 5 → learner sees 5 questions
2. Two learners take same quiz → different question selection
3. Bank has 3 questions, draw 5 → all 3 questions appear (not error)
4. Mixed quiz (2 static + 3 from bank) → 5 questions total appear

### 🗄️ Data Engineer

Two new tables. RLS required on both. `draw_from_bank` RPC uses `ORDER BY random()` which is
unseeded — different call = different result. This is intentional for randomization.

Indexes: `idx_bank_questions_bank_id(bank_id)` for the draw query.

## Council Verdict (6/6 Unanimous)

| Decision | Verdict |
|---|---|
| Tables | question_banks + bank_questions |
| Randomization | Postgres ORDER BY random() in draw_from_bank RPC |
| Hybrid quiz | Static questions + bank_draws in content JSONB |
| Answer exposure | Full question content to client (v1); v2 adds session storage |
| Bank scope | org-scoped; cross-org sharing deferred |
| Admin UI | /admin/question-banks listing + edit pages |

4. CREATE TABLE bank_questions:
   id UUID PK, bank_id UUID NOT NULL → question_banks(id) ON DELETE CASCADE,

5. ENABLE ROW LEVEL SECURITY on bank_questions

6. RLS on bank_questions:
   - SELECT: EXISTS (SELECT 1 FROM question_banks WHERE id = bank_questions.bank_id AND (is_platform_admin() OR current_user_org_id() = org_id) AND current_user_role() IN ('admin','manager','teacher'))
   - INSERT/DELETE: same check but role IN ('admin','manager')

7. INDEX idx_bank_questions_bank ON bank_questions(bank_id)

8. CREATE FUNCTION draw_from_bank(p_bank_id UUID, p_count INTEGER) RETURNS SETOF JSONB
   LANGUAGE sql SECURITY DEFINER SET search_path = public AS $$
     SELECT question_content
     FROM public.bank_questions bq
     JOIN public.question_banks qb ON qb.id = bq.bank_id
     WHERE bq.bank_id = p_bank_id
       AND qb.org_id = public.current_user_org_id()
     ORDER BY random()
     LIMIT p_count
   $$;
   GRANT EXECUTE ON FUNCTION draw_from_bank TO authenticated;
