# COUNCIL-2026-006: Engagement Tracker + Ledger

**Status:** RATIFIED — Pending Implementation
**Date:** 2026-06-21
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)
**Related:** ADR-2026-004 (Immutable Ledger Pattern)
**Tags:** engagement, xp, streaks, analytics, ledger, dashboard

---

## User Story

**As a learner**, I want to see my engagement history, XP earned per activity, and current streak
so that I understand my progress and feel motivated to continue.

**As a teacher/admin**, I want to view engagement summaries per learner and per cohort so that I
can identify disengaged students and intervene early.

**Acceptance Criteria:**
1. Every block completion, quiz submission, discussion post, and daily login records an `engagement_events` row
2. The student dashboard shows: total XP, current streak (days), longest streak, recent events (last 10)
3. The admin/teacher view shows: per-learner XP + streak, cohort engagement heatmap (optional v2)
4. XP rules: block completion = 10 XP, quiz pass = 25 XP, discussion post = 5 XP, daily login = 2 XP
5. Streak: increments if learner completes any block or logs in on consecutive calendar days; resets on miss
6. `award_xp()` is called for every XP-earning event; `profiles.xp_points` stays in sync

**Edge Cases:**
- Duplicate events (same block completed twice) must not double-award XP — use UNIQUE constraint on `(user_id, source_type, source_id)` for completion events
- Streak must handle timezone correctly (reset is midnight in the org's timezone)
- Teachers viewing their own engagement data see the student view; admin role shows the admin dashboard

**Out of Scope (this sprint):**
- Engagement heatmap visualization
- Cohort-level aggregate charts
- Custom XP rules per course (flat rules only in this sprint)
- Push notifications for streak milestones

---

## Gap Being Addressed

`profiles.xp_points` and `profiles.current_level` exist but there is no event log, no streak
tracking, no per-event attribution, and no engagement dashboard for teachers or admins. XP is
currently awarded manually or via `award_xp()` with no UI trigger from block completions.

The engagement system is the #1 proprietary differentiator and the foundation for leaderboards,
badge auto-triggers, and compliance reporting.

---

## Wildcard Pre-Audit

**Reframe: this is a Discipleship Ledger, not a gamification system.**

Every church org implementing ChurchCore will be asked "how do I prove my members are engaged
in our programs?" The engagement tracker answers that question — but only if it's positioned as
a spiritual formation accountability tool, not as a points/badges dopamine loop.

The marketing reframe matters for implementation because it shapes:
1. The default XP rules (should be activity-based, not score-based — completing a reading is worth
   points even if not "tested")
2. The admin dashboard language ("Engagement Summary" not "Leaderboard Feed")
3. The event types that matter (bible reading completion, attendance, service participation — not just quiz scores)

**Architectural concern:** The current `award_xp()` function validates that callers award XP only
to themselves or if they're staff. But block completion XP will be awarded server-side when a student
calls a server action. This means the server action should call `award_xp()` as the service role,
not the student calling it themselves. Verify this before implementing — the SECURITY DEFINER
context inside `award_xp()` uses `current_user_role()` which reads from `profile_roles`. A service
role call will not have a profile. Consider adding a SECURITY DEFINER wrapper `record_engagement_event()`
that handles both the event log insert AND the `award_xp()` call, executed as a single atomic
Postgres function rather than two separate calls from the application layer.

**Missing event source:** The current block completion flow in `BlockPlayer.tsx` does NOT call
`award_xp()`. This gap means XP has been accumulating nowhere for most learner interactions.
The engagement tracker implementation must audit which user actions currently award XP and which
do not, and add the missing hooks.

## Positions

### 🏛️ The Architect

Two new tables following ADR-2026-004:

```sql
CREATE TABLE public.engagement_events (
  id          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID        NOT NULL REFERENCES public.profiles(uid) ON DELETE CASCADE,
  org_id      UUID        NOT NULL REFERENCES public.organizations(id) ON DELETE CASCADE,
  event_type  TEXT        NOT NULL, -- 'block_completion','quiz_pass','discussion_post','daily_login'
  source_type TEXT,                 -- 'block','quiz','discussion','session'
  source_id   UUID,                 -- the block/quiz/discussion id
  xp_earned   INTEGER     NOT NULL DEFAULT 0,
  metadata    JSONB       DEFAULT '{}',
  recorded_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Deduplication: one completion event per user per source
CREATE UNIQUE INDEX idx_engagement_events_dedup
  ON public.engagement_events(user_id, source_type, source_id, event_type)
  WHERE source_id IS NOT NULL;

CREATE INDEX idx_engagement_events_user_time
  ON public.engagement_events(user_id, recorded_at DESC);

CREATE INDEX idx_engagement_events_org_time
  ON public.engagement_events(org_id, recorded_at DESC);

CREATE TABLE public.engagement_streaks (
  user_id        UUID    NOT NULL REFERENCES public.profiles(uid) ON DELETE CASCADE,
  org_id         UUID    NOT NULL REFERENCES public.organizations(id) ON DELETE CASCADE,
  current_streak INTEGER NOT NULL DEFAULT 0,
  longest_streak INTEGER NOT NULL DEFAULT 0,
  last_event_date DATE,
  PRIMARY KEY (user_id, org_id)
);
```

A Postgres function `record_engagement_event(p_uid, p_org_id, p_event_type, p_source_type, p_source_id, p_xp)` SECURITY DEFINER handles the atomic insert + `award_xp()` + streak update. Application code calls this single RPC, never the tables directly.

### 🛡️ Security Lead

`engagement_events` and `engagement_streaks`:
- INSERT only via `record_engagement_event()` SECURITY DEFINER (service role context)
- SELECT: users see own rows; teachers/admins see all in their org
- NO UPDATE or DELETE policies on `engagement_events` (immutable by design)
- `record_engagement_event()` must validate `p_uid = current_user_uid()` OR caller is staff
- XP deduplication prevents farming: UNIQUE index on `(user_id, source_type, source_id, event_type)`
  means same block can only give XP once per student

### 🔬 QA Lead

Unit tests for `recordEngagement` action:
1. Student calls with valid block_id → event inserted, XP returned
2. Student calls same block_id twice → second call is no-op (returns `xpEarned: 0`)
3. Unauthenticated call → `{ error: 'Not authenticated' }`
4. Invalid event_type → `{ error: 'Invalid event type' }`

Manual verification:
- Complete a block → XP counter in dashboard increments
- Log in on 2 consecutive days → streak shows "2"
- Skip a day → streak resets to 0 (or 1 if logged in today)

### 🗄️ Data Engineer

`record_engagement_event` function handles streak computation:
```sql
INSERT INTO public.engagement_streaks (user_id, org_id, current_streak, longest_streak, last_event_date)
VALUES (p_uid, p_org_id, 1, 1, CURRENT_DATE)
ON CONFLICT (user_id, org_id) DO UPDATE SET
  current_streak = CASE
    WHEN engagement_streaks.last_event_date = CURRENT_DATE - 1 THEN engagement_streaks.current_streak + 1
    WHEN engagement_streaks.last_event_date = CURRENT_DATE THEN engagement_streaks.current_streak
    ELSE 1
  END,
  longest_streak = GREATEST(engagement_streaks.longest_streak, ...),
  last_event_date = CURRENT_DATE;
```

Indexes: `idx_engagement_events_user_time` for the student dashboard query; `idx_engagement_events_org_time` for admin cross-learner queries. No materialized view needed at current scale.

## Council Verdict (6/6 Unanimous)

| Decision | Verdict |
|---|---|
| Data model | Immutable event log (ADR-2026-004) |
| Atomic operation | Single `record_engagement_event()` SECURITY DEFINER RPC |
| XP rules | Block: 10, Quiz pass: 25, Discussion post: 5, Daily login: 2 |
| Deduplication | UNIQUE index on (user_id, source_type, source_id, event_type) |
| Streak reset | Midnight UTC initially; org timezone in v2 |
| Student UI | Dashboard widget: XP + streak + last 5 events |
| Admin UI | Member detail page engagement tab |
| Cohort-level charts | Deferred to v2 |
| Block completion wiring | BlockPlayer must call recordEngagement on completion |
