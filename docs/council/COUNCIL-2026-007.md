# COUNCIL-2026-007: Teacher Plug Module

**Status:** RATIFIED — Pending Implementation
**Date:** 2026-06-21
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)
**Related:** none
**Tags:** teacher, profile, block, bio, content

---

## User Story

**As a teacher**, I want to add a "Teacher Plug" block to any course so that learners can see
who I am, what I teach, and how to connect with me.

**As a learner**, I want to see the instructor's bio and credentials at the top of a course
so that I know who is teaching me and can trust their expertise.

**Acceptance Criteria:**
1. Teachers can add a "Teacher Plug" block type in the course builder
2. The block renders: profile photo, display_name, bio (rich text), specialty tags (optional), website link (optional)
3. The block reads from the teacher's existing profile data — no separate input required in the builder (auto-populated from the course owner's profile)
4. Teachers can optionally override bio and specialty in the block config (for course-specific context)
5. The block is view-only for learners — no interaction required

**Edge Cases:**
- Teacher has no profile photo → show initials avatar fallback
- Bio is null → show "No bio provided" placeholder in builder preview; don't render the bio section for learners
- Course has multiple teachers (team-taught) → each adds their own Teacher Plug block

**Out of Scope:**
- A public teacher profile page accessible outside the course
- Teacher-specific role stats (avg grade given, courses taught count)
- Social media links beyond one website URL

---

## Gap Being Addressed

No teacher identity block exists in the course builder. Learners have no in-course context about
who is teaching them. Teachers cannot introduce themselves within the course content flow.
The block type system is extensible — adding a new block type is low effort.

---

## Wildcard Pre-Audit

**Pre-populate from profile, don't re-enter.** The most common mistake in LMS teacher plug
implementations is asking the teacher to type their bio into a per-course form. Teachers have
already filled out their profile. The block should auto-pull from `profiles` (display_name, bio,
profile_photo_url) and let the teacher add a course-specific override note if needed.

**Check the existing profile fields.** Read `supabase/migrations/` to verify which fields exist
on `profiles` today: `display_name`, `bio`, `profile_photo_url` may or may not exist. If `bio`
is missing, add it in the migration. If `profile_photo_url` references the storage bucket,
use a signed URL (private bucket).

**The block's `content` JSONB** should store: `{ teacher_uid: string, bio_override: string | null,
specialty_override: string | null, website: string | null }`. The renderer fetches live profile
data by `teacher_uid` at render time — the profile_photo_url is fetched live (signed URL).
Do NOT store the photo URL in block content (it would expire).

## Positions

### 🏛️ The Architect

Block type: `block_type_id = 'teacher_plug'`. Follows the existing block type pattern.
The `course_blocks.content` JSONB stores: `{ teacher_uid, bio_override, specialty, website }`.
The builder form (TeacherPlugForm) auto-fills `teacher_uid` from the currently authenticated user's uid.

`profiles` needs at minimum: `bio TEXT`, `specialty TEXT[]`. Check if these exist. If not, add them in the migration (`ALTER TABLE profiles ADD COLUMN IF NOT EXISTS bio TEXT; ADD COLUMN IF NOT EXISTS specialty TEXT[]`).

### 🧭 Product Manager

Builder UX: When a teacher adds a Teacher Plug block, a form appears pre-filled with their profile data. They can:
- Override the bio for this specific course context
- Add their specialty (e.g., "Greek & Hebrew", "Youth Ministry")
- Add an optional website link

Learner render: Clean card at the top/wherever placed — photo, name, bio, specialty chips, website link.

### ⚙️ The Engineer

New files:
- `src/components/builder/node-forms/TeacherPlugForm.tsx` — builder form (client component)
- `src/components/learning/TeacherPlugPlayer.tsx` — learner renderer (client component, fetches signed URL)

Modify:
- `src/components/learning/BlockPlayer.tsx` — add `teacher_plug` case, render `<TeacherPlugPlayer blockContent={block.content} />`
- Builder block type registry — add `teacher_plug` as a selectable block type

TeacherPlugPlayer: fetches the teacher's profile data server-side (pass as prop from parent) or fetches client-side via Supabase. If `profile_photo_url` exists, call `createSignedUrl()` on the `content-images` bucket.

### 🛡️ Security Lead

Profile data read: teacher's profile is readable by any org member via existing profile RLS policies (confirm). Photo URL: use `createSignedUrl()` — NEVER `getPublicUrl()` (ADR-2026-003).

The `teacher_uid` in block content must belong to the same org as the course. The renderer must verify this before querying. A learner cannot trick the renderer into fetching another org's teacher profile by editing the block.

Verification: In `TeacherPlugPlayer`, validate that the fetched profile's `org_id` matches the course's `org_id` (pass `orgId` as a prop from the course page).

### 🔬 QA Lead

Manual tests:
1. Add Teacher Plug block → bio and photo auto-fill from teacher's profile
2. Learner views the block → sees teacher name, bio, specialty
3. Teacher has no photo → initials avatar renders
4. Bio override saved in builder → learner sees override, not profile bio

### 🗄️ Data Engineer

Check migration for `profiles.bio` and `profiles.specialty`. If missing, new migration:
```sql
ALTER TABLE public.profiles ADD COLUMN IF NOT EXISTS bio TEXT;
ALTER TABLE public.profiles ADD COLUMN IF NOT EXISTS specialty TEXT[] DEFAULT '{}';
ALTER TABLE public.profiles ADD COLUMN IF NOT EXISTS website_url TEXT;
```
No new table. No RLS change needed (profiles RLS already allows org-member reads).

## Council Verdict (6/6 Unanimous)

| Decision | Verdict |
|---|---|
| Block type id | `teacher_plug` |
| Content JSONB | `{ teacher_uid, bio_override, specialty, website }` |
| Profile fields | Add bio, specialty[], website_url via migration if missing |
| Photo URL | createSignedUrl() on content-images bucket |
| Data source | Live profile lookup by teacher_uid (not stored in block) |
| Org validation | TeacherPlugPlayer verifies profile org_id matches course org_id |
| New tables | None |
