# COUNCIL-2025-016: MVP Completeness and Market Competitiveness Evaluation

**Date:** 2026-06-18
**Status:** Ratified — Unanimous (7/7)
**Tags:** mvp, product-strategy, competitive-analysis, go-to-market, church-lms

---

## Question

ChurchCore LMS has completed Phase 2 tenant isolation and now runs 63 migrations, 8 Edge
Functions, 23 test files, and a full platform admin console. Evaluate whether the product
constitutes a shippable MVP. Identify where it is competitive, where it is not, and what
the highest-leverage next actions are before the first paying tenant goes live.

---

## Feature Inventory (pre-session baseline)

**Core learning loop:**
`courses → modules → blocks → content_pages → learn → submissions → grading → certificates`

**Academic structure:** program tracks, terms, blueprints, sections, access windows,
meeting schedules, cohort enrollment, direct enrollment, enrollment jobs

**Communication:** direct messages, announcements, calendar, notifications, digest

**Collaboration:** section groups, group threads, group posts, my-groups

**Governance / HQ:** tasks, decisions, risks, sessions (AI-assisted)

**AI:** embedded semantic tutor (gpt-4o + text-embedding-3-small), related concepts,
confusion topics, weekly summary, global search

**Platform:** full multi-tenant console, tenant lifecycle, feature flags, per-tenant
branding, onboarding checklist, Edge Functions for trial expiry and tenant purge

**Auxiliary:** guardian portal, leaderboard, badges, performance page, reports
(materialized views, signed-URL artifacts), realtime publications, system health checks

## Positions

### 🏛️ The Architect

**Technical MVP verdict: Yes, with two blocking gaps.**

The data model is production-grade. 46 tables with full RLS isolation is more complete
than any open-source LMS I know. The helper function pattern
(`current_user_org_id() / current_user_role() / is_platform_admin()`) is clean and
consistent. The academic structure (terms → blueprints → sections) gives this platform
the rigor of a college SIS, not just a course website.

**Blocking gap 1 — Email delivery.** User invitations, password resets, enrollment
confirmations, and digest emails all require a transactional email provider. There is an
`/api/digest` route and an `InviteUserForm` but no wiring to Resend/SendGrid/Postmark.
Without this, the only way to onboard a user is a manual Supabase auth invite from the
dashboard. This is not shippable.

**Blocking gap 2 — No billing integration.** The platform tenant lifecycle is fully
implemented (`trial → active → suspended → deleted`) but there is no Stripe (or
equivalent) integration to actually charge tenants. The trial-to-paid conversion is a
manual operation by the platform admin. This is viable for a closed beta but not for
scale.

Non-blocking but important: video is not a first-class block type. The block content model
can hold anything (`course_blocks` is polymorphic), but there is no video upload/hosting
or embed block. Church training is heavily video-driven.

**Technical strengths vs. competitors:**
- RLS isolation at this depth is a security posture no church LMS competitor matches
- The Edge Function architecture (separate Deno functions for embedding generation,
  report lifecycle, certificate generation, health checks) is operationally mature
- AI tutor scoped to course content via vector search is genuinely novel in this market

### 🧭 Product Manager

**Market verdict: The product exists in an underserved niche. The current feature set
is differentiated but the positioning is unclear. That is the primary risk.**

**Competitor landscape:**

| Product | Positioning | Weakness |
|---------|-------------|----------|
| Ministry Grid (LifeWay) | Video-first church training | No academic structure, no AI, no cohorts |
| Pathwright | Cohort-based learning, church-friendly | No church governance, no guardian |
| RightNow Media | "Netflix for churches" (video library) | Not a learning platform — no quizzes, no grading |
| Disciplr | Discipleship tracking | Minimal course functionality |
| Training in Motion | Church staff training | Small catalog, no custom courses |
| Canvas / Moodle | Academic LMS | Over-engineered for churches, not church-specific |

**ChurchCore LMS's defensible moat:**
1. **Guardian portal** — no competitor has parental oversight at the data layer. Huge for
   youth ministry and children's programs (COPPA adjacent).
2. **HQ governance module** — tasks, decisions, risks tied to the LMS context. No
   competitor combines leadership governance with learning. This is a genuine blue ocean.
3. **AI tutor on your own content** — Ministry Grid uses video, not semantic search.
   None of the church competitors have an AI tutor. This is a 2–3 year advantage window.
4. **Academic rigor** (program tracks, terms, blueprints, sections) — serves seminaries,
   Bible colleges, and larger denominational bodies. No church LMS competitor has a
   real SIS-style academic structure.
5. **True multi-tenancy** — a denomination can run 500 church tenants on one platform.

**Positioning recommendation:** Don't sell this as a church LMS. Sell it as a
**Church Ministry Platform with integrated learning, governance, and AI coaching.**
The HQ module is the differentiator that makes it genuinely novel.

**Product gaps for the church buyer:**
- No Giving/stewardship connection (not blocking — out of scope)
- No sermon/media library integration (videos in course blocks would close 80% of this)
- No Planning Center / Breeze / ChMS API integration (Phase 3)
- No self-registration flow — students can only be invited by an admin. Churches often
  want a public link for member self-enrollment.
- No mobile app or PWA. Church members learn on phones during commute.

### ⚙️ The Engineer

**Build completeness: ~78%.**

What is fully wired end-to-end:
- Full course authoring + publishing pipeline
- Enrollment (cohort and direct)
- Learning + grading + certificate generation
- AI tutor with vector search
- Reporting with materialized views
- Real-time notifications (Supabase Realtime publications enabled)
- Platform admin console with tenant lifecycle
- Guardian portal
- System health checks

**What is NOT wired:**

1. **Email delivery** — `InviteUserForm` exists but no email provider is connected.
   Supabase Auth's built-in email is rate-limited (3/hour on free tier) and not
   production-grade. Estimate: 1 day to wire Resend.

2. **Stripe billing** — Trial expiry Edge Function exists and suspends orgs, but there
   is no Stripe webhook handler to move `trial → active` when payment succeeds, or
   `active → suspended` on payment failure. Estimate: 3 days for basic subscription flow.

3. **Video block type** — The block model is polymorphic. Adding a video block (YouTube
   embed, Mux, or Cloudflare Stream) is additive only. No schema change needed if using
   embed URL. Estimate: 1 day for YouTube/Vimeo embed block.

4. **Self-registration / public enrollment link** — Currently all users are admin-invited.
   Adding a public `/join/[slug]` page with org-scoped sign-up is straightforward.
   Estimate: 1 day.

5. **Rate limiting on AI routes** — `/api/ai/tutor`, `/api/ai/weekly-summary`, and
   `/api/ai/related-concepts` have no rate limiting. A single user can run up thousands
   of dollars in OpenAI calls. Estimate: 4 hours with Upstash Redis or Vercel KV.

6. **CI/CD** — No `.github/workflows` visible. Tests exist (23 files) but are not
   running automatically. Estimate: 4 hours to wire GitHub Actions.

7. **RLS penetration tests** — Unit tests cover business logic well. There are no tests
   that assert cross-tenant queries return empty results. These are the most critical
   tests for a multi-tenant product. Estimate: 1 day.

**Revised MVP launch checklist:**

- [ ] Wire Resend for transactional email
- [ ] Wire Stripe for subscription lifecycle
- [ ] Add rate limiting to AI routes
- [ ] Add video embed block type
- [ ] Add public self-registration `/join/[slug]`
- [ ] Wire CI/CD (GitHub Actions)
- [ ] Add cross-tenant RLS penetration tests

### 🛡️ Security Officer

**Security verdict: Safe to run a closed beta. Not safe for open public launch without
addressing AI rate limiting and CSP headers.**

**Strengths (genuine):**
- Phase 1 + 2 RLS isolation across 46 tables is production-grade
- `is_platform_admin()` reads only `platform_admins` — the platform identity cannot be
  forged via profile manipulation
- `profile_roles.tenant_active` blocks suspended-tenant users at the RLS layer — not
  in application code
- AI route stores SHA-256 hash of query, never plaintext — GDPR-compliant audit trail
- OpenAI API key server-side only — never in client bundle, confirmed from code
- Service role key server-side only — `createServiceClient()` used only in server
  components and Edge Functions
- `is_group_member()` is SECURITY DEFINER — no RLS bypass via group membership check

**Risks that block public launch:**

1. **AI cost DoS** — No rate limiting on `/api/ai/*`. An attacker (or a misconfigured
   client) can make unlimited OpenAI calls billed to the platform. This is a financial
   attack surface, not just a performance concern. Fix before any public URL is shared.

2. **CSP headers** — No Content-Security-Policy header configuration is visible. XSS
   attacks on the course content block player could exfiltrate session tokens. A tight
   CSP (`default-src 'self'`, `script-src 'self'`) is needed, with explicit allowlists
   for Supabase and OpenAI origins.

3. **GDPR / tenant data deletion** — The `purge-deleted-tenants` Edge Function hard-deletes
   org rows in FK-safe order. However, it does not delete Supabase Auth users belonging
   to that org. A user from a deleted tenant still has a valid Auth identity. Their
   `profile_roles` row is gone, so RLS blocks data access — but the Auth account persists.
   This is a GDPR right-to-erasure gap.

4. **Missing: audit trail for AI responses** — The `ai_query_log` records the query hash
   and model used but does not record the response. If a model hallucinates harmful
   content (theological error, safety risk for youth programs), there is no audit record
   of what was said. Consider logging a response hash or summary.

5. **Guardian portal COPPA exposure** — The guardian portal accesses student data
   (XP, level, enrollment, completions) without any explicit age verification gate.
   If ChurchCore LMS is used for youth under 13 (children's ministry), COPPA applies to
   U.S. operators. This needs a legal review and potentially a COPPA-specific consent
   flow before launch.

### 🔬 The Tester

**Test coverage verdict: Good on business logic, absent on security boundaries.**

**What is tested:**
- Grading utilities (`grading.test.ts`)
- Certificate generation (`certificate.test.ts`)
- Enrollment actions (`cohorts.test.ts`, `academic.test.ts`)
- Learning actions (`learning.test.ts`)
- Message actions (`messages.test.ts`)
- Auth permissions (`permissions.test.ts`)
- Realtime hooks (`useMessages.test.ts`, `useNotifications.test.ts`,
  `useRealtimeChannel.test.ts`)
- Health checks (`getHealthChecks.test.ts`, `health-route.test.ts`)
- Reporting (`reporting/unit.test.ts`, `reporting/api-routes.test.ts`)
- E2E critical path (`critical-path.test.ts`)
- E2E enrollment bridge (`enrollment-bridge.test.ts`)
- E2E guardian RPCs (`guardian-rpcs.test.ts`)
- E2E search (`course-search.test.ts`, `search-users.test.ts`)

**What is NOT tested:**

1. **RLS cross-tenant isolation** — This is the single most critical untested surface.
   There is no test that: signs in as a Tenant A user, queries a table, and asserts that
   Tenant B rows are not returned. This must exist before the first real tenant is live.

2. **Platform admin actions** — `createTenant`, `suspendTenant`, `softDeleteTenant`,
   and `updateTenant` are not tested. The platform admin console is brand new.

3. **Tenant lifecycle state machine** — `trial → suspended → restored → deleted` has no
   test coverage. The Edge Functions (`expire-trials`, `purge-deleted-tenants`) are not
   tested.

4. **AI tutor security** — No test confirms that the AI tutor returns only content from
   the enrolled student's section and not from another tenant's course embeddings.

5. **Profile_roles sync trigger** — The trigger that sets `tenant_active` on profile_roles
   when an org is suspended has no test. If the trigger breaks, suspended-tenant users
   regain access silently.

6. **Edge Function tests** — 0 test files in `supabase/functions/`. The nightly jobs
   that expire trials and purge tenants have no automated validation.

**Recommended test additions before launch (priority order):**
1. RLS cross-tenant penetration test suite (Postgres-level, not application-level)
2. Profile_roles sync trigger test
3. Platform admin action tests
4. Tenant lifecycle E2E
5. AI tutor isolation test (embeddings scoped to section only)

### 🚀 DevOps Officer

**Deployment readiness: 60%.**

**What is solid:**
- Supabase-hosted Postgres — no DB ops burden
- Edge Functions deployed to Supabase's Deno runtime
- `supabase db push` as the migration pipeline — proven through 63 migrations
- Health check endpoint at `/api/health` with dedicated system health checks table
- Report lifecycle manager as Edge Function — background processing is handled

**What is missing:**

1. **CI/CD pipeline** — No `.github/workflows`. Test suite is not running on every push.
   A migration error merged to main could corrupt the remote DB schema. This is the
   first infrastructure item to wire.

2. **Environment separation** — There is one remote Supabase project. There is no
   staging environment. Any migration that fails (as migrations 200000 and 200300 did in
   Phase 2) goes directly to production. A staging Supabase project should be created
   before the first paying tenant.

3. **Error tracking** — No Sentry or equivalent visible. Application errors in production
   are invisible. A single unhandled exception in `generate-embedding` or
   `report-lifecycle-manager` would fail silently.

4. **Asset CDN** — Course images and uploaded content go through `/api/upload/image`.
   Where is the storage backend? If Supabase Storage, confirm bucket policies are
   org-scoped. If not, cross-tenant asset access may be possible via direct URL guessing.

5. **OpenAI cost monitoring** — No OpenAI usage dashboard integration or budget alert.
   Combined with the missing rate limiting flagged by the Security Officer, a bad actor
   or a busy launch weekend could generate a very large bill with no alert.

6. **Vercel deployment config** — Assuming Next.js is deployed on Vercel. No
   `vercel.json` is visible. Edge runtime vs. Node runtime for AI routes should be
   explicitly configured (`export const runtime = 'nodejs'` is in the AI tutor — good —
   but not confirmed for all routes).

### 🃏 The Wildcard

**Gaps and observations nobody raised:**

1. **The HQ module is actually the product.** Every council member evaluated ChurchCore
   LMS as a learning platform that also has governance features. Flip it: this is a
   **Church Leadership Operating System** that happens to have a great LMS built in.
   The HQ council sessions (AI-assisted decision and risk tracking) is a product that
   church elder boards, denominational executives, and ministry leaders would pay for
   independently of any courses. The positioning opportunity is "Mission Control for
   your church" — LMS is one module. The council should ask: should HQ be the front door
   of the product, with Learning as a tab, not the other way around?

2. **The mobile gap is fatal for the student journey.** Adults learning during commutes,
   parents watching their kids' progress on a phone, youth ministers running a training
   session in the parking lot — all of this requires a mobile experience. The current
   app is likely responsive CSS but not a PWA. Adding a `manifest.json`, service worker,
   and offline-capable course player would cost ~3 days and dramatically expand the
   usable surface. Without it, ChurchCore LMS competes at a disadvantage against
   Ministry Grid's native mobile apps.

3. **The guardian portal is COPPA-relevant but also a premium selling point.**
   No church LMS competitor has it. Market it explicitly: "Built for youth ministry —
   parent dashboard included." A linked guardian can see completion rates, XP, and
   enrolled courses without access to course content. This closes the "parents want to
   know" use case that children's directors have been asking about for years.

4. **Supabase Storage bucket isolation.** The Engineer and DevOps flagged asset CDN.
   The specific risk: if Supabase Storage buckets are not org-scoped by RLS, a user
   from Tenant A who guesses a Tenant B asset URL can access it directly. Supabase
   Storage supports RLS policies on buckets — confirm this is wired for `course-images`
   and any other content bucket.

5. **Self-registration is a distribution multiplier.** Currently, every user must be
   invited by an admin. Many churches would want a QR code on a bulletin that leads to
   `/join/[org-slug]` for self-enrollment into a new member class. This is one day of
   engineering and it transforms the distribution model from "admin-gated" to
   "viral within a church community."

6. **The leaderboard and badge system have no notification bridge.** When a student
   earns a badge or reaches a new level, there is no push notification or in-app alert.
   Gamification without feedback loops is inert. The notification system exists
   (`notifications` table, Realtime publications) — wire badge awards to it.

7. **The "weekly summary" AI feature is a retention lever nobody is talking about.**
   `/api/ai/weekly-summary` generates a summary of course activity. If this is emailed
   to students each Monday ("You completed 3 modules last week — here's what you
   learned"), it is one of the highest-retention features in any LMS product. It needs
   the Resend integration to work, but once email is wired, this should be an automatic
   weekly digest job (Edge Function). No competitor does this.

## Council Verdict (7/7 Unanimous)

### MVP Verdict

**Closed beta: Ready now.** The core learning loop is complete, the data model is
production-grade, and multi-tenancy is fully isolated. A handful of hand-invited tenants
can use this today.

**Public launch: 2–3 weeks of focused work away.** The blocking items are:

| Item | Owner | Estimate |
|------|-------|----------|
| Resend transactional email integration | Engineer | 1 day |
| Rate limiting on `/api/ai/*` routes | Engineer | 4 hours |
| Stripe subscription lifecycle | Engineer | 3 days |
| Cross-tenant RLS penetration test suite | Tester | 1 day |
| Supabase Storage bucket RLS audit | Engineer | 4 hours |
| CI/CD (GitHub Actions) | DevOps | 4 hours |
| Staging environment (second Supabase project) | DevOps | 2 hours |
| Error tracking (Sentry) | DevOps | 2 hours |
| CSP headers | Security | 4 hours |
| Video embed block type | Engineer | 1 day |
| Self-registration `/join/[slug]` | Engineer | 1 day |

**Total: ~10–12 working days** before first paying tenant.

### Competitive Verdict

**ChurchCore LMS is differentiated** in three areas no competitor matches simultaneously:
1. Guardian portal with parental visibility
2. AI tutor on org-owned course content
3. HQ governance module (tasks, decisions, risks, AI council sessions)

**ChurchCore LMS is at parity or behind** in:
- Mobile experience (no native app, unclear PWA status)
- Video-first content (Ministry Grid, RightNow Media are video-native)
- Distribution (no public self-registration)

### Strategic Recommendation (unanimous)

**Reframe the product.** Do not pitch this as "another church LMS." The guardian portal,
AI tutor, and HQ governance module together constitute a product category that does not
exist yet: a **Church Ministry Operating System**. The LMS is the content delivery layer.
The HQ module is the leadership layer. The guardian portal is the community safety layer.

The first pitch deck should lead with: *"Ministry Grid runs your training videos.
ChurchCore runs your ministry."*
