# COUNCIL-2026-010: Quiz Extended Types — Matching, Fill-in-the-Blank, Timer

**Status:** RATIFIED — Pending Implementation
**Date:** 2026-06-21
**Council:** Architecture Council, ChurchCore LMS (6/6 unanimous)
**Related:** none
**Tags:** quiz, matching, fill-blank, timer, assessment, block

---

## User Story

**As a teacher**, I want to create matching questions (pair term to definition) and fill-in-the-blank
questions in quizzes so that I can test different types of knowledge retention.

**As a teacher**, I want to set a time limit on a quiz so that learners complete it under
timed conditions, simulating real assessments.

**As a learner**, I want to see a countdown timer when a quiz has a time limit so that
I can manage my time during the assessment.

**Acceptance Criteria:**
1. Quiz builder supports two new question types: `matching` and `fill_blank`
2. Matching: teacher defines pairs (e.g., "Grace" → "Unmerited favor"). Learner drags or selects to match each left item to its right pair
3. Fill-in-the-blank: teacher writes a sentence with `[blank]` markers. Learner types answers. Grading is case-insensitive string match (exact) or fuzzy (acceptable_answers array)
4. Timer: teacher can set `time_limit_minutes` on the quiz block. Learner sees countdown. On expiry, quiz auto-submits with current answers
5. Existing multiple-choice and true/false question types are unchanged

**Edge Cases:**
- Learner navigates away mid-timer → timer state persisted in localStorage to survive refreshes
- Timer expires while submitting → submit the in-progress state
- fill_blank with multiple blanks in one sentence → each `[blank]` is a separate input
- matching with unequal pairs (teacher error) → builder validates pairs are complete before saving

**Out of Scope:**
- Image-based matching questions
- Audio questions
- Partial credit on fill-blank (correct or incorrect only in this sprint)
- Question-level timers (timer is per-quiz, not per-question)

---

## Gap Being Addressed

The quiz system (`QuizForm.tsx`, `QuizPlayer.tsx`) supports only multiple-choice and true/false.
Matching and fill-in-the-blank are the next most requested question types. The timer is required
for formal assessments (Bible college exams, ordination prep, certification tests).

---

## Wildcard Pre-Audit

**Matching via select, not drag.** Implementing drag-and-drop matching requires @dnd-kit inside
the quiz player. This is a significant additional implementation cost. An alternative: each left
item has a dropdown/select populated with the shuffled right-side options. Functionally equivalent
for testing purposes, much simpler to implement. The council should decide: drag-match or select-match.

**`fill_blank` grading complexity.** Case-insensitive exact match will produce false negatives
("fulfilled" vs "Fulfilled" — fine; but "God's grace" vs "Gods grace" — fail). Consider
`acceptable_answers: string[]` where teacher provides 2-3 acceptable phrasings. The matcher
checks if the learner's answer matches any of them (case-insensitive). This is implemented in
the server action grading logic, not the DB.

**Timer state in localStorage.** The countdown starts when the quiz loads. If the learner
refreshes, the timer should resume from where it left off (not restart). Store `{ quizBlockId,
startedAt, timeLimitMs }` in localStorage. On load, if a quiz session exists for this blockId,
compute remaining time from (startedAt + timeLimitMs - Date.now()). If expired, auto-submit.

**`time_limit_minutes` in block content JSONB.** No migration needed — quiz config is already
stored in `course_blocks.content`. Add `time_limit_minutes: number | null` to the JSONB schema.

## Positions

### 🏛️ The Architect

Question types are stored in `course_blocks.content.questions[].type`. Current values: `'multiple_choice'`, `'true_false'`. Add: `'matching'`, `'fill_blank'`.

Matching question structure:
```json
{
  "type": "matching",
  "prompt": "Match each term to its definition",
  "pairs": [
    { "id": "1", "left": "Grace", "right": "Unmerited favor from God" },
    { "id": "2", "left": "Mercy", "right": "Not receiving deserved punishment" }
  ]
}
```

Fill-blank question structure:
```json
{
  "type": "fill_blank",
  "template": "The fruit of the Spirit is [blank], joy, [blank], and peace.",
  "blanks": [
    { "id": "1", "acceptable_answers": ["love", "Love"] },
    { "id": "2", "acceptable_answers": ["longsuffering", "patience"] }
  ]
}
```

Quiz-level timer: `course_blocks.content.time_limit_minutes: number | null`

### 🧭 Product Manager

**Builder additions:**
- Matching: teacher adds pairs via a "term" / "definition" two-column input. Add/remove pair buttons. Preview shows the pairs in order.
- Fill-blank: teacher writes the sentence with `[blank]` markers. UI parses and shows each blank with an "acceptable answers" input. One acceptable answer per blank minimum.
- Timer: a simple number input "Time limit (minutes)" in the quiz block settings. Leave blank for no limit.

**Player additions:**
- Matching: left column of terms, right column with dropdowns (shuffled options). Learner selects one right-side option per left item.
- Fill-blank: sentence rendered with `<input type="text" />` in place of each `[blank]`.
- Timer: countdown display at the top of the quiz. Red when < 1 minute remaining. Auto-submit on zero.

### ⚙️ The Engineer

**QuizForm.tsx** (builder): Add question type selector option for 'matching' and 'fill_blank'. Add conditional form sections for each type. Add time_limit_minutes input at quiz level.

**QuizPlayer.tsx** (learner): 
- Add renderers for matching (dropdown per left term) and fill_blank (inline inputs)
- Add timer logic: `useEffect` with `setInterval` countdown; localStorage persistence; auto-submit on zero

**Grading logic** (in the submit server action, wherever quiz answers are graded):
- `matching`: check each pair — learner's selected right ID matches the correct pair's right item
- `fill_blank`: for each blank, check if learner's answer is in `blank.acceptable_answers` (case-insensitive `.trim()` comparison)

No migration needed — all new data goes in existing JSONB `content` and `block_submissions.content`.

### 🛡️ Security Lead

The quiz grading happens server-side (in the existing quiz submit server action). The
`acceptable_answers` array is server-side truth; the client never makes the grading decision.

Timer auto-submit: the client sends the submit request with whatever answers are current. The
server action validates that the quiz was actually started within the time window (compare
`block_submissions.submitted_at` vs the quiz's `time_limit_minutes` — if the quiz has a timer,
reject late submissions? Discuss: probably too strict for v1; just submit whatever arrives).

For v1: accept all submissions regardless of timing. The timer is UX-enforcing, not server-enforced.
(Server-side timer enforcement is a v2 consideration — it requires tracking quiz start time server-side.)

### 🔬 QA Lead

Manual tests:
1. Matching: teacher creates 3 pairs → learner selects correct matches → 100% score
2. Matching: learner selects wrong match → question marked incorrect
3. Fill-blank: "love" matches "Love" (case-insensitive)
4. Fill-blank: "patience" matches acceptable_answers ["longsuffering", "patience"]
5. Timer: set 1-minute quiz → countdown visible → reaches 0 → auto-submit
6. Timer: refresh during countdown → timer resumes from correct remaining time

### 🗄️ Data Engineer

No migration. Question types and timer config are JSONB. `block_submissions.content` already stores learner answers as JSONB (existing pattern). No schema change.

---

## Council Verdict (6/6 Unanimous)

| Decision | Verdict |
|---|---|
| Matching interaction | Dropdown select (not drag-and-drop) |
| Fill-blank | Inline inputs; acceptable_answers[] for fuzzy match |
| Timer state | localStorage-persisted (startedAt + timeLimitMs) |
| Server timer enforcement | v1: none; v2 consideration |
| Case sensitivity | fill_blank grading: case-insensitive |
| Migration | None — JSONB only |
