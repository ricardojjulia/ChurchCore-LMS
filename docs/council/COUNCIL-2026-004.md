# COUNCIL-2026-004: PWA Manifest + Offline Course Player

**Date:** 2026-06-21
**Status:** Ratified — Unanimous (7/7)
**Tags:** pwa, offline, service-worker, next-pwa, workbox, manifest

---

## Question

ChurchCore LMS has no PWA infrastructure. Church members often attend services in buildings
with poor connectivity, or access the platform on mobile data with spotty coverage. Which
block types should be offline-capable, what is the cache invalidation strategy, and what is
the minimal correct implementation using Next.js App Router?

---

## Wildcard Pre-Audit

**What exists:**

Block types registered in `BlockPlayer.tsx`:
- `page` — static HTML/markdown content. **Offline-capable.**
- `video_stream` — YouTube/Vimeo embed (iframe). **NOT offline-capable** — requires network.
- `resource_file` — downloadable file via signed Supabase Storage URL. **Conditionally
  offline-capable** — only if the file has been previously fetched. Signed URLs expire.
- `external_url` — link to an external website. **NOT offline-capable.**
- `assignment` — static prompt (offline-readable). Submission requires network.
- `quiz` — questions are static (offline-readable). Submission requires network.
- `live_session` — meeting URL (Zoom/Meet). **NOT offline-capable.**
- `discussion` — shows cached posts (offline-readable). Posting requires network.

**No PWA files exist:** No `manifest.json`, no `manifest.webmanifest`, no service worker,
no `next-pwa` or `workbox` in `package.json`.

**App Router constraint:** Next.js App Router does not support the `next-pwa` wrapper
config approach used in Pages Router (`withPWA(nextConfig)`). `next-pwa` v5 does NOT support
App Router. The maintained fork **`@ducanh2912/next-pwa`** supports App Router and is the
current recommended package.

**`public/` directory:** Confirm it exists and is used for static assets. `manifest.json`
and the service worker must be served from `public/`.

## Positions

### 🏛️ The Architect

**Three-layer PWA architecture:**

Layer 1 — **Web App Manifest** (`public/manifest.json`):
Declares the app as installable. Required fields: `name`, `short_name`, `start_url`,
`display: standalone`, `background_color`, `theme_color`, `icons` (192×192 and 512×512 PNG).

Layer 2 — **Service Worker** via `@ducanh2912/next-pwa`:
Wraps Next.js config with Workbox. Handles:
- **App Shell caching** (stale-while-revalidate): Next.js JS/CSS chunks, fonts.
- **`/courses/[id]/pages/[blockId]` offline cache** (network-first with fallback): Page
  content for `block_type_id = 'page'` is RSC HTML — cache the rendered page. If offline,
  serve cached version.
- **Offline fallback page** (`/offline`): Shown when a requested page is not in cache and
  the user is offline.

Layer 3 — **`/offline` page**:
A simple static page that renders without any data fetching. Message: "You are offline. This
content is not available without an internet connection."

**What is NOT cached:**
- Video streams (YouTube/Vimeo iframes — these make their own requests, not interceptable)
- Supabase API calls (dynamic data — signed URLs expire, RLS requires a live JWT)
- Submission POST requests (writes cannot be safely queued without conflict-resolution)
- `resource_file` signed URLs (expire in ≤ 60 minutes — caching is harmful, not helpful)

**Cache invalidation:** Workbox handles JS/CSS via content-hash busting (Next.js already
hashes all chunks). For page HTML: `network-first` strategy with 7-day max age. When the
user is online, the service worker fetches fresh. When offline, it serves cached. On deploy,
the service worker is reinstalled (new SW file hash) and old caches are purged.

### 🧭 Product Manager

**Install prompt:** `display: standalone` triggers the browser's native "Add to home screen"
prompt on Chrome Android and Safari iOS (after repeated visits). Do not build a custom
install banner — the native prompt is sufficient for v1.

**Offline UX requirements:**
1. When offline and a cached page is available: render it normally. No "offline" indicator
   needed — the content is real.
2. When offline and the page is NOT cached: show the `/offline` fallback. Message must
   include: "Visit this page while online to read it offline next time."
3. Network status indicator: a small banner at the top of the course player when offline
   ("You are offline — interactive features are disabled"). This prevents confusion when
   quiz submissions silently fail.

**What students can do offline (with prior visit):**
- Read `page` blocks ✓
- Read `assignment` prompts ✓
- Read `quiz` questions ✓ (cannot submit)
- Read their own `discussion` post ✓ (cannot see others' new posts)

**What students cannot do offline:**
- Submit assignments, quizzes, or discussion replies
- Watch videos
- Download files (signed URLs expire)
- View any page they haven't visited while online

**Teacher/admin offline:** Not a use case. Teachers are assumed to be on reliable connections
when grading or building courses. Do not add offline caching for admin routes.

**Offline banner component `src/components/layout/OfflineBanner.tsx`:** `'use client'`
```tsx
'use client'
import { useState, useEffect } from 'react'
export function OfflineBanner() {
  const [offline, setOffline] = useState(false)
  useEffect(() => {
    setOffline(!navigator.onLine)
    const on  = () => setOffline(false)
    const off = () => setOffline(true)
    window.addEventListener('online', on)
    window.addEventListener('offline', off)
    return () => { window.removeEventListener('online', on); window.removeEventListener('offline', off) }
  }, [])
  if (!offline) return null
  return (
    <div role="status" className="bg-amber-50 border-b border-amber-200 px-4 py-2 text-xs text-amber-800 text-center">
      You are offline — interactive features (submissions, quizzes) are disabled.
    </div>
  )
}
```

Add `<OfflineBanner />` to the course layout at `src/app/courses/[id]/layout.tsx` (or the
nearest layout that wraps the course player pages).

**`workbox-fallback`:** Configure via `withPWA` options:
```typescript
workboxOptions: {
  fallbacks: {
    document: '/offline',
  },
}
```

### 🛡️ Security Officer

**Service worker scope:** The SW is registered at `/sw.js` (root scope by default with
`@ducanh2912/next-pwa`). This is correct — it covers all routes.

**What the SW must NOT cache:**
- Any request to `*.supabase.co` — these include auth tokens and signed URLs. Caching them
  would serve stale/expired JWTs or broken storage URLs. Add to Workbox runtime caching
  exclusion: `exclude: [/supabase\.co/, /resend\.com/]`.
- Any POST/PUT/DELETE requests — never cache writes.
- `/api/*` routes — these serve dynamic data and auth-sensitive responses.

**Workbox exclusion config:**
```typescript
workboxOptions: {
  fallbacks: { document: '/offline' },
  runtimeCaching: [], // use only the default App Shell caching from next-pwa
  exclude: [
    /supabase\.co/,
    /^\/api\//,
    /\/auth\//,
  ],
}
```

**The SW does not have access to Supabase session tokens.** It caches rendered HTML (RSC),
not raw API responses. The rendered HTML does not contain session tokens (those live in
cookies). This is safe.

**HTTPS requirement:** Service workers only register on HTTPS (or localhost). Verify the
production deployment is HTTPS (Vercel is always HTTPS). No action needed, just confirming.

### 🔬 The Tester

**Manual tests (no automated e2e for SW — Playwright has limited SW support):**

1. Install test: Open in Chrome → DevTools → Application → Manifest → confirm all fields valid,
   icons resolve, `display: standalone`.
2. Offline test (Chrome DevTools → Network → Offline):
   - Visit `/courses/[id]/pages/[blockId]` for a `page` block while online → cache primed.
   - Toggle offline → reload → page renders from cache.
3. Unvisited page offline: Toggle offline → navigate to an unvisited course page → `/offline`
   fallback renders.
4. Offline banner: Toggle offline → banner appears on course pages → toggle online → banner disappears.
5. Submission disabled offline: Toggle offline → try to submit a quiz → confirm the submit
   fails gracefully (network error, not a silent drop).
6. Supabase requests not cached: DevTools → Application → Cache Storage → confirm no
   `supabase.co` URLs are cached.

**TypeScript check:** `npm run typecheck` after adding `@ducanh2912/next-pwa` import to
`next.config.ts` — type definitions are included in the package.

### 🚀 DevOps Officer

**`@ducanh2912/next-pwa` generates `public/sw.js` and `public/workbox-*.js` at build time.**
Add these to `.gitignore`:
```
public/sw.js
public/workbox-*.js
```
They are generated on every build — committing them causes unnecessary diffs.

**`disable: process.env.NODE_ENV === 'development'`** disables the SW in dev mode.
In dev, the SW would cache RSC responses and cause confusing stale-data bugs. Always disable
in development.

**Vercel deployment:** No additional config. Vercel serves `public/` statically. The
generated `sw.js` is built before the Vercel deploy step (`next build` → outputs to `public/`
→ Vercel deploys `public/`).

**iOS Safari note:** iOS Safari requires the `apple-touch-icon` link and does not support
the Web App Manifest install prompt (no automatic prompt — user must manually use
"Add to Home Screen"). The meta tags in `layout.tsx` handle this.

### 🃏 The Wildcard

1. **Icon generation must happen before the manifest is valid.** If `icon-192.png` and
   `icon-512.png` do not exist in `public/icons/`, the manifest fails Lighthouse validation
   and the install prompt never appears. The implementation agent must create these files.
   Minimal approach: generate them using the `canvas` module in a one-time script, OR check
   in hand-crafted PNGs, OR use a placeholder from a public CDN (not recommended for prod).

2. **`next.config.ts` uses TypeScript but `withPWA` may expect CommonJS.** If the project
   uses `next.config.ts` (TypeScript config), check the import syntax. `@ducanh2912/next-pwa`
   supports ESM imports. If there are type errors, add `// @ts-expect-error` with a note that
   the types are in the package but not auto-detected by older tsc versions.

3. **`resource_file` signed URLs must be excluded from cache.** Supabase Storage signed URLs
   contain `?token=...` query params and expire in 60 minutes. If Workbox caches a signed URL,
   it will serve a 400 error after expiry. Confirm the Workbox exclusion rule covers
   Supabase Storage URLs (`storage.supabase.co` or `*.supabase.co/storage/`).

4. **The course layout may not exist.** Check whether `src/app/courses/[id]/layout.tsx`
   exists. If it does not, `OfflineBanner` should go in a layout that wraps course pages
   without wrapping admin pages (to avoid the banner appearing in the course builder).

5. **Background sync for offline submissions** is explicitly out of scope for v1. Do NOT
   implement background sync (queuing submissions while offline for later submission).
   This requires conflict resolution logic that belongs in its own council session.
   The current spec is: fail gracefully when offline, tell the user to retry when connected.

---

## Council Verdict (7/7 Unanimous)

Minimum viable PWA: manifest + icons + service worker via `@ducanh2912/next-pwa` + offline
fallback page + offline banner on course pages. No background sync. No custom install prompt.

| Decision | Verdict |
|---|---|
| Package | `@ducanh2912/next-pwa` (App Router compatible) |
| Offline-capable block types | `page`, `assignment` (read), `quiz` (read) |
| Cache strategy | Network-first for page HTML, SW App Shell for JS/CSS |
| Supabase requests | Never cached (excluded from SW) |
| Offline banner | Yes — on course layout only |
| Offline fallback | `/offline` page |
| Background sync | Out of scope — v1 fails gracefully |
| Icons | Must be created as part of this prompt |
| SW in dev | Disabled (`process.env.NODE_ENV === 'development'`) |

## Implementation Prompt

**New dependency:** `npm install @ducanh2912/next-pwa`

**Files to create:**
- `public/manifest.json`
- `public/icons/icon-192.png` (generate programmatically or check in)
- `public/icons/icon-512.png`
- `src/app/offline/page.tsx`
- `src/components/layout/OfflineBanner.tsx`

**Files to modify:**
- `next.config.ts` — wrap with `withPWA`
- `src/app/layout.tsx` — add manifest link, theme-color, apple meta tags
- `.gitignore` — add `public/sw.js` and `public/workbox-*.js`
- `src/app/courses/[id]/layout.tsx` (or nearest course layout) — add `<OfflineBanner />`

**Work:**

1. **`npm install @ducanh2912/next-pwa`** — add the package.

2. **`next.config.ts`**: Import `withPWA`, wrap the existing config. Set `dest: 'public'`,
   `disable: process.env.NODE_ENV === 'development'`. Configure Workbox to exclude
   `/api/*` and `*.supabase.co` from runtime caching. Set `fallbacks: { document: '/offline' }`.

3. **`public/manifest.json`**: Full manifest as specified in Engineer section. Theme color
   `#4f46e5`, background `#0f172a`, `start_url: '/dashboard'`, `display: 'standalone'`.

4. **Icons**: Create `public/icons/` directory. Generate `icon-192.png` and `icon-512.png`.
   Use Node.js `canvas` package in a one-time script (`scripts/generate-icons.js`), OR if
   canvas is unavailable, create minimal valid PNGs using a binary-safe approach. The icons
   must be real PNG files (not placeholders) for the manifest to pass validation.

5. **`src/app/layout.tsx`**: Add manifest link, theme-color meta, apple-mobile-web-app
   meta tags, and apple-touch-icon link to the `<head>` section.

6. **`src/app/offline/page.tsx`**: Static page, no data fetching. Shows offline message
   with "Visit this page while online" note. No auth required.

7. **`src/components/layout/OfflineBanner.tsx`**: Client component, listens to
   `window.addEventListener('online'/'offline')`. Renders amber banner only when offline.
   Has `role="status"` for screen reader announcement.

8. **Course layout**: Find the layout file that wraps course pages (check
   `src/app/courses/[id]/layout.tsx`). Add `<OfflineBanner />` at the top of the layout body,
   above the main content. Import from `@/components/layout/OfflineBanner`.

9. **`.gitignore`**: Add `public/sw.js` and `public/workbox-*.js` at the end.

**Security:**
- Workbox must never cache `*.supabase.co` requests (auth tokens, signed URLs).
- Never cache POST/PUT/DELETE.
- Offline fallback page must not contain auth-sensitive content.

**Verification:**
- `npm run typecheck`
- `npm run build` — confirm SW files are generated in `public/`
- Chrome DevTools → Application → Manifest → all green, icons load
- Chrome DevTools → Application → Service Workers → SW registered
- Chrome DevTools → Network → Offline → visit cached page → renders
- Chrome DevTools → Network → Offline → visit uncached page → `/offline` renders
